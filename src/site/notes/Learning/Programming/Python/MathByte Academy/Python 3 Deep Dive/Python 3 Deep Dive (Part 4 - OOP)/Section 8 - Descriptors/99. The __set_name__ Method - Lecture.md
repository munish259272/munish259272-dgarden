# Using `__set_name__` in Descriptors

## Back to Instance Dictionary

Let’s store data in the instance’s `__dict__` instead of a descriptor dictionary:

- **Why**: Simpler than managing internal storage, if `__dict__` exists.
- **Catch**: Assumes no slots (slotted classes lack `__dict__`).

---

## Meet `__set_name__`

New in Python 3.6+, `__set_name__` runs once when the descriptor is assigned:

```python
class IntegerValue:
    def __set_name__(self, owner, name):
        self.name = name  # Captures attribute name (e.g., 'x')
    def __set__(self, instance, value):
        instance.__dict__[self.name] = int(value)
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return instance.__dict__.get(self.name)

class Point:
    x = IntegerValue()
```

- **How**: Called with `owner` (class) and `name` (attribute name) during class creation.
- **Use**: Stores the attribute name (e.g., `x`) for later use.
- **Key Point**: **Knows its own name at setup**.

---

## Benefits in Action

Validate and store with the name:

```python
p = Point()
p.x = 10.1
print(p.x)         # 10
print(p.__dict__)  # {'x': 10}
```

- **Validation**: `__set__` can use `self.name` in error messages (e.g., "Invalid value for x").
- **Storage**: Saves to `instance.__dict__[self.name]`, avoiding name clashes with unrelated attributes.
- **Key Point**: **Matches attribute name naturally**.


> [!question]- Does `__set_name__` gets called when it encounters the expression `p = Point()` or when python encounters the definition of the class `class Point: x = IntegerValue()` ?
> 
> `__set_name__` is called during the class creation phase when Python encounters the definition of the class, specifically when it processes the class attribute assignment `x = IntegerValue()`. At this point, `__set_name__` is triggered for each descriptor in the class body, allowing it to capture and store its own name (e.g., 'x').
> 
> ```python
> class IntegerValue:
>     def __set_name__(self, owner, name):
>         print(f"__set_name__ called with owner={owner}, name={name}")
>         self.name = name  # Captures attribute name (e.g., 'x')
>     def __set__(self, instance, value):
>         instance.__dict__[self.name] = int(value)
>     def __get__(self, instance, owner):
>         if instance is None:
>             return self
>         return instance.__dict__.get(self.name)
> 
> class Point:
>     x = IntegerValue()
> ```
> 
> In your example, when Python processes `class Point: x = IntegerValue()`, it calls `IntegerValue.__set_name__(Point, 'x')`, where `Point` is the owner class and `'x'` is the name of the attribute. This happens once during class definition and not when instances of the class are created (e.g., with `p = Point()`).

---

## Shadowing Puzzle

Doesn’t `__dict__['x']` hide the descriptor?

- **Expectation**: Normally, `p.x` would fetch from `__dict__`, not the class.
- **Reality**: **Data descriptors (with `__set__`) take priority over `__dict__`**:
  - `p.x` calls `__get__`, not `__dict__['x']` directly.
- **Non-Data Descriptors**: Without `__set__`, `__dict__` would shadow.

```python
p.x = 20
print(p.x)  # 20 (via __get__, not shadowed)
```

- **Key Point**: **Data descriptors override instance dict lookup**.
