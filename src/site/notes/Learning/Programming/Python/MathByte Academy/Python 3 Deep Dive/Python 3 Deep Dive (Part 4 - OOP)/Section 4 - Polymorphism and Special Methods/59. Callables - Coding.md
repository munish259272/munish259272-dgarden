# Making Objects Callable: From Basics to Decorators

## Basic Callable Objects

Make an instance callable with `__call__`:

```python
class Person:
    def __call__(self):
        print("Dunder call was called")

p = Person()  # Instantiates (not __call__)
p()           # Dunder call was called
```

- **Class vs Instance**: `Person()` calls `__init__`; `p()` calls `__call__`.
- **Type**: `type(p)` is `Person`, but it’s callable.
- **Key Point**: ** `__call__` turns instances into callables**.

---

## Callables Beyond Functions

Python’s “callables” aren’t just functions:

- **Functions**: `def my_func(): ...` → `callable(my_func)` is `True`.
- **Classes**: `type(Person)` is `type`, callable to create instances.
- **Instances**: With `__call__`, like `p` above.
- **Example**: `functools.partial`:
  ```python
  from functools import partial
  def my_func(a, b, c): return (a, b, c)
  pf = partial(my_func, 10, 20)
  print(pf(30))  # (10, 20, 30)
  print(type(pf))  # <class 'functools.partial'>
  ```
- **Note**: `partial` is a callable class, not a function.
- **Takeaway**: **Callables include more than functions**.

## Recreating `partial`

Build a simple `Partial` class:

```python
class Partial:
    def __init__(self, func, *args):
        self._func = func
        self._args = args
    
    def __call__(self, *more_args):
        return self._func(*self._args, *more_args)

def my_func(a, b, c): return (a, b, c)
pf = Partial(my_func, 10, 20)
print(pf(30))  # (10, 20, 30)
```

- **How**: Stores `func` and initial `args`, appends `more_args` on call.
- **Check**: `callable(pf)` is `True`.
- **Key Point**: ** `__call__` enables flexible invocation**.

```python
my_func = Partial(my_func, 10, 20)
print(type(my_func)) #  <class '__main__.Partial'>
print(my_func(10)) # output(10, 20, 10)
```

## Tracking Cache Misses with `defaultdict` ([[Learning/Programming/Python/Practice/Interview questions/use of defaultdict\|use of defaultdict]])

Use a callable class to count misses:

```python
from collections import defaultdict

class DefaultValue:
    def __init__(self, default=None):
        self.counter = 0
        self.default = default
    
    def __call__(self):
        self.counter += 1
        return self.default

cache1 = defaultdict(DefaultValue("A"))
cache2 = defaultdict(DefaultValue(0))
print(cache1["x"])  # A
print(cache1["y"])  # A
print(cache1.counter)  # AttributeError (cache1 is dict)
print(cache2["z"])  # 0
```

- **Fix**: Access the callable’s counter:
  ```python
  dv1 = DefaultValue("A")
  dv2 = DefaultValue(0)
  cache1 = defaultdict(dv1)
  cache2 = defaultdict(dv2)
  cache1["x"]
  cache1["y"]
  cache1["x"]
  print(dv1.counter)  # 2
  cache2["z"]
  print(dv2.counter)  # 1
  ```
- **Why**: Each instance tracks its own misses.
- **Takeaway**: **Callables can maintain state**.

> ### What is `defaultdict`?
> A `defaultdict` is a special kind of dictionary from the `collections` module in Python. It’s like a regular `dict`, but with a twist: if you try to access a key that doesn’t exist, it automatically creates that key and assigns it a default value, instead of raising a `KeyError`.
> 
> - **Regular `dict`**:
>   ```python
>   d = {}
>   print(d["x"])  # KeyError: 'x'
>   ```
> - **`defaultdict`**:
>   ```python
>   from collections import defaultdict
>   d = defaultdict(int)  # Default value is 0 (int() returns 0)
>   print(d["x"])  # 0 (no error, "x" is added with value 0)
>   ```
> 
> - **How It Works**: You give `defaultdict` a "factory" (a callable, like `int`, `list`, or a custom function/class) that produces the default value when a missing key is accessed. The factory is called with no arguments.

## Class-Based Profiler Decorator

Profile functions with a class:

```python
from time import perf_counter

class Profiler:
    def __init__(self, func):
        self.func = func
        self.counter = 0
        self.total_elapsed = 0
    
    def __call__(self, *args, **kwargs):
        self.counter += 1
        start = perf_counter()
        result = self.func(*args, **kwargs)
        self.total_elapsed += perf_counter() - start
        return result
    
    @property
    def avg_time(self):
        return self.total_elapsed / self.counter if self.counter else 0

@Profiler
def func1(a, b):
    from time import sleep
    from random import random
    sleep(random())
    return a, b

print(func1(1, 2))  # (1, 2)
func1(3, 4)
print(func1.counter)    # 2
print(func1.avg_time)   # Some float (seconds)
```

- **How**: `__call__` runs the function, tracks count and time.
- **Access**: `counter` and `avg_time` are instance attributes/property.
- **Contrast**: Closure version needs extra functions for access; this is cleaner.
- **Key Point**: **Classes simplify stateful decorators**.

> [!question]- What is perf_counter in time module ?
> 
> `perf_counter()` is a function in Python's `time` module that returns a performance counter with high resolution. It's used to measure the **duration of short events**.
> 
> ### Key Characteristics
> 
> - **Monotonic**: The counter **always increases** and never decreases. This is crucial for accurately measuring durations, as it's not affected by system time changes (like a user manually adjusting the clock).
>     
> - **Not System-wide**: The starting point of the counter is undefined and can vary between system restarts. Therefore, its absolute value is meaningless; it's the **difference between two calls** that matters.
>     
> - **High Resolution**: It offers the highest possible resolution available on your operating system, making it suitable for timing small code segments with precision.
>     
> 
> ### How to Use It
> 
> You use `perf_counter()` by calling it at the beginning and end of the code you want to time, then subtracting the two values.
> 
> ```python
> import time
> 
> start_time = time.perf_counter()
> 
> # Code to be timed
> for _ in range(1000000):
>     pass
> 
> end_time = time.perf_counter()
> 
> duration = end_time - start_time
> print(f"The loop took {duration:.6f} seconds to run.")
> ```
> 
> ---
> 
> ### `perf_counter()` vs. `time.time()`
> 
> While `time.time()` also measures time, it's a **system-wide clock** that's susceptible to changes. If a user changes their system time, `time.time()` will jump, making it unreliable for measuring code performance. `perf_counter()` is the preferred method for performance measurement due to its monotonic nature.

#### Tried it with functions as well

```python
from time import perf_counter
from functools import wraps
from time import sleep
import random

random.seed(0)

def profiler(fn):
    counter = 0
    total_elapsed = 0
    avg_time = 0

    @wraps(fn)
    def inner(*args, **kwargs):
        nonlocal counter
        nonlocal total_elapsed
        nonlocal avg_time
        counter += 1
        start = perf_counter()
        result = fn(*args, **kwargs)
        end = perf_counter()
        total_elapsed += (end - start)
        avg_time = total_elapsed / counter
        return result

    # Our user might want access to counter and avg_time so we can provide it this way.
    # But there is a problem with this.
    inner.counter = counter
    inner.avg_time = avg_time
    return inner


@profiler
def func1():
    sleep(random.random())

print(func1()) # None
print(func1.counter) # 0
print(func1.avg_time) # 0

> ### Why `func1.counter` and `func1.avg_time` Are `0`
> The problem stems from **when and how you assign `counter` and `avg_time` to `inner`**:
> 
> 1. **Initial Assignment**:
>    - Inside `profiler`, you define `counter = 0`, `total_elapsed = 0`, and `avg_time = 0`.
>    - After defining `inner`, you attach these values to `inner`:
>      ```python
>      inner.counter = counter  # Sets inner.counter to 0
>      inner.avg_time = avg_time  # Sets inner.avg_time to 0
>      ```
>    - This happens *once*, when the decorator is applied to `func1`, before any calls to `func1()` occur.
> 
> 1. **Closure Updates**:
>    - When `func1()` is called, `inner` runs and updates `counter`, `total_elapsed`, and `avg_time` using `nonlocal`. For example:
>      - First call: `counter` becomes `1`, `total_elapsed` gets some time, `avg_time` is calculated.
>      - Second call: `counter` becomes `2`, etc.
>    - These variables are part of the closure—they persist and update across calls.
> 
> 1. **Static Attributes**:
>    - However, `inner.counter` and `inner.avg_time` are *function attributes* set to the initial values (`0`) at decoration time.
>    - Updating `counter` and `avg_time` in the closure doesn’t update `inner.counter` or `inner.avg_time` because they’re separate copies assigned *before* any calls happen.
>    - So, `func1.counter` and `func1.avg_time` remain `0`, reflecting their state when `inner` was returned, not the live values in the closure.
> 
> ### Visualizing the Issue
> - At decoration:
>   - `counter = 0`, `avg_time = 0`.
>   - `inner.counter = 0`, `inner.avg_time = 0`.
>   - `func1` becomes `inner`.
> - After `func1()` calls:
>   - Closure: `counter = 2`, `avg_time = some_value`.
>   - Attributes: `func1.counter = 0`, `func1.avg_time = 0` (unchanged).
> 
> ### Fixing It: Expose Live Values
> To make `func1.counter` and `func1.avg_time` reflect the updated values, you need to provide access to the live closure variables. Here are two solutions:
> 
> #### Solution 1: Use Callable Attributes (Closures)
> Add functions to `inner` that return the current values:
> 
> ```python
> from time import perf_counter
> from functools import wraps
> from time import sleep
> import random
> 
> random.seed(0)
> 
> def profiler(fn):
>     counter = 0
>     total_elapsed = 0
>     avg_time = 0
> 
>     @wraps(fn)
>     def inner(*args, **kwargs):
>         nonlocal counter, total_elapsed, avg_time
>         counter += 1
>         start = perf_counter()
>         result = fn(*args, **kwargs)
>         end = perf_counter()
>         total_elapsed += (end - start)
>         avg_time = total_elapsed / counter
>         return result
> 
>     def get_counter():
>         return counter
>     
>     def get_avg_time():
>         return avg_time
> 
>     inner.counter = get_counter
>     inner.avg_time = get_avg_time
>     return inner
> 
> @profiler
> def func1():
>     sleep(random.random())
> 
> print(func1())  # None
> print(func1())  # None
> print(func1.counter())  # 2
> print(func1.avg_time())  # Some float (e.g., 0.524...)
> ```
> 
> - **Change**: Instead of `inner.counter = counter`, attach a function `get_counter` that accesses the closure’s `counter`. Same for `avg_time`.
> - **Usage**: Call them with `()`: `func1.counter()`, `func1.avg_time()`.
> - **Why It Works**: Each call fetches the current value from the closure.
> 
> #### Solution 2: Class-Based Approach
> Use a class to avoid closure complexity:
> 
> ```python
> from time import perf_counter
> from time import sleep
> import random
> 
> random.seed(0)
> 
> class Profiler:
>     def __init__(self, fn):
>         self.fn = fn
>         self.counter = 0
>         self.total_elapsed = 0
>

>         self.counter += 1
>         start = perf_counter()

>         self.total_elapsed += perf_counter() - start

> ```
> 
> - **Change**: State (`counter`, `total_elapsed`) is stored as instance attributes, updated in `__call__`.
> - **Access**: `func1.counter` and `func1.avg_time` directly reflect the live state.
> - **Why It Works**: No copying issue—attributes are part of the instance, not static snapshots.
> 
> ### Why Your Original Code Failed
> - **Snapshot Problem**: `inner.counter = counter` assigns the *value* `0` at decoration time. It’s not a live reference to the closure’s `counter`.
> - **Closure Isolation**: Updates to `counter` in `inner` don’t propagate to `inner.counter`.
> 
> ### Key Takeaway
> - **To Track State**: Either use callable attributes to peek into the closure or switch to a class where state is naturally live and accessible.

Let’s unpack why `nonlocal counter, total_elapsed, avg_time` doesn’t make `inner.counter` and `inner.avg_time` reflect the updated values, even though it updates the variables in the enclosing scope. The confusion stems from the difference between the closure’s live variables and the static attribute assignments. I’ll clarify step-by-step.
> > 
> > ### Your Code Recap
> > Here’s the key part:
> > 
> > ```python
> > def profiler(fn):

> > Here’s the critical part:
> > - **Assignment Timing**: `inner.counter = counter` and `inner.avg_time = avg_time` happen *once*, when `profiler` creates and returns `inner`, *before* any calls to `func1()`.

> > - **Function Attributes**: `inner.counter` and `inner.avg_time` are just properties attached to `inner`. They’re not linked to the closure cells—they’re separate variables that happen to be initialized with the initial values of `counter` and `avg_time`.

> > - **No Automatic Updates**: Function attributes don’t “watch” the closure variables for changes—they’re independent after assignment.

> > 
> > 1. **Initial State**:
> >    - `counter = 0`: `counter` references the integer object `0`.
> >    - `inner.counter = counter`: `inner.counter` also references `0`.

> >    - Inside `inner`, `counter += 1` runs.
> >    - For immutable objects like integers:
> >      - `+=` doesn’t modify the `0` object (impossible—it’s immutable).
> >      - Instead, it computes `0 + 1 = 1` and reassigns `counter` to the new integer object `1`.
> >    - Now:
> >      - `counter` references `1`.
> >      - `inner.counter` *still references `0`* because it was set to the original object, not a dynamic link to `counter`’s name.

> >    - `counter` becomes `2` (a new integer object).

> > - **Closure Cells**: `counter`, `total_elapsed`, and `avg_time` are stored in *cells* (special objects Python uses for closures). `inner` accesses these cells via `nonlocal`, and updates change what the cells point to (e.g., `0` → `1` → `2`).
> > - **Attribute Assignment**: `inner.counter = counter` reads the cell’s current value (`0`) and assigns that object to `inner.counter`. It doesn’t link `inner.counter` to the cell itself—just to the object `0`.
> > 
> > ### Why No Live Link?
> > Python doesn’t make `inner.counter` a dynamic reference to `counter`’s cell because:
> > - **Assignment Semantics**: `=` binds a name/attribute to an object, not to another name. There’s no syntax like `inner.counter := counter` to alias the variable itself.

    @wraps(fn)  
    def inner(*args, **kwargs):  
        nonlocal counter  
        nonlocal total_elapsed  
        nonlocal avg_time  
        counter += 1  
        start = perf_counter()  
        result = fn(*args, **kwargs)  
        end = perf_counter()  
        total_elapsed += (end - start)  
        avg_time = total_elapsed / counter  
        return result  
  
    def get_counter():  
        return counter  
  
    def get_avg_time():  
        return avg_time  
  
    inner.counter = get_counter  
    inner.avg_time = get_avg_time  
  
    return inner  
  
  
@profiler  
def func1():  
    sleep(random.random())  
  
  
print(func1())  # None
print(func1())  # None
print(func1.counter())  # 2
print(func1.avg_time()) # 0.8013335165014723
```

---

## When to Use Classes vs Functions

- **Simple Decorators**: Use functions/closures—less overhead.
- **Complex Logic/State**: Use classes—easier to read and extend.
- **Philosophy**: Python supports both; prioritize **readability**.
