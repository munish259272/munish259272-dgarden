# Storing Data in Descriptors

## Where to Store Attribute Values?

Descriptors get `instance` in `__get__` and `__set__`:

```python
class IntegerValue:
    def __set__(self, instance, value):
        instance.__dict__['x'] = int(value)  # Simplest approach
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return instance.__dict__['x']
```

- **Option 1**: Use `instance.__dict__`.
- **Pros**: Easy, works most of the time.
- **Issue**: What name should we give to the attribute shall we name it `x` or `_x`. What happens a user is using the same name, it will overwrite that name.
- **Key Point**: **Instance dictionary is a quick fix—unless slots are involved**.


> [!question]- "<font color="#000000" style="font-weight=normal;font-style:italic;">**Issue**: What name should we give to the attribute shall we name it `x` or `_x`. What happens a user is using the same name, it will overwrite that name.</font>" - What does this means since as far as i know if the user uses the same name then it will invoke the descriptor. How can it overwrite that name ?
> 
> You’ve hit on a very subtle and important distinction in how Python handles attribute lookups.
> 
> You are correct that **if a data descriptor (one that defines `__set__`) exists on the class, it will always be invoked when you try to set that attribute.** However, the "overwriting" issue mentioned in your text refers to what happens **inside the instance's `__dict__`**, not the descriptor itself.
> 
> Here is the breakdown of why naming the storage key is tricky:
> 
> ---
> 
> ### 1. The Collision in the `__dict__`
> 
> The descriptor is a "gatekeeper" sitting on the class. When you do `obj.x = 5`, the gatekeeper catches it. But the gatekeeper needs a place to put that 5.
> 
> If your descriptor uses `instance.__dict__['x'] = value`, it is putting a value into the dictionary using the key `'x'`. If the user happens to have another attribute they _intended_ to be named `x` (perhaps handled by a different part of the code or a subclass), that entry in the `__dict__` is now occupied by your descriptor's data.
> 
> ### 2. The "Masking" vs. "Overwriting"
> 
> In Python, attribute lookup follows a specific order:
> 
> 1. **Data Descriptors** (like your `IntegerValue`)
>     
> 2. **Instance `__dict__`**
>     
> 3. **Non-Data Descriptors** (only `__get__`)
>     
> 
> If you name your storage key in the `__dict__` exactly the same as the descriptor's name on the class (e.g., the descriptor is named `x` and it stores data in `__dict__['x']`):
> 
> - The descriptor **shadows** the dictionary entry.
>     
> - You haven't "overwritten" the descriptor (it's still there on the class), but you have made that specific slot in the `__dict__` unusable for anything else.
> 
> > [!question]- "<font color="#000000" style="font-weight=normal;font-style:italic;">You haven't "overwritten" the descriptor (it's still there on the class), but you have made that specific slot in the __dict__ unusable for anything else.</font>" -- You mean we should use `_x` for adding in instance dictionary or masking is good ?
> > In short: **You should almost always use a different name (like `_x`) for the dictionary key.** While "masking" works technically, using the same name for both the descriptor and the dictionary key is considered bad practice. Here is why choosing a unique storage name (like `_x` or a private mangled name) is the better path:
> > 
> > ### 1. Why `_x` is better than `x`
> > 
> > When you store the value in `instance.__dict__['_x']`, you are keeping the "public" name (`x`) and the "internal" storage (`_x`) separate.
> > 
> > - **Clarity:** If you look at `instance.__dict__`, you clearly see what is raw data (`_x`) versus what is a managed property (`x`).
> >     
> > - **Avoids "Accidental" Bypassing:** If you use the same name `x` for both, and a developer tries to do `setattr(obj, 'x', value)`, they trigger the descriptor. But if they try to manipulate the object state via `obj.__dict__['x'] = ...`, they are writing directly into the slot the descriptor relies on without any validation.
> >     
> > 
> > ### 2. The Danger of Masking (The Lookup Trap)
> > 
> > The "Lookup Chain" in Python is the reason why using the same name is confusing. When you call `obj.x`, Python follows this priority:
> > 
> > 1. **Data Descriptor:** If `x` is a data descriptor (has `__set__`), Python **ignores** the dictionary and runs the descriptor.
> >     
> > 2. **Instance Dictionary:** Python only looks here if no data descriptor is found.
> >     
> > 
> > If you name your storage key `x`, the dictionary entry is "masked" or "shadowed." It exists, but it can **never** be accessed via normal `obj.x` syntax because the descriptor always intercepts the request. This makes the dictionary entry a "ghost" that can only be reached by manually digging into `__dict__`.
> > 
> > ---
> > 
> > ### Best Practice: Use `_` Prefix or `__set_name__`
> > 
> > To avoid name collisions, modern Python (3.6+) uses the `__set_name__` method to automatically create a safe storage name.
> > 
> > 
> > ```python
> > class ValidatedNumber:
> >     def __set_name__(self, owner, name):
> >         # If the class says 'age = ValidatedNumber()', 
> >         # this sets self.storage_name to '_age'
> >         self.storage_name = '_' + name
> > 
> >     def __set__(self, instance, value):
> >         print(f"Updating {self.storage_name}")
> >         instance.__dict__[self.storage_name] = int(value)
> > 
> >     def __get__(self, instance, owner):
> >         if instance is None: return self
> >         return instance.__dict__.get(self.storage_name)
> > 
> > class User:
> >     age = ValidatedNumber() # name is 'age', storage is '_age'
> > ```
> > 
> > ### Summary Comparison
> > 
> > |**Strategy**|**instance.__dict__['x']**|**instance.__dict__['_x']**|
> > |---|---|---|
> > |**Logic**|Same name as descriptor.|Different name (internal).|
> > |**Inspection**|`__dict__` looks "clean" but confusing.|`__dict__` shows the "raw" backing variable.|
> > |**Risk**|High chance of collision with other logic.|Standard Python convention; safe.|
> > |**Recommendation**|**Avoid.**|**Preferred.**|
> > 
> > Would you like to see how this logic applies to the `Celsius` and `Temperature` code you shared earlier to fix the `__set__` method?
> 
> ### 3. The Real Danger: Manual `__dict__` Access
> 
> While most users will use `obj.x`, advanced users or libraries might interact with `obj.__dict__` directly for serialization (like JSON dumping) or debugging.
> 
> If your descriptor stores its value under a generic name like `x`:
> 
> - **Confusion:** A developer looking at `obj.__dict__` sees `{'x': 10}`. They might think `x` is a normal attribute and try to delete it or change it via the dictionary, bypassing your descriptor's logic (like the `int(value)` conversion).
>     
> - **Subclassing:** If a subclass defines a normal attribute `x` without realizing the parent class has a descriptor named `x` that uses the same dictionary key, the two will fight over that single string key in the dictionary.
>     
> 
> ---
> 
> ## The Modern Solution: `__set_name__`
> 
> To solve the "what do we name it?" problem, Python 3.6 introduced `__set_name__`. This allows the descriptor to automatically know what name it was assigned to in the class and pick a unique storage name:
> 
> ```python
> class IntegerValue:
>     def __set_name__(self, owner, name):
>         # If the user does 'age = IntegerValue()', name is 'age'
>         # We store it in '_age' to avoid looking like the public attribute
>         self.storage_name = '_' + name
> 
>     def __set__(self, instance, value):
>         instance.__dict__[self.storage_name] = int(value)
> 
>     def __get__(self, instance, owner):
>         if instance is None: return self
>         return instance.__dict__.get(self.storage_name)
> ```
> 
> **Summary:** You don't overwrite the _descriptor_, you overwrite the _namespace_ inside the object, which can lead to bugs in inheritance or when using tools that inspect the object's internal state.

## Problem with Slots

Slots remove `__dict__`:

```python
class Point2D:
    __slots__ = ()  # No dictionary
    x = IntegerValue()

p = Point2D()
p.x = 100  # AttributeError: 'Point2D' object has no attribute '__dict__'
```

- **Issue**: No `__dict__` means nowhere to store `x`.
- **Takeaway**: **Slots break the simple approach**.

---

## Alternative: Descriptor’s Dictionary

Store data in the descriptor instead:

```python
class IntegerValue:
    def __init__(self):
        self.data = {}  # Internal storage
    def __set__(self, instance, value):
        self.data[instance] = int(value)
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return self.data.get(instance)  # None if not set
```
> [!hint] Usage of get : [[Learning/Programming/Python/python MOC/Data Types & Collections MOC/dictionary#**2. Accessing Values**\|dictionary#**2. Accessing Values**]]

- **How**: Use `instance` as a key in `self.data`.
- **Works**:

```python
class Point2D:
    x = IntegerValue()
    y = IntegerValue()

p1 = Point2D()
p2 = Point2D()
p1.x = 100.1
p2.x = 200.2
print(p1.x, p2.x)  # 100 200
```

- **Key Point**: **Descriptor storage supports slots**.

## Memory Leak Risk

Shared descriptor causes issues:

```python
p = Point2D()
p.x = 100.1  # self.data[p] = 100
del p        # p still in self.data!
```

- **Ref Count**: `p`’s reference count drops from 2 (variable + `self.data`) to 1.
- ⚠️ **Warning**: **Descriptor holds a reference—memory leak!**
- **Takeaway**: **We need a cleanup strategy**.

> [!question]- If we use an instance as a key (`self.data[instance] = int(value)`), doesn't that prevent us from storing more than one value for it?
> 
> When using an instance as a key in a descriptor's internal storage (like `self.data[instance] = int(value)`), you are indeed associating a single value with that specific instance for each descriptor. However, this does not prevent you from storing multiple values for a single instance; it just means that each descriptor can store one value per instance.
> 
> In the context of descriptors, each descriptor is typically associated with one attribute. So, if you have multiple descriptors in a class (such as `x` and `y` in the `Point2D` class), each descriptor manages its own separate dictionary to store values. Here's how it works:
> 
> - **Single Descriptor Per Attribute**: Each attribute like `x` or `y` is managed by its own descriptor (`IntegerValue`). So, `self.data[instance] = int(value)` ensures that the specific attribute holds one integer value for that particular instance.
>   
> - **Multiple Attributes**: When you have multiple attributes (like both `x` and `y`) on an object, each attribute is managed by its own instance of the descriptor class. Therefore, they can store separate values independently.
> 
> - **Example**:
>   ```python
>   class Point2D:
>       x = IntegerValue()
>       y = IntegerValue()
> 
>   p1 = Point2D()
>   p2 = Point2D()
>   p1.x = 100
>   p1.y = 200
>   p2.x = 300
>   p2.y = 400
>   
>   print(p1.x, p1.y) # Outputs: (100, 200)
>   print(p2.x, p2.y) # Outputs: (300, 400)
>   ```
>   
> In this example:
> - Descriptor instances manage storage separately for each attribute (`x`, `y`) across different instances (`p1`, `p2`). 
> - Each individual instance of the class can have unique values for these attributes because they are stored in separate dictionaries maintained by their respective descriptors.
> 
> Thus, using an instance as a key does not restrict storing more than one value per object; rather it helps associate specific values with specific instances for each attribute managed by descriptors.
