# Understanding Strong and Weak References

## Strong References Basics

Strong references keep objects alive:

```python
import ctypes

def ref_count(address):
    return ctypes.c_long.from_address(address).value

class Person:
    def __init__(self, name):
        self.name = name
    def __repr__(self):
        return f"Person(name={self.name})"

p1 = Person("Guido")  # Strong ref (count: 1)
p2 = p1               # Another strong ref (count: 2)
print(ref_count(id(p1)))  # ref_count 
del p2            # Count: 1
del p1            # Count: 0, garbage collected
```

- **Rule**: Object lives until strong ref count hits 0.
- **Key Point**: **Strong refs drive garbage collection**.

---

## Weak References Unveiled

Weak refs don’t count for garbage collection:

```python
import weakref

p1 = Person("Guido")      # Strong ref (count: 1)
w1 = weakref.ref(p1)      # Weak ref (count still 1)
w1()
print(ref_count(id(p1)))
print(w1())               # Person(name=Guido)
print(ref_count(id(p1)))
del p1                    # No strong refs, collected
print(w1())               # None (dead)
```

- **How**: `weakref.ref(obj)` creates a callable weak ref.
- **Check**: `w1()` returns object or `None` if gone.
- **Takeaway**: **Weak refs track without pinning**.

## Weak Ref Pitfall

Calling weak refs can create strong refs:

```python
p1 = Person("Guido")
w1 = weakref.ref(p1)
p3 = w1()             # Strong ref (count: 2)
del p1                # Count: 1 (p3 holds it)
del p3                # Count: 0, collected
```

- ⚠️ **Warning**: **Assigning `w1()` adds a strong ref**.
- **Key Point**: **Use weak refs cautiously**.

---

## WeakKeyDictionary for Descriptors

Solve memory leaks with `WeakKeyDictionary`:

```python
from weakref import WeakKeyDictionary

class Person:
    pass

p1 = Person()
d = WeakKeyDictionary()
d[p1] = "data"
print(ref_count(id(p1)))  # 1 (weak ref doesn’t count)
del p1                    # Collected, d auto-clears
print(len(d))             # 0
```

- **Why**: Keys are weak refs, auto-removed when objects die.
- **Key Point**: **WeakKeyDictionary prevents leaks**.

## Limitations and Requirements

Weak refs have constraints:

```python
l = [1, 2, 3]
try:
    weakref.ref(l)  # TypeError: lists don’t support weak refs
except TypeError as e:
    print(e)

class Person:
    def __eq__(self, other):
        return isinstance(other, Person) and self.name == other.name
p = Person()
d[p] = 1  # TypeError: unhashable without __hash__
```

- **Limits**: Built-ins (lists, dicts, etc.) often can’t be weak-referenced.
- **Hashability**: Keys must be hashable—add `__hash__` if `__eq__` is defined.
- **Takeaway**: **Custom classes need tuning for weak refs**.

> [!note] 
> - `weakref.getweakrefcount(p1)` to get the count of weak refs. If you have a `__slot__` then you wont be able to create weak references to those objects.
> - `d.keyrefs()` to get the list of weakrefs.
> - We cannot do `d.['some string']` because we cannot create weak references to string objects

> In Python, when you define a custom class and override the `__eq__` method, the object becomes unhashable by default because Python cannot guarantee that the object's hash value will remain constant if it can be compared for equality. The reason is that if two objects are considered equal (`a == b`), they must have the same hash value (`hash(a) == hash(b)`).
> 
> However, if you don’t override `__eq__`, or if you define `__hash__` alongside `__eq__`, then instances of your class can be used as dictionary keys or stored in sets. In the code snippet provided:
> 
> ```python
> class Person:
>     def __eq__(self, other):
>         return isinstance(other, Person) and self.name == other.name
> ```
> 
> The `Person` class overrides `__eq__`, but it does not override `__hash__`. This technically makes instances of `Person` unhashable, and attempting to use them as keys in a dictionary or another context that requires hashing should raise a `TypeError`.
> 
> If it seems like it's working without defining `__hash__`, there might be an oversight in the code execution or explanation. To resolve this and ensure instances of your custom class can be used as dictionary keys (such as with a `WeakKeyDictionary`), you should define both methods:
> 
> ```python
> class Person:
>     def __init__(self, name):
>         self.name = name
>     
>     def __repr__(self):
>         return f"Person(name={self.name})"
>     
>     def __eq__(self, other):
>         return isinstance(other, Person) and self.name == other.name
>     
>     def __hash__(self):
>         return hash(self.name)
> ```
> 
> By providing a consistent implementation for both equality checking (`__eq__`) and hashing (`__hash__`), you ensure that objects of your class can safely participate in collections that rely on these operations.
