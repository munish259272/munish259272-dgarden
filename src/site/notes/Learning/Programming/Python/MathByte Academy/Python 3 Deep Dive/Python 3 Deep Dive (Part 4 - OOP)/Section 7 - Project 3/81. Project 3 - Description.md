# Inventory Application Project Overview

This project focuses on building an **inventory management application** to track computer components for a video channel that creates computer builds. The goal is to manage resources like CPUs, GPUs, memory, and storage devices, keeping track of their availability and usage in builds.

## Project Objectives
- **Track inventory** for computer components, including CPUs, GPUs, memory, hard drives (HDD), and solid-state drives (SSD).
- Monitor:
  - **Total inventory**: The total number of a specific component (e.g., `total` for Intel Core i9 9900K CPUs).
  - **Allocated inventory**: How many components are currently used in builds (e.g., `allocated`).
  - **Available inventory**: Components not in use, calculated as `total - allocated` (use a **computed property** for this, e.g., `remaining`).
- Allow operations like:
  - **Claiming** components for a build (e.g., `claim(n)` to allocate `n` units).
  - **Freeing up** components when a build is disassembled (e.g., `free_up(n)` to return `n` units to inventory).
  - **Marking components as dead** if they break (e.g., `died(n)` to reduce both `total` and `allocated` by `n`).
  - **Purchasing** new components to add to inventory (e.g., `purchased(n)` to increase `total` by `n`).

## Class Structure
### Base Class: `Resource`
The `Resource` class serves as the foundation for all inventory items, providing common functionality.

- **Attributes** (private, read-only):
  - `name`: A string for display (e.g., "Intel Core i9 9900K").
  - `manufacturer`: A string identifying the manufacturer (e.g., "Intel", "Nvidia").
  - `total`: An integer for the total number of items in inventory.
  - `allocated`: An integer for the number of items currently in use (`allocated <= total`).
  - `category`: A lowercase string of the class name (e.g., "resource" for the `Resource` class).
- **Computed Property**:
  - `remaining`: Returns `total - allocated` to show available items.
- **Methods**:
  - `claim(n)`: Allocates `n` items, increasing `allocated` by `n`.
  - `free_up(n)`: Returns `n` items to inventory, decreasing `allocated` by `n`.
  - `died(n)`: Removes `n` broken items, reducing both `total` and `allocated` by `n`.
  - `purchased(n)`: Adds `n` new items, increasing `total` by `n`.
  - `__str__`: Returns a simple string representation (e.g., the `name`).
  - `__repr__`: Provides a detailed string representation with more attributes.

- **Validation**:
  - Ensure `total` and `allocated` are non-negative integers.
  - Ensure `allocated` never exceeds `total` in methods like `claim`.
  - Prevent invalid inputs (e.g., negative values for `total` or `allocated`).

### Child Class: `CPU`
Inherits from `Resource` and adds CPU-specific properties.

- **Additional Attributes**:
  - `cores`: An integer for the number of CPU cores.
  - `socket`: A string for the CPU socket type (e.g., "LGA 1151").
  - `power_watts`: An integer for power consumption in watts.
- **Validation**:
  - Ensure `cores` and `power_watts` are positive integers.

### Intermediate Class: `Storage`
An intermediate class for storage devices (HDD and SSD) to share the `capacity` property.

- **Additional Attribute**:
  - `capacity`: The storage capacity (e.g., in GB or TB, as an integer or string).

### Child Class: `HDD`
Inherits from `Storage`.

- **Additional Attributes**:
  - `size`: The physical size of the drive (e.g., "2.5in" or "3.5in").
  - `rpm`: The rotational speed (e.g., 7200 or 10000 RPM).
- **Validation**:
  - Restrict `size` to valid values ("2.5in", "3.5in").
  - Limit `rpm` to reasonable values (e.g., between 5400 and 15000).

### Child Class: `SSD`
Inherits from `Storage`.

- **Additional Attribute**:
  - `interface`: A string for the interface type (e.g., "NVMe", "SATA").
- **Validation**:
  - Ensure `interface` is a valid string.

## Implementation Details
- **Constructors**: Each class (`Resource`, `CPU`, `Storage`, `HDD`, `SSD`) should have a full constructor (e.g., `__init__`) to initialize all attributes.
- **Private Read-Only Attributes**: Use properties with getters but no setters for attributes like `total` and `allocated`. Modify these only through methods like `claim`, `free_up`, `died`, and `purchased`.
- **String Representations**:
  - `__str__`: Simple output (e.g., `name`).
  - `__repr__`: Detailed output including key attributes.
- **Validation**:
  - Enforce constraints on numeric inputs (e.g., `total >= 0`, `allocated <= total`).
  - Validate specific fields (e.g., `size` for `HDD`, `interface` for `SSD`).

## Unit Testing
- Write unit tests for all classes and methods using the **PyTest** library.
- Test:
  - Constructor initialization.
  - Validation logic (e.g., rejecting negative `total` or invalid `size`).
  - Method behavior (e.g., `claim` increases `allocated`, `died` reduces `total`).
  - Computed property (`remaining`).

## Development Environment
- Use **PyCharm** or plain text editors (e.g., Vim) for coding.
- Avoid Jupyter Notebook for this project.
- Structure the project with separate files for classes and tests.

## Notes
- 📌 **Focus**: The project centers on creating class models, not on data persistence (e.g., databases) or building a full website/API.
- ⚠️ **Validation is critical**: Ensure robust checks for all inputs and operations.
- 🎯 **Key Methods**: `claim`, `free_up`, `died`, `purchased` are the primary interfaces for modifying inventory.
- ✅ **Testing**: Comprehensive unit tests are essential to verify functionality.
