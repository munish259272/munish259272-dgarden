# Creating Attributes at Runtime in Python

## Adding Data Attributes

We can add attributes to objects at runtime using dot notation or `setattr`:
```python
class MyClass:
    language = "Python"

obj = MyClass()
obj.version = "3.7"  # or setattr(obj, "version", "3.7")
print(obj.__dict__)  # {'version': '3.7'}
```

- **Instance-Specific**: `version` is in `obj.__dict__`, not `MyClass.__dict__`.
- **Dynamic**: Added after the class definition.

**Key Point**: **Runtime attributes** modify the instance namespace.

> [!question]- "<font color="#000000" style=font-weight:normal;font-style:italic;>Runtime attributes modify the instance namespace.</font>" - Why are the attributes like `obj.version = "3.7"` are not determined/executed at compile time ?
> Let’s dive into why attributes like `obj.version = "3.7"` are considered **runtime attributes** that modify the instance namespace and aren’t determined or executed at **compile time**.
> 
> ### Compile Time vs. Runtime in Python
> 
> - **Compile Time**: When Python parses your code into bytecode. This happens once, before execution begins. At this stage, Python defines the structure of classes, functions, and their static content (e.g., class attributes like `language = "Python"` inside a `class` block). However, it doesn’t execute statements like assignments outside of class/function definitions.
> - **Runtime**: When the bytecode is actually executed by the Python interpreter. This is when objects are created, methods are called, and dynamic assignments (like `obj.version = "3.7"`) occur.
> 
> ### Why `obj.version = "3.7"` Happens at Runtime
> 
> 1. **Instance Creation is Dynamic**:
>    - The line `obj = MyClass()` executes at runtime, creating a new instance. At compile time, Python only knows there’s a class `MyClass`—it doesn’t know about `obj` yet because no instances exist until the code runs.
>    - The instance’s namespace (`obj.__dict__`) is initialized when `obj` is created, not at compile time.
> 
> 1. **Assignment is an Executed Statement**:
>    - `obj.version = "3.7"` is a statement executed by the interpreter during runtime. At compile time, Python sees this as a potential attribute assignment but doesn’t perform it—it doesn’t even know what `obj` is until `MyClass()` runs.
>    - Compare this to `language = "Python"` inside `class MyClass`: that’s parsed and stored in `MyClass.__dict__` at compile time because it’s part of the class definition.
> 
> 1. **Dynamic Nature of Python**:
>    - Python is a **dynamically typed language**. Attributes can be added to objects at any point during execution, not just when the class is defined. The compiler doesn’t "lock in" an object’s attributes—it leaves that flexibility for runtime.
>    - When `obj.version = "3.7"` runs, it updates `obj.__dict__` (e.g., `{'version': '3.7'}`), modifying the instance namespace on the fly.
> 
> ### Evidence in Action
> 
> ```python
> class MyClass:
>     language = "Python"  # Compile-time: added to MyClass.__dict__
> 
> obj = MyClass()  # Runtime: creates instance, initializes obj.__dict__ = {}
> obj.version = "3.7"  # Runtime: modifies obj.__dict__
> print(obj.__dict__)  # {'version': '3.7'}
> print(MyClass.__dict__)  # {'language': 'Python', ...} (no 'version')
> ```
> 
> - **`language`**: Set at compile time in the class definition, stored in `MyClass.__dict__`.
> - **`version`**: Added at runtime, stored in `obj.__dict__`. The compiler doesn’t predict or execute this—it’s a runtime decision.
> 
> ### Why Not Compile Time?
> 
> - **No Static Blueprint**: Unlike languages like C++ or Java, where object attributes are often fixed at compile time (via class declarations), Python doesn’t require you to predefine instance attributes. The compiler can’t know `version` will exist because it’s not in the class definition.
> - **Execution Dependency**: `obj.version = "3.7"` depends on `obj` existing, which only happens when `MyClass()` executes. Compile time can’t resolve this—it’s too early.
> - **Flexibility**: If this were compile-time, you couldn’t add attributes dynamically (e.g., based on user input or conditions), which is a core feature of Python.
> 
> ### Compile-Time Counterexample
> 
> Class-level attributes are determined at compile time:
> ```python
> class MyClass:
>     language = "Python"  # Parsed into MyClass.__dict__ at compile time
> ```
> 
> But instance-level assignments like `obj.version = "3.7"` require execution, making them **runtime operations**.
> 
> ### Conclusion
> 
> Attributes like `obj.version = "3.7"` modify the instance namespace at **runtime** because:
> - They depend on an instance existing, which happens during execution.
> - They’re executed statements, not static definitions.
> - Python’s dynamic nature allows this flexibility, deferring such changes to runtime.
> 
> **Key Point**: **Runtime attributes** like `obj.version` are executed, not pre-determined, enabling Python’s dynamic behavior.

## Adding Function Attributes

Now, let’s add a function as an attribute:
```python
obj.say_hello = lambda: "Hello World"
print(obj.say_hello())  # "Hello World"
print(obj.say_hello)  # <function <lambda> at ...>
```

- **Not a Method**: It’s a plain function, not bound to `obj`.
- **Contrast**: If defined in the class with `def say_hello(self):`, it’d be a bound method.

**Why?** Python only auto-binds functions defined **in the class**, not added later.

**Takeaway**: **Instance-level functions** aren’t methods by default.

---

## Why Not Bound?

- **No `self`**: `lambda` doesn’t receive `obj` as an argument.
- **Class vs. Instance**: 
  - `def` inside `class`: Becomes a method when accessed via an instance.
  - Added to `obj`: Just a function in `obj.__dict__`.

Check:
```python
print(obj.__dict__)  # {'version': '3.7', 'say_hello': <function <lambda> at ...>}
```

**Key Point**: **Runtime functions lack binding** unless we make them methods.

---

## Binding Methods at Runtime

To bind a function as a method, use `types.MethodType`:
```python
from types import MethodType

class MyClass:
    language = "Python"

obj = MyClass()
obj.say_hello = MethodType(lambda self: f"Hello {self.language}", obj)
```

- **`MethodType`**: Takes a function and the instance to bind it to.
- **`self`**: The bound instance (`obj`), passed automatically.

Test it:
```python
print(obj.say_hello())  # "Hello Python"
print(obj.say_hello)  # <bound method <lambda> of <__main__.MyClass object at ...>>
```

- **Lookup**: `self.language` finds `"Python"` in `MyClass.__dict__` since `obj.__dict__` lacks it.

**Takeaway**: **`MethodType` binds functions** to instances at runtime.

---

## Instance-Specific Methods

Only `obj` has `say_hello`:
```python
obj2 = MyClass()
print(hasattr(obj2, "say_hello"))  # False
print(obj2.__dict__)  # {}
```

- **Unique**: `say_hello` is in `obj.__dict__`, not shared with other instances.

**Key Point**: **Runtime methods** affect only the targeted instance.

> [!answer]- In detail:
> In Python, a **data descriptor** is a class that implements the descriptor protocol to control access to an attribute. The descriptor protocol is defined by special methods (`__get__`, `__set__`, `__delete__`, etc.) that allow you to customize how attributes are accessed, modified, or deleted. The specific combination of these methods determines the type and behavior of the descriptor. Let’s break down what makes a descriptor a **data descriptor** and how the presence of `__get__`, `__set__`, `__delete__`, and other methods influences its classification and functionality.
> 
> ### Descriptor Basics
> A descriptor is any object that defines one or more of the following methods:
> - `__get__(self, obj, type=None)`: Called when the attribute is accessed.
> - `__set__(self, obj, value)`: Called when the attribute is assigned a value.
> - `__delete__(self, obj)`: Called when the attribute is deleted (e.g., via `del`).
> - Other methods like `__set_name__` (Python 3.6+) are used for setup but don’t affect the descriptor type.
> 
> These methods are part of the descriptor protocol, and their implementation determines how the descriptor interacts with the instance and class.
> 
> ### What Makes a Data Descriptor?
> A descriptor is classified as a **data descriptor** if it implements **both `__get__` and `__set__`** (or `__delete__`, though `__set__` is the key differentiator). The presence of `__set__` is what distinguishes a data descriptor from a non-data descriptor (also called a **non-data descriptor**), which typically implements only `__get__`.
> 
> #### Key Rules
> - **Data Descriptor**: Implements `__get__` and `__set__` (and optionally `__delete__`). This gives the descriptor full control over attribute access and modification, overriding instance dictionary lookups.
> - **Non-Data Descriptor**: Implements only `__get__` (and optionally other methods like `__set_name__`). It allows instance dictionary values to take precedence over the descriptor.
> - **No Descriptor**: If none of these methods are implemented, the object is treated as a regular attribute.
> 
> ### Descriptor Lookup Precedence
> Python’s attribute lookup follows this precedence order when accessing `obj.attr`:
> 1. **Data Descriptor** (if present in the class): The descriptor’s `__get__` is called, and `__set__`/`__delete__` override instance dictionary assignments/deletions.
> 2. **Instance Dictionary**: If no data descriptor exists, the instance’s `__dict__` is checked.
> 3. **Non-Data Descriptor** (if present in the class): The descriptor’s `__get__` is called, but instance dictionary values can override it.
> 4. **Class Dictionary**: If no descriptor or instance value exists, the class’s `__dict__` is checked.
> 
> This precedence is critical to understanding the behavior of descriptors.
> 
> ### Detailed Breakdown
> 
> #### 1. **Descriptor with Only `__get__` (Non-Data Descriptor)**
> - **Definition**: A descriptor with only `__get__` is a non-data descriptor.
> - **Behavior**:
>   - Controls attribute retrieval via `__get__`.
>   - Does not control assignment or deletion (these fall back to the instance `__dict__`).
>   - Instance attributes can override the descriptor.
> - **Example**:
>   ```python
>   class ReadOnlyDescriptor:
>       def __get__(self, obj, objtype=None):
>           return "Read-only value"
> 
>   class Test:
>       attr = ReadOnlyDescriptor()
> 
>   t = Test()
>   print(t.attr)  # "Read-only value"
>   t.attr = 42    # Works, sets t.__dict__['attr'] = 42
>   print(t.attr)  # 42 (overrides descriptor)
>   ```
> - **Use Case**: Useful for read-only properties or methods (e.g., `property()` with only a getter).
> 
> #### 2. **Descriptor with `__get__` and `__set__` (Data Descriptor)**
> - **Definition**: A descriptor with both `__get__` and `__set__` is a data descriptor.
> - **Behavior**:
>   - Controls both retrieval (`__get__`) and assignment (`__set__`).
>   - Overrides instance dictionary assignments, ensuring the descriptor manages the attribute.
> - **Example**:
>   ```python
>   class ManagedDescriptor:
>       def __init__(self):
>           self._value = 0
>       def __get__(self, obj, objtype=None):
>           return self._value
>       def __set__(self, obj, value):
>           self._value = value
> 
>   class Test:
>       attr = ManagedDescriptor()
> 
>   t = Test()
>   print(t.attr)  # 0
>   t.attr = 42    # Calls __set__, sets _value to 42
>   print(t.attr)  # 42
>   t.__dict__['attr'] = 100  # Ignored, data descriptor takes precedence
>   print(t.attr)  # 42
>   ```
> - **Use Case**: Ideal for managed attributes (e.g., properties with getters and setters, like `property()` with both).
> 
> #### 3. **Descriptor with `__get__`, `__set__`, and `__delete__` (Data Descriptor)**
> - **Definition**: Adding `__delete__` to a descriptor with `__get__` and `__set__` still makes it a data descriptor, as `__set__` is the defining factor.
> - **Behavior**:
>   - Controls retrieval, assignment, and deletion.
>   - Fully manages the attribute lifecycle.
> - **Example**:
>   ```python
>   class FullDescriptor:
>       def __init__(self):
>           self._value = 0
>       def __get__(self, obj, objtype=None):
>           return self._value
>       def __set__(self, obj, value):
>           self._value = value
>       def __delete__(self, obj):
>           del self._value
> 
>   class Test:
>       attr = FullDescriptor()
> 
>   t = Test()
>   print(t.attr)  # 0
>   t.attr = 42    # Calls __set__
>   print(t.attr)  # 42
>   del t.attr     # Calls __delete__
>   # print(t.attr)  # AttributeError: attr
>   ```
> - **Use Case**: Used for attributes with controlled deletion (e.g., resource cleanup).
> 
> #### 4. **Role of `__name__`**
> - **`__name__` is not part of the descriptor protocol**: It’s not a method like `__get__`, `__set__`, or `__delete__`. Instead, it’s an attribute that can be set on a descriptor instance to store its name, often automatically via `__set_name__` (introduced in Python 3.6).
> - **Behavior**: `__set_name__(self, owner, name)` is called when the descriptor is assigned to a class, setting the descriptor’s name (e.g., `self.__name__ = name`). This is useful for descriptors that need to know their attribute name (e.g., in validation).
> - **Example**:
>   ```python
>   class NamedDescriptor:
>       def __set_name__(self, owner, name):
>           self.__name__ = name
>       def __get__(self, obj, objtype=None):
>           return f"Accessed {self.__name__}"
> 
>   class Test:
>       attr = NamedDescriptor()
> 
>   t = Test()
>   print(t.attr)  # "Accessed attr"
>   ```
> - **Impact on Descriptor Type**: `__name__` or `__set_name__` doesn’t affect whether a descriptor is a data or non-data descriptor; it’s orthogonal to the `__get__`/`__set__` distinction.
> 
> ### Summary Table
> 
> | Descriptor Type         | `__get__` | `__set__` | `__delete__` | `__set_name__` | Behavior                                      |
> |--------------------------|-----------|-----------|--------------|----------------|-----------------------------------------------|
> | Non-Data Descriptor      | Yes       | No        | No           | Optional       | Read-only, overridden by instance attributes  |
> | Data Descriptor          | Yes       | Yes       | No/Yes       | Optional       | Full control, overrides instance attributes   |
> | Full Data Descriptor     | Yes       | Yes       | Yes          | Optional       | Full lifecycle control                        |
> | No Descriptor Effect     | No        | No        | No           | N/A            | Treated as regular attribute                  |
> 
> ### Practical Implications
> - **Choosing Descriptor Type**:
>   - Use a non-data descriptor (only `__get__`) for read-only properties or method-like behavior.
>   - Use a data descriptor (with `__set__`) when you need to enforce validation, computation, or synchronization on attribute assignment.
>   - Add `__delete__` if deletion requires custom logic (e.g., resource management).
> - **Common Use**: The `property()` built-in is a data descriptor that wraps getter, setter, and deleter functions, making it a flexible tool for attribute management.
> 
> ### Conclusion
> A descriptor becomes a **data descriptor** when it implements `__set__` alongside `__get__`, giving it precedence over instance dictionary assignments. Adding `__delete__` extends its control to deletions, while `__set_name__` (and `__name__`) aids in setup but doesn’t change the descriptor type.
