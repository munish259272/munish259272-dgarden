## Bank Account Class Project Tutorial

🎯 **Objective**: Design a **BankAccount** class to manage bank accounts with functionality for transactions, interest calculations, and time zone-aware confirmations.

### Key Requirements
- **Class Structure**:
  - Create a **BankAccount** class to handle account details and transactions.
  - Optionally, implement a **TimeZone** class to manage time zone offsets.
- **Account Attributes**:
  - **account_number**: Unique identifier (passed to initializer, no generation needed).
  - **first_name**, **last_name**: Account holder's names.
  - **full_name**: Computed property concatenating first and last names.
  - **preferred_timezone**: Stores user’s time zone offset (default to UTC).
  - **balance**: Read-only, non-negative, modified only via transactions.
  - **interest_rate**: Class-level attribute, uniform across all accounts.
- **Transaction Methods**:
  - **deposit(amount)**: Adds money to the balance.
  - **withdraw(amount)**: Subtracts money if sufficient balance exists, else declines.
  - **pay_interest()**: Applies monthly interest to the balance based on **interest_rate**.
- **Transaction Confirmation**:
  - Generate a **confirmation_number** for each transaction (deposit, withdrawal, interest, or declined).
  - Format: `<transaction_code>-<account_number>-<utc_timestamp>-<transaction_id>`.
    - **transaction_code**: 'D' (deposit), 'W' (withdrawal), 'I' (interest), 'X' (declined).
    - **utc_timestamp**: Compact UTC time (e.g., `YYYYMMDDHHMMSS`).
    - **transaction_id**: Incremental ID across all accounts, starting at 0 on program start.
- **Confirmation Parsing**:
  - **parse_confirmation(confirmation_number, timezone)**: Returns a structured object (e.g., **ConfirmationDetails**) with:
    - **account_number**
    - **transaction_code**
    - **transaction_id**
    - **utc_time**: UTC timestamp.
    - **local_time**: Time in the specified **timezone**.
- **TimeZone Class** (Optional):
  - Attributes: **name**( it is a name for the timezone), **hour_offset**, **minute_offset**.
  - Method to calculate total offset from UTC.
Example Timezone usage:
```python
tz1 = TimeZone('india', 5, 30)  
print(datetime.now(timezone.utc)) # 2025-08-09 10:44:30.172387+00:00  
print(datetime.now(timezone.utc) + tz1.offset) # 2025-08-09 16:14:30.172402+00:00  
try:  
    tz = TimeZone('', 5, 30)  
except ValueError as e:  
    print(f'{e}') # Timezone name cannot be empty
```
- **Additional Notes**:
  - Use **float** for calculations (note: **decimal** is preferred in production).
  - ⚠️ Ensure **balance** remains non-negative and is not directly modifiable.
  - 📌 Use **PyTZ** library for production-grade time zone handling.
  - ✅ Test code thoroughly, with unit tests covered in a later video.

### Example Scenario
- Account with **account_number** `140568`, **preferred_timezone** offset `-7` (MST), **balance** `100`.
- Deposit `50` at UTC time `20250809123045`, with last **transaction_id** `123`.
- New **balance**: `150`.
- **confirmation_number**: `D-140568-20250809123045-124`.
- **parse_confirmation** returns **account_number**, **transaction_code** (`D`), **transaction_id** (`124`), **utc_time**, and **local_time** in MST.

### Design Tips
- Use **class methods** (**set_interest**, **get_interest**) for managing **interest_rate**.
- Implement **properties** for **first_name**, **last_name** (read/write), **full_name**, and **balance** (read-only).
- Ensure **parse_confirmation** is flexible for time zone conversions without instance data.
- Consider private attributes (e.g., **_balance**, **_transaction_counter**) for encapsulation.
