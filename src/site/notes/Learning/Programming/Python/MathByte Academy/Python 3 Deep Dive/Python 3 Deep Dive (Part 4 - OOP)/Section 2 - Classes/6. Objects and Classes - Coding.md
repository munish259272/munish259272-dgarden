# Defining Classes in Python

## Creating a Simple Class

Let’s define a basic class:
```python
class Person:
    pass
```

- Uses the **`class` keyword**.
- Seems empty, but **Python adds functionality** automatically.

**Key Point**: A **class is an object** and a **type**, even if it does nothing yet.

---

## Class as an Object

The class itself is an object with a **type**:
```python
print(type(Person))  # <class 'type'>
```

- **Type of `Person`**: `type` (a built-in class).
- **Type of `type`**: Also `type`—a self-referencing concept (more on this later).

Python gives `Person` built-in **attributes** and **behaviors**:
- **Attribute**: `Person.__name__` returns `"Person"`.
- **Behavior**: `Person` is **callable** to create instances.

```python
print(Person.__name__)  # "Person"
```

**Takeaway**: **Classes are objects** with state and behavior from Python.

---

## Instances from Classes

Call the class to create an **instance**:
```python
p = Person()
print(type(p))  # <class '__main__.Person'>
```

- **Type of `p`**: `Person`, prefixed with `__main__` (the current module).
- **Instance**: A unique object made from the `Person` blueprint.

> [!question]- Why does this gives `<class '__main__.Person'>` instead of just `<class 'Person'>` ? How is `__main__` is involved here ?
> The output `<class '__main__.Person'>` instead of `<class 'Person'>` when printing `type(p)` in Python is due to how Python handles **module namespaces** and **class definitions**. The `__main__` part indicates the module in which the `Person` class is defined. Let’s break this down clearly, using proper Markdown formatting, code examples, and key points highlighted with bold and unicodes.
> 
> ---
> 
> ### Why `<class '__main__.Person'>`?
> 
> In Python, every class is associated with the **module** in which it is defined. The **module name** becomes part of the class’s **fully qualified name**, which is reflected when you inspect the class using `type()` or `__class__`. Here’s why `__main__` appears:
> 
> 1. **Module Context** 📌:
>    - When you run a Python script directly (e.g., `python script.py`), Python assigns the special module name **`__main__`** to the top-level module (the script being executed).
>    - Any classes defined in this script are associated with the `__main__` module.
>    - In your example, the `Person` class is defined in the script you’re running, so its fully qualified name is `__main__.Person`.
> 
> 2. **Fully Qualified Name**:
>    - The `type(p)` function returns the class object of `p`, and Python includes the **module name** in the class’s string representation to uniquely identify it.
>    - The format `<class 'module_name.class_name'>` ensures clarity, especially when multiple modules might define classes with the same name (e.g., `module1.Person` vs. `module2.Person`).
> 
> 3. **Why Not Just `<class 'Person'>`?** ⚠️:
>    - Python avoids using just `<class 'Person'>` to prevent ambiguity. If two modules define a `Person` class, the module name (e.g., `__main__`, `module1`) distinguishes them.
>    - Including the module name provides a **namespace**, ensuring the class’s origin is clear.
> 
> ---
> 
> ### How is `__main__` Involved?
> 
> The `__main__` module is a special module in Python that represents the **entry point** of a program. Here’s how it works:
> 
> - **Direct Execution**:
>   - When you run a script directly (e.g., `python script.py`), Python loads the script as the `__main__` module.
>   - Any code, including class definitions, in this script is executed in the `__main__` namespace.
>   - Example:
>     ```python
>     # script.py
>     class Person:
>         pass
> 
>     p = Person()
>     print(type(p))  # Output: <class '__main__.Person'>
>     ```
>     - Since `Person` is defined in `script.py`, which is the `__main__` module when run directly, the class is named `__main__.Person`.
> 
> - **Imported Module**:
>   - If the same script is **imported** as a module (e.g., `import script`), it is not treated as `__main__`. Instead, it takes the module’s name (e.g., `script`).
>   - Example:
>     ```python
>     # script.py
>     class Person:
>         pass
>     ```
> 
>     ```python
>     # another_script.py
>     import script
> 
>     p = script.Person()
>     print(type(p))  # Output: <class 'script.Person'>
>     ```
>     - Here, `Person` is defined in the `script` module, so the class is `script.Person`, not `__main__.Person`.
> 
> - **Interactive Shell**:
>   - In an interactive Python shell (e.g., REPL), code you type is executed in the `__main__` module, so classes defined there are also prefixed with `__main__`.
>     ```python
>     >>> class Person:
>     ...     pass
>     ...
>     >>> p = Person()
>     >>> print(type(p))
>     <class '__main__.Person'>
>     ```
> 
> ---
> 
> ### Detailed Example
> 
> Let’s explore this with a complete example to show the difference between running a script directly and importing it.
> 
> **File: `person.py`**
> ```python
> class Person:
>     pass
> 
> if __name__ == "__main__":
>     p = Person()
>     print(type(p))  # Output: <class '__main__.Person'>
> ```
> 
> **File: `main.py`**
> ```python
> import person
> 
> p = person.Person()
> print(type(p))  # Output: <class 'person.Person'>
> ```
> 
> **Explanation**:
> - When you run `person.py` directly (`python person.py`):
>   - The script is the `__main__` module.
>   - The `Person` class is defined in `__main__`, so `type(p)` shows `<class '__main__.Person'>`.
> - When you run `main.py` (`python main.py`):
>   - The `person.py` script is imported as the `person` module.
>   - The `Person` class is defined in the `person` module, so `type(p)` shows `<class 'person.Person'>`.
> 
> **Note on `__name__`** 📌:
> - The special variable `__name__` is set to `"__main__"` when a script is run directly, or to the module’s name (e.g., `person`) when imported.
> - This is why `if __name__ == "__main__":` is used to run code only when the script is executed directly.
> 
> ---
> 
> ### Why Does This Matter?
> 
> Understanding the `__main__` prefix is important for:
> - **Namespace Clarity**: It helps distinguish classes from different modules, avoiding conflicts in larger projects.
> - **Debugging**: When you see `<class '__main__.Person'>`, you know the class is defined in the script you’re running, not an imported module.
> - **Module Design**: When writing reusable code, you’ll define classes in modules that are imported, so their names reflect the module (e.g., `mymodule.Person`), not `__main__`.
> 
> **Practical Tip** ✅:
> - If you want to avoid `__main__` in class names, define classes in a separate module and import them:
>   ```python
>   # mymodule.py
>   class Person:
>       pass
>   ```
> 
>   ```python
>   # main.py
>   from mymodule import Person
>   p = Person()
>   print(type(p))  # Output: <class 'mymodule.Person'>
>   ```
> 
> ---
> 
> ### Key Takeaways
> 
> - **Why `<class '__main__.Person'>`?** 🎯:
>   - The `Person` class is defined in the `__main__` module, which is the script you’re running directly.
>   - Python includes the module name in the class’s fully qualified name to avoid ambiguity.
> 
> - **Role of `__main__`** 📌:
>   - `__main__` is the special module name for the top-level script or interactive session.
>   - When a script is imported, its module name (e.g., `person`) replaces `__main__` in class names.
> 
> - **Avoiding `__main__`**:
>   - Define classes in separate modules and import them to use the module’s name instead of `__main__`.
> 
> **Edge Case** ⚠️:
> - If you define a class in a script and import it elsewhere, ensure the script doesn’t execute unwanted code (e.g., tests) when imported. Use `if __name__ == "__main__":` to isolate such code:
>   ```python
>   class Person:
>       pass
> 
>   # Only runs when script is executed directly
>   if __name__ == "__main__":
>       p = Person()
>       print(type(p))
>   ```
> 
> ---
>

Instances get their own attributes:
```python
print(p.__class__)  # <class '__main__.Person'>
print(type(p) == p.__class__)  # True
```

- **`__class__`**: Links back to the class (same as `type(p)`).

**Key Point**: **Instances reflect their class type**.

## Checking Types and Instances

Use **`isinstance()`** to test relationships:
```python
print(isinstance(p, Person))  # True
print(isinstance(p, str))     # False

s = "hello"
print(isinstance(s, str))     # True
```

- **Why `isinstance`?**: More flexible than `type()` (especially with inheritance—later topic).
- **`str`**: A built-in class/type, just like `Person` is our custom type.

Check `str`’s type:
```python
print(type(str))  # <class 'type'>
```

**Takeaway**: **All classes are of type `type`**, a meta-concept.
