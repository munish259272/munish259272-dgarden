# Tutorial: Decorators and Descriptors as Metaprogramming

This video reviews how **decorators** and **descriptors** are practical forms of metaprogramming, allowing you to **use code to modify the behavior of other code at runtime**.

## 🎯 Decorator Example: A Debugger

A decorator can inject debugging logic into functions without modifying their original code.

```python
from functools import wraps

def debugger(fn):
    @wraps(fn)
    def inner(*args, **kwargs):
        # This is the injected "debugging" code
        print(f'{fn.__qualname__} called with args: {args}, kwargs: {kwargs}')
        # Then call the original function
        result = fn(*args, **kwargs)
        return result
    return inner
```

**Applying the decorator:**

```python
@debugger
def func1(*args, **kwargs):
    pass

@debugger
def func2(*args, **kwargs):
    pass

func1(10, 20, a='hello')  # Prints: func1 called with args: (10, 20), kwargs: {'a': 'hello'}
func2(30)                 # Prints: func2 called with args: (30,), kwargs: {}
```

**Key Advantage:** The debugging logic is written **once**. To change it, you modify only the `debugger` decorator. All decorated functions automatically inherit the change—this keeps your code **DRY**.

## 🎯 Descriptor Example: An Integer Field Validator

A descriptor lets you **customize the behavior of the dot (`.`) operator** for attribute access. It's a powerful way to reuse validation or logic across multiple attributes.

**Basic `Point` class WITHOUT a descriptor:**
```python
class Point:
    def __init__(self, x, y):
        # The dot operator here just stores values in the instance dictionary
        self.x = x
        self.y = y

p = Point(10, 20)
print(p.x)  # 10 - Simple attribute lookup from `p.__dict__`
```

**Creating a descriptor for integer validation:**

```python
class IntegerField:
    def __set_name__(self, owner, name):
        # Stores the attribute name (e.g., 'x' or 'y')
        self.storage_name = name

    def __get__(self, instance, owner):
        if instance is None:
            return self
        # Custom get logic: fetch from the instance's dictionary
        return instance.__dict__.get(self.storage_name)

    def __set__(self, instance, value):
        # Custom set logic: validate type before storing
        if not isinstance(value, int):
            raise TypeError('must be an integer')
        # Store the validated value in the instance's dictionary
        instance.__dict__[self.storage_name] = value
```


> [!question]- What is the need of `if instance is None: return self` in `__get__` ?
> 
> That line handles the case when the descriptor is accessed **on the class itself** instead of on an instance. Here's what happens:
> 
> ## 1. **Instance Access (Normal Case)**
> ```python
> p = Point(10, 20)
> print(p.x)  # Access through instance
> ```
> - `instance` = `p` (the Point object)
> - `owner` = `Point` (the class)
> - **Result:** The descriptor returns the value from the instance's dictionary.
> 
> ## 2. **Class Access (Special Case)**
> ```python
> print(Point.x)  # Access through class
> ```
> - `instance` = `None` (no instance involved)
> - `owner` = `Point` (the class)
> - **Result:** The descriptor returns **itself** (`self`)
> 
> ## Why This Matters:
> 
> ### ✅ **For Inspection and Debugging**
> ```python
> # You can inspect the descriptor object itself
> print(Point.x)          # <__main__.IntegerField object at 0x...>
> print(type(Point.x))    # <class '__main__.IntegerField'>
> ```
> 
> ### ✅ **Required for Some Built-in Functions**
> ```python
> # Functions like getattr() work properly on the class
> desc = getattr(Point, 'x')  # Gets the descriptor object
> print(desc)                 # <__main__.IntegerField object at 0x...>
> ```
> 
> ### ✅ **Essential for Documentation Tools**
> Documentation generators (like Sphinx) and IDEs need to access the descriptor directly to:
> - Show its type
> - Access its docstring
> - Understand its interface
> 
> ## What Happens Without It?
> ```python
> class BadDescriptor:
>     def __get__(self, instance, owner):
>         # Missing: if instance is None: return self
>         return instance.__dict__.get(self.storage_name)
> 
> class Point:
>     x = BadDescriptor()
>     
> print(Point.x)  # ❌ AttributeError: 'NoneType' object has no attribute '__dict__'
> ```
> 
> ## Practical Example:
> ```python
> class LoggedAccess:
>     def __get__(self, instance, owner):
>         if instance is None:
>             # Accessed via class: return descriptor for inspection
>             return self
>         print(f"Accessing attribute from {instance}")
>         return instance.__dict__.get(self.name)
>     
>     def __set_name__(self, owner, name):
>         self.name = name
> 
> class MyClass:
>     attr = LoggedAccess()
> 
> # Class access - gets descriptor object
> print(MyClass.attr)  # <__main__.LoggedAccess object at 0x...>
> 
> # Instance access - triggers logging
> obj = MyClass()
> obj.attr = "value"
> print(obj.attr)      # Prints: "Accessing attribute from <MyClass object>"
>                     # Returns: "value"
> ```
> 
> **Bottom line:** The `if instance is None: return self` pattern is a **standard descriptor convention** that ensures your descriptor works correctly in **both instance and class contexts**. It's not strictly required for basic functionality, but omitting it will break class-level access and various tools that inspect classes.

**Using the descriptor in a class:**

```python
class Point:
    # These class attributes are instances of the descriptor
    x = IntegerField()
    y = IntegerField()

    def __init__(self, x, y):
        # These assignments trigger IntegerField.__set__()
        self.x = x  # ✅ Validates that x is an int
        self.y = y  # ✅ Validates that y is an int

# Usage
p = Point(10, 20)   # Works
print(p.x)          # 10 (triggers IntegerField.__get__)
p.x = 30            # Works (triggers IntegerField.__set__)
p.x = "thirty"      # ❌ Raises TypeError: must be an integer
```

**How this is metaprogramming:** The descriptor **replaces the default behavior** of `self.x = value` and `p.x`. When those lines run, Python's attribute lookup protocol finds the `IntegerField` descriptor and calls its `__set__` or `__get__` methods instead of directly accessing the instance's dictionary.

## 📌 Summary: The Metaprogramming Connection

| Technique | What it Modifies | Metaprogramming Principle |
| :--- | :--- | :--- |
| **Decorator** | Modifies the **behavior of a function or class**. | Wraps code around existing code to change runtime behavior. |
| **Descriptor** | Modifies the **behavior of the dot (`.`) operator** for attribute access. | Intercepts attribute get/set operations with custom logic. |

**Common Goal:** Both techniques let you write **reusable, centralized code** that changes how other parts of your program behave, fulfilling the core idea of metaprogramming: **code that writes or manipulates code**.
