# Delegating to Parent Classes in Python

Let's dive into this tutorial on **delegating back to the parent class**, a key concept in object-oriented programming. When overriding methods, we often need to **call a method from the parent class**—this is what we mean by delegating upwards in the ancestry hierarchy. It starts with the parent class and, in cases of multiple inheritance, might extend to sibling classes. Essentially, we’re passing responsibility **upwards from our current class**.

A classic example is the `__init__` method. Imagine a `Person` class with an initializer that takes `name` and `age` and sets them as instance attributes:

```python
class Person:
    def __init__(self, name, age):
        self.name = name
        self.age = age
```

Now, we create a `Student` class that inherits from `Person`. A student is a person, so it needs `name` and `age`, but it also has a `major`. One approach could be:

```python
class Student(Person):
    def __init__(self, name, age, major):
        self.name = name
        self.age = age
        self.major = major
```

This works—any method in `Person` called by a `Student` instance would have access to `name` and `age`. But it’s not ideal. **Repeating the parent’s initialization logic defeats the purpose of inheritance**, which is to reduce repetitive code. Instead, we **delegate to the parent’s `__init__`** using `super()`:

```python
class Student(Person):
    def __init__(self, name, age, major):
        super().__init__(name, age)
        self.major = major
```

Here, `super()` gives us a reference to the parent class (`Person`), letting us call its `__init__` method as defined, even if we’ve overridden it in `Student`. The method remains **bound to the current instance**, so it operates on the `Student` object we’re creating.

- **Avoid Repetition**: No need to rewrite self.name = name.
- **Key Point**: **Super() calls the parent’s method**.

## Delegation Up the Chain

Consider another example with a `sing` method:

```python
class Person:
    def sing(self):
        return "I'm a lumberjack and I'm okay"

class Student(Person):
    def sing(self):
        parent_sing = super().sing()
        return parent_sing + "\nI sleep all night and I work all day"
```

When we call `sing()` on a `Student` instance, it delegates to `Person`’s `sing`, gets "I'm a lumberjack and I'm okay," and appends the extra lyrics. **Be cautious**: using `self.sing()` instead of `super().sing()` would call the `Student`’s `sing` again, causing **infinite recursion** and a crash. Always use `super()` to access the parent’s version safely.

This delegation works up the inheritance chain. Suppose `Student` doesn’t define `sing`, but a `MusicStudent` does:

```python
class Student(Person):
    pass

class MusicStudent(Student):
    def sing(self):
        return super().sing() + "\nI sleep all night and I work all day"
```

Calling `sing()` on a `MusicStudent` instance still finds `Person`’s `sing` through `Student`, because `super()` follows the hierarchy. The output? "I'm a lumberjack and I'm okay\nI sleep all night and I work all day."

- **Path**: MusicStudent → Student → Person.sing.
- **Key Point**: **Super() finds the method wherever it’s defined**.

## Order Matters with __init__

Back to `__init__`, **order matters**. Typically, call `super().__init__()` first:

```python
class Student(Person):
    def __init__(self, name, age, major):
        super().__init__(name, age)
        self.major = major
```

Why? If the parent’s `__init__` sets something you’ve already set, it could overwrite your work. For example:

```python
class Person:
    def __init__(self, name, age):
        self.name = name
        self.age = age
        self.major = "N/A"

class Student(Person):
    def __init__(self, name, age, major):
        self.major = major  # Set first
        super().__init__(name, age)  # Then parent overrides it
```

Here, `major` ends up as "N/A" instead of the intended value. **==Call `super().__init__()` first==** unless you need to compute something beforehand—then just be mindful of overrides.

- **Why**: Parent might override your attributes.
- **Exception**: Precompute args before super() if needed.
- **Takeaway**: **Order impacts outcome—usually delegate first**.

## Binding and Delegation

**Method binding is crucial**: When delegating, the parent method is still bound to the current instance. Look at this:

```python
class Person:
    def hello(self):
        print("In Person class:", self)

class Student(Person):
    def hello(self):
        print("In Student class:", self)
        super().hello()

p = Person() 
s = Student() 

p.hello() # In Person: <__main__.Person object at 0x...> 
s.hello() 
# In Student: <__main__.Student object at 0x...> 
# In Person: <__main__.Student object at 0x...>
```

For a `Student` instance, `super().hello()` prints "In Person class" but with the `Student` instance as `self`. The binding sticks to the instance that started the call.

- **Binding**: `super().hello()` uses `Person.hello`, but self is `s`.
- **Key Point**: **Instance dictates binding, not method location**.


## Overrides in Delegated Calls

Finally, consider overridden methods in a chain:

```python
class Person:
    def wake_up(self):
        print("Person awakes")
    def do_work(self):
        print("Person works")
    def sleep(self):
        print("Person sleeps")
    def routine(self):
        self.wake_up()
        self.do_work()
        self.sleep()

class Student(Person):
    def do_work(self):
        print("Student studies")
    def routine(self):
        super().routine()
        print("But not before a quick game")
```

A `Person`’s `routine()` prints awake, work, sleep. A `Student`’s `routine()` calls the parent’s `routine` but uses `Student`’s `do_work`, yielding: "Person awakes\nStudent studies\nPerson sleeps\nBut not before a quick game." **The parent’s method uses the instance’s overrides** where they exist.
