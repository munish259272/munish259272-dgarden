# Solving Repetition with Descriptors

## The Problem: Repetitive Properties

Imagine a `Point2D` class needing integer coordinates:

```python
class Point2D:
    def __init__(self, x, y):
        self.x = x  # No type check
        self.y = y
```

- **Issue**: Bare attributes don’t enforce `int`.
- **Fix with Properties**:

```python
class Point2D:
    def __init__(self, x, y):
        self.x = x
        self.y = y
    @property
    def x(self):
        return self._x
    @x.setter
    def x(self, value):
        self._x = int(value)
    @property
    def y(self):
        return self._y
    @y.setter
    def y(self, value):
        self._y = int(value)

p = Point2D(100.1, 200.2)
print(p.x, p.y)  # 100 200
```

- **Downside**: **Repetitive boilerplate** for `x` and `y`.

---

## A Better Way: Descriptors

Create a reusable `IntegerValue` class:

```python
class IntegerValue:
    def get(self):
        return instance._value
    def set(self):
        instance._value = int(value)
    def __init__(self, value=None):
	    if value:
		    self.set(value)

class Point2D:
    x = IntegerValue()
    y = IntegerValue()
    def __init__(self, x, y):
        self.x = x
        self.y = y

p = Point2D(100.1, 200.2)
print(p.x, p.y)  # 100 200
```

- **Goal**: Define `x` and `y` as class attributes that bind to instances.
	- `x = IntegerValue` → this is meant to be bound to instances at run-time
	- `p.x` → use these get and set methods of the `IntegerValue` instance
- **Problem**: This won’t work yet—needs the **descriptor protocol**. (Since `p.x` and `p.y` shadow the class attributes.)

---

## The Descriptor Protocol

Descriptors use special methods:

1. **`__get__(self, instance, owner)`**: Fetches the value (like `@property`).
2. **`__set__(self, instance, value)`**: Sets the value (like `@setter`).
3. **`__delete__(self, instance)`**: Deletes the attribute.
4. **`__set_name__(self, owner, name)`**: Sets the attribute name (Python 3.6+).

- **Fix**:

```python
class IntegerValue:
    def __set__(self, instance, value):
        instance.__dict__[self.name] = int(value)
    def __get__(self, instance, owner):
        return instance.__dict__[self.name]
    def __set_name__(self, owner, name):
        self.name = f"_{name}"
```

- **Key Point**: **Descriptors bind class attributes to instance data**.


> [!question]- Does __get__ and __delete__ make a data descriptor ?
> No, a descriptor is not made solely by implementing `__get__` and `__delete__`.
> 
> A **data descriptor** must implement either `__set__` or `__delete__` (or both). It's the presence of at least one of these methods that distinguishes a data descriptor from a non-data descriptor. A **non-data descriptor** only implements `__get__`.
> 
> ---
> 
> ### Key Differences
> 
> - **Data Descriptor**: Has `__get__`, and at least one of `__set__` or `__delete__`.
>     
> - **Non-data Descriptor**: Only has `__get__`.
>     
> 
> The distinction is crucial for Python's attribute lookup rules. When an attribute is accessed on an object, the lookup process gives precedence to data descriptors. This means if both a data descriptor and an entry in the instance's `__dict__` have the same name, the data descriptor's `__get__` method will be called. For non-data descriptors, the lookup process checks the instance's `__dict__` first, and the non-data descriptor's `__get__` is only called if the attribute isn't found there.

---

## Descriptor Types

Two categories:

- **Non-Data Descriptors**: Only `__get__` (e.g., methods).
- **Data Descriptors**: `__get__` + `__set__` and/or `__delete__` (e.g., properties, slots).

- **Why It Matters**: Affects attribute lookup priority (later!).

## Example: Non-Data Descriptor

A read-only UTC time descriptor:

```python
from datetime import datetime, timezone

class TimeUTC:
    def __get__(self, instance, owner):
        return datetime.now(timezone.utc).isoformat()

class Logger:
    current_time = TimeUTC()

l = Logger()
print(l.current_time)  # e.g., 2025-03-30T12:34:56.789012
```

- **How**: `l.current_time` calls `__get__` automatically.
- **Takeaway**: **Non-data descriptors compute values on access**.


> [!question]- What does `isoformat` in `datetime.now(timezone.utc).isoformat()` does ?
> 
> 
> The `isoformat()` method in Python's `datetime` module is used to convert a `datetime` object into a string in ISO 8601 format. ISO 8601 is an internationally accepted way to represent dates and times, and it provides a standardized way to convey this information.
> 
> Here's how `isoformat()` works:
> 
> - **Date and Time Representation**: The method returns the date and time in the format `YYYY-MM-DDTHH:MM:SS.ssssss+HH:MM`, where:
>   - `YYYY-MM-DD` is the year, month, and day.
>   - `T` is a separator indicating the start of the time component.
>   - `HH:MM:SS.ssssss` is the hour, minute, second, and microsecond.
>   - `+HH:MM` or `-HH:MM` represents the UTC offset (time zone difference from UTC).
> 
> For example, if you call:
> 
> ```python
> from datetime import datetime, timezone
> 
> now_utc = datetime.now(timezone.utc)
> iso_string = now_utc.isoformat()
> print(iso_string)
> ```
> 
> You might see output like:
> 
> ```
> 2025-03-30T12:34:56.789012+00:00
> ```
> 
> In this example:
> - The date is March 30th, 2025.
> - The time is 12 hours, 34 minutes, and 56.789012 seconds.
> - The UTC offset is +00:00, indicating that it's in Coordinated Universal Time (UTC) without any offset.
> 
> Using `isoformat()` ensures that your date-time strings are standardized and easily interpretable across different systems and locales.
