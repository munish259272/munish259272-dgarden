# Mastering Exception Handling

## The `try` Statement

Interrupt exception **propagation** with `try`:

```python
try:
    result = 1 / 0  # Guarded code
except ZeroDivisionError:
    print("Caught division by zero")
```

- **Purpose**: Protect code and handle issues.
- **Key Point**: **`try` guards specific actions**.

---

## Full `try` Structure

Four clauses for **flexible handling**:

- **`try`**: Guards code.
- **`except`**: Catches exceptions:
  ```python
  try:
      l = [1]
      l[5]
  except IndexError as e:
      print(f"Error: {e}")
  ```
- **`finally`**: Always runs:
  ```python
  try:
      f = open("file.txt")
  finally:
      f.close()  # Guaranteed cleanup
  ```
- **`else`**: Runs if no exception:
  ```python
  try:
      x = 1
  except ValueError:
      print("Error")
  else:
      print("No issues")  # Only if try succeeds
  ```

- **Rules**:
  - `else` requires `except`.
  - Keep `try` **short** for clarity.
- **Key Point**: **Each clause serves a purpose**.

---

## Best Practices

- **Short `try`**: Guard only what you can handle.
- **Actionable**: Catch only fixable exceptions.
- **Key Point**: **Precision improves recovery**.

---

## Clause Flexibility

Customize `except` for **multiple scenarios**:

- **Optional**: Zero or more `except` clauses.
- **Multiple Types**:
  ```python
  try:
      x = l[5] + "a"
  except IndexError:
      print("Bad index")
  except TypeError:
      print("Bad operation")
  ```

- **Order Matters**: Specific to broad:
  ```python
  try:
      l[5]
  except LookupError:
      print("Lookup issue")  # Catches IndexError too
  except IndexError:
      print("Index issue")   # Never reached
  ```
- **Key Point**: **Hierarchy dictates catch order**.

### Grouping Exceptions
Handle multiple types together:

```python
try:
    x = l[5] + "a"
except (IndexError, TypeError) as e:
    print(f"Error: {e}")
```

- **Benefit**: Shared handling, less code.
- **Key Point**: **Group for efficiency**.

---

## Bare `except` Risks

Catch everything with no type:

```python
try:
    l[5]
except:
    print("Something happened")
```

- **Problem**: Too broad, hides intent.
- **Use Case**: Logging then re-raising:
  ```python
  import sys
  try:
      l[5]
  except:
      print("Logging:", sys.exc_info()[0])
      raise  # Continues propagation
  ```
- ⚠️ **Caution**: Avoid unless specific need.
- **Key Point**: **Bare catches need purpose**.

---

## Exception Info

Access details with `sys.exc_info()`:

```python
import sys
try:
    l[5]
except:
    exc_type, exc_value, traceback = sys.exc_info()
    print(f"Type: {exc_type}, Value: {exc_value}, Traceback: {traceback}")
```

- **Returns**: Tuple (type, value, traceback).
- **Key Point**: **Unpacks exception context**.

---

## Exception Properties

All inherit from `BaseException`:

- **`args`**: Init arguments (e.g., error message).
- **`__traceback__`**: Stack trace object.
- **Trace Visualization**:
  ```python
  import traceback
  try:
      raise ValueError("Oops")
  except ValueError as e:
      traceback.print_exc()  # Formats stack trace
  ```
- **Key Point**: **Standard attributes aid debugging**.

---

## `finally` Guarantees

Runs **always**, exception or not:

```python
try:
    raise ValueError("Test")
finally:
    print("Cleanup")  # Executes before propagation
```

- **Use**: Cleanup (e.g., closing files).
- **Key Point**: **`finally` ensures closure**.

---

## Nested Handling

`try` can go anywhere:

```python
try:
    l[5]
except IndexError:
    try:
        print(l + "a")
    except TypeError:
        print("Nested catch")
```

- **Scope**: `except`, `else`, `finally` too.
- **Key Point**: **Nesting refines control**.

---

## EAFP vs. LBYL

Python favors **EAFP** (Easier to Ask Forgiveness):

```python
# EAFP (Pythonic)
try:
    with open("file.txt") as f:
        f.read()
except FileNotFoundError:
    print("File missing")
```

- **Contrast**: LBYL (Look Before You Leap):
  ```python
  import os
  if os.path.exists("file.txt") and os.path.isfile("file.txt"):
      with open("file.txt") as f:
          f.read()
  ```
- **Why EAFP**: Simpler, handles more cases.
- **Key Point**: **Try first, handle later**.
