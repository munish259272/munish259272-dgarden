# Beyond the Property Class

## Can We Skip the Property Class?

Yes, you can define properties without the `property` class or its decorators! **The `property` class is a convenience**, not a requirement. It’s user-friendly and works well most of the time, but alternatives exist. We’ll explore those later (hint: data descriptors). For now, let’s stick with `property`.

---

## Where `property` Falls Short

### Reusability Challenges

Imagine needing the same property logic across multiple classes—like an email validator:

```python
class User:
    def __init__(self, email):
        self._email = email
    
    @property
    def email(self):
        return self._email
    
    @email.setter
    def email(self, value):
        if "@" not in value:
            raise ValueError("Invalid email")
        self._email = value

class Admin:
    def __init__(self, email):
        self._email = email
    
    @property
    def email(self):
        return self._email
    
    @email.setter
    def email(self, value):
        if "@" not in value:
            raise ValueError("Invalid email")
        self._email = value
```

- **Problem**: Duplicate code for `email` validation.
- **Workaround**: Define the getter/setter outside and reuse:

```python
def get_email(self):
    return self._email

def set_email(self, value):
    if "@" not in value:
        raise ValueError("Invalid email")
    self._email = value

class User:
    def __init__(self, email):
        self._email = email
    email = property(get_email, set_email)
```

- **Still Clunky**: Better, but not ideal.

**Key Point**: **Reusability is limited** with `property`—you rewrite or awkwardly share methods.

### Same-Class Duplication

Within one class, multiple properties might need identical validation:

```python
class Person:
    def __init__(self, first, last):
        self._first = first
        self._last = last
    
    @property
    def first(self):
        return self._first
    @first.setter
    def first(self, value):
        if not value:
            raise ValueError("Name cannot be empty")
        self._first = value
    
    @property
    def last(self):
        return self._last
    @last.setter
    def last(self, value):
        if not value:
            raise ValueError("Name cannot be empty")
        self._last = value
```

- **Issue**: Repeated validation logic for `first` and `last`.

**Takeaway**: **Duplication creeps in** without reusable solutions.

### More Control

Sometimes, you need **advanced control** over property behavior—beyond what `property` offers. This ties into **data descriptors**, which we’ll cover later. For now, know `property` is a simplified wrapper.

## Class-Level Properties?

Can we create **class properties** (bound to the class, not instances) with accessor methods? Yes! Think class attributes, but managed:

```python
class MyClass:
    _value = "class-level"
    
    @property
    def value(self):
        return self._value  # Instance-bound, not class-bound
```

- **Catch**: `@property` makes instance properties. For true class properties, you need **metaprogramming**—advanced stuff we’ll tackle later.

**Key Point**: **Class properties are possible**, but require deeper techniques.
