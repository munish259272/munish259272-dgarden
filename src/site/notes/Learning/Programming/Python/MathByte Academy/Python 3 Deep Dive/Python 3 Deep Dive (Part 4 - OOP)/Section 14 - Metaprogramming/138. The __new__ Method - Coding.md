# Tutorial: Mastering `__new__` - Object Creation and Inheritance

This tutorial covers the `__new__` method in depth, showing how it controls **object creation**, works with **inheritance**, and differs from `__init__`.

## 📌 Key Difference: `__new__` vs `__init__`

| | `__new__` | `__init__` |
| :--- | :--- | :--- |
| **Purpose** | Creates the object (allocates memory) | Initializes the object (sets attributes) |
| **First Argument** | `cls` (the class to create) | `self` (the created instance) |
| **Returns** | Must return the new object | Returns `None` (initializes in-place) |
| **Called When** | First, to **create** the instance | Second, to **initialize** the instance |
| **Inheritance** | Can be safely overridden | Cannot be overridden for built-in types |

## Basic `__new__` Example

```python
class Point:
    def __new__(cls, x, y):
        print(f"__new__ called for {cls.__name__} with ({x}, {y})")
        # Delegate actual creation to parent class
        instance = super().__new__(cls)
        return instance
    
    def __init__(self, x, y):
        print(f"__init__ called with ({x}, {y})")
        self.x = x
        self.y = y

# Creating an instance
p = Point(3, 4)
# Output:
# __new__ called for Point with (3, 4)
# __init__ called with (3, 4)
```

## ⚠️ Critical: Always Use `super().__new__()` for Inheritance

```python
class Person:
    def __new__(cls, name):
        print(f"Person.__new__ called for {name}")
        # ❌ WRONG: Breaks inheritance chain
        # instance = object.__new__(cls)
        
        # ✅ CORRECT: Maintains inheritance
        instance = super().__new__(cls)
        return instance
    
    def __init__(self, name):
        print(f"Person.__init__ called for {name}")
        self.name = name

class Student(Person):
    def __new__(cls, name, major):
        print(f"Student.__new__ called for {name}, {major}")
        # This ensures Person.__new__ is called too
        instance = super().__new__(cls, name)
        return instance
    
    def __init__(self, name, major):
        super().__init__(name)
        self.major = major

# Testing
s = Student("John", "Computer Science")
# Output:
# Student.__new__ called for John, Computer Science
# Person.__new__ called for John
# Person.__init__ called for John
```

**Why `super()` matters:** If `Student.__new__` used `object.__new__()`, it would skip `Person.__new__` entirely, potentially missing important setup logic.

## 🎯 Special Case: Inheriting from Built-in Types

When inheriting from built-in types (`int`, `str`, `list`, etc.), **you must override `__new__`, not `__init__`:**

```python
class SquaredInt(int):
    def __new__(cls, value):
        # Calculate the squared value BEFORE creating the int
        squared_value = value ** 2
        # Delegate to int.__new__ with the squared value
        return super().__new__(cls, squared_value)
    
    # ❌ __init__ won't work for built-in types
    # def __init__(self, value):
    #     super().__init__(value ** 2)  # TypeError!

# Usage
num = SquaredInt(4)
print(num)          # 16
print(type(num))    # <class '__main__.SquaredInt'>
print(isinstance(num, int))  # True
```

**Reason:** Built-in types in C often do all their setup in `__new__` and have minimal or no `__init__` implementation.

## Using `__new__` to Inject Methods or Attributes

You can modify the class or instance during creation:

```python
class Square:
    def __new__(cls, width, length):
        # Add a method to the CLASS during creation
        cls.area = lambda self: self.w * self.l
        
        # Create the instance
        instance = super().__new__(cls)
        
        # Initialize attributes directly (no __init__ needed)
        instance.w = width
        instance.l = length
        
        return instance

# Usage
s = Square(2, 3)
print(s.area())     # 6
print(s.__dict__)   # {'w': 2, 'l': 3}
```

## The `__init__` Call Rule

**Python only calls `__init__` if `__new__` returns an instance of the correct type:**

```python
class Person:
    def __new__(cls, name):
        print("__new__ called")
        # Returning a string instead of a Person instance
        return f"Not a Person: {name}"
    
    def __init__(self, name):
        print("__init__ called - This will NEVER execute!")
        self.name = name

p = Person("Alice")
print(p)            # "Not a Person: Alice"
print(type(p))      # <class 'str'>
# __init__ was NOT called because __new__ didn't return a Person!
```

## Manual Object Creation (Advanced)

You can manually replicate what Python does:

```python
class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

# Manual creation (what Python does internally)
# 1. Call __new__ to create the object
p = Point.__new__(Point, 10, 20)  # p is an "empty" Point instance

# 2. Only call __init__ if __new__ returned a Point
if isinstance(p, Point):
    Point.__init__(p, 10, 20)     # Now initialize it

print(p.x, p.y)  # 10 20
```

## When to Use `__new__` vs `__init__`

### ✅ Use `__init__` when:
* Setting initial attribute values
* Basic instance setup
* **95% of normal use cases**

### ✅ Use `__new__` when:
1. **Inheriting from built-in types**
2. **Implementing singletons or object pools**
3. **Creating immutable objects** (need to set values before `__init__`)
4. **Injecting methods/attributes into the class during creation**
5. **Controlling what type of object gets returned**

## Singleton Pattern Example

```python
class Singleton:
    _instance = None
    
    def __new__(cls):
        if cls._instance is None:
            print("Creating new singleton instance")
            cls._instance = super().__new__(cls)
        else:
            print("Returning existing singleton instance")
        return cls._instance
    
    def __init__(self):
        # Called each time, but same instance
        print("Singleton initialized")

# Test
s1 = Singleton()  # Creating new... Singleton initialized
s2 = Singleton()  # Returning existing... Singleton initialized
print(s1 is s2)   # True
```

## Summary

1. **`__new__` creates, `__init__` initializes**
2. **Always use `super().__new__(cls)`** for proper inheritance
3. **For built-in types, override `__new__`, not `__init__`**
4. **`__init__` only runs if `__new__` returns correct type**
5. **You can do all initialization in `__new__`** and skip `__init__`
6. **Think carefully before overriding `__new__`** - `__init__` is usually simpler


> [!question]- Why we need to use `__new__` when inheriting from builtin types and not `__init__` ?
> 
> The reason lies in **how built-in types (written in C) are implemented** versus regular Python classes. Here's the detailed breakdown:
> 
> ## 1. **Built-in Types Implement Most Logic in `__new__`**
> 
> Built-in types like `int`, `str`, `list`, `tuple`, etc., are implemented in C and follow a different pattern:
> 
> ```python
> # For built-in int type, the REAL implementation looks conceptually like:
> class int:
>     def __new__(cls, value):
>         # C code: Allocate memory, convert value to integer
>         # C code: Set the internal integer value
>         return integer_object
>     
>     def __init__(self, value):
>         # Often empty or minimal in C implementation
>         # Cannot modify the already-set value!
>         pass
> ```
> 
> **Key Insight:** For immutable built-in types, **the value must be set during object creation (`__new__`)** because once created, it cannot be changed.
> 
> ## 2. **Immutable Types CANNOT Be Modified After Creation**
> 
> Consider `int` - it's immutable:
> ```python
> x = 5
> # There's NO way to change x's value after creation
> # x.value = 10  # Doesn't exist!
> ```
> 
> Since `int` objects are immutable:
> - The integer value **must** be set in `__new__` (during creation)
> - `__init__` cannot modify it afterward (would violate immutability)
> 
> ## 3. **Practical Example: What Happens**
> 
> ```python
> class MyInt(int):
>     def __new__(cls, value):
>         # Modify value BEFORE object creation
>         modified_value = value * 2
>         return super().__new__(cls, modified_value)
>     
>     # ❌ This WON'T work
>     def __init__(self, value):
>         # Too late! The int value was already set in __new__
>         # Cannot change an immutable int
>         super().__init__(value * 2)  # Error or does nothing
> 
> num = MyInt(5)
> print(num)  # 10 (works with __new__ override)
> ```
> 
> ## 4. **The `__init__` Problem with Built-ins**
> 
> ```python
> class BadInt(int):
>     def __init__(self, value):
>         print("Trying to call int.__init__")
>         # This either:
>         # 1. Doesn't exist (for some built-ins)
>         # 2. Takes different arguments
>         # 3. Does nothing
>         super().__init__(value * 2)  # Usually raises TypeError!
> 
> # ❌ Typical error:
> # TypeError: object.__init__() takes exactly one argument (the instance to initialize)
> # OR
> # TypeError: int.__init__() takes no arguments
> ```
> 
> **Why this error?** Many built-in types' `__init__` methods:
> - Take **no arguments** (just `self`)
> - Are designed **not to modify** the object (for immutability)
> - May not even be callable from Python
> 
> ## 5. **Visualizing the Object Creation Flow**
> 
> ```
> Regular Python class:
> 1. __new__(cls, args) → creates empty object
> 2. __init__(self, args) → sets attributes on empty object
> 
> Built-in int class:
> 1. __new__(cls, value) → creates AND sets the integer value
> 2. __init__(self) → (optional) minimal setup, takes no args
> ```
> 
> ## 6. **Example with Mutable Built-in: `list`**
> 
> Even mutable built-ins often follow this pattern:
> 
> ```python
> class DoubleList(list):
>     def __new__(cls, iterable):
>         # Double each element BEFORE list creation
>         doubled = [x * 2 for x in iterable]
>         return super().__new__(cls, doubled)
>     
>     # ❌ This is problematic
>     def __init__(self, iterable):
>         # List is already created with original values
>         # Can't easily replace them
>         super().__init__(x * 2 for x in iterable)  # Would append, not replace
> 
> # Using __new__
> dl = DoubleList([1, 2, 3])
>
> print(dl)  # [2, 4, 6] ✓
> 
> # Using only __init__ would give: [1, 2, 3, 2, 4, 6] (wrong!)
> ```
> 
> ## 7. **When CAN You Use `__init__` with Built-ins?**
> 
> Only when you **don't need to modify the initial value**:
> 
> ```python
> class LoggedInt(int):
>     def __init__(self, value):
>         # Can add attributes, but CAN'T change the int value
>         self.created_at = datetime.now()
>         # The int value was already set in __new__
>         super().__init__()  # No arguments!
>     
>     # Still need __new__ to ensure proper creation
>     def __new__(cls, value):
>         return super().__new__(cls, value)
> 
> num = LoggedInt(5)
> print(num)           # 5
> print(num.created_at) # timestamp
> ```
> 
> ## 📌 Summary: The Golden Rules
> 
> 1. **`__new__` sets up the CORE VALUE** of built-in types
> 2. **`__init__` is for ADDITIONAL SETUP** (adding attributes)
> 3. **For immutable types**, value MUST be set in `__new__`
> 4. **Always override `__new__`** when inheriting from built-ins if you need to modify the initial value
> 5. **You can override both** if you need to modify value AND add attributes
> 
> ```python
> # Correct pattern for built-in inheritance:
> class CustomBuiltin(BuiltinType):
>     def __new__(cls, *args):
>         # 1. Process arguments
>         # 2. Call super().__new__ with processed values
>         return super().__new__(cls, processed_args)
>     
>     def __init__(self, *args):
>         # 3. Add additional attributes if needed
>         self.custom_attr = something
>         super().__init__()  # Often no arguments
> ```
