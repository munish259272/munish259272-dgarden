# Refining Descriptor Storage

## WeakKeyDictionary Recap

Fix memory leaks with `WeakKeyDictionary`:

```python
from weakref import WeakKeyDictionary

class IntegerValue:
    def __init__(self):
        self.values = WeakKeyDictionary()
    def __set__(self, instance, value):
        self.values[instance] = int(value)
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return self.values.get(instance)

class Point:
    x = IntegerValue()

p = Point()
p_id = id(p)
p.x = 10
print(p.x)               # 10
print(Point.x.values)    # {<p>: 10}
del p                    # Last strong ref gone
print(Point.x.values)    # {} (auto-cleaned)
```

- **How**: Keys are weak refs, auto-removed when objects die.
- **Pros**: No leaks, works with slots.
- **Limit**: **Requires hashable objects**.
- **Key Point**: **Simple and clean for hashable cases**.

## Handling Unhashable Objects with IDs

Use IDs for non-hashable instances:

```python
class IntegerValue:
    def __init__(self):
        self.values = {}
    def __set__(self, instance, value):
        self.values[id(instance)] = int(value)
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return self.values.get(id(instance))

class Point:
    x = IntegerValue()
    def __init__(self, x):
        self.x = x   # calls IntegerValue.__set__()
    def __eq__(self, other):
        return isinstance(other, Point) and self.x == other.x

p = Point(10.1)
print(type(p.x), p.x) # <class 'int'> 10
p.x = 20.2
print(type(p.x), p.x) # <class 'int'> 20
print(Point.x.values) # {id(p): 20}
del p                 # Collected, but entry lingers
print(Point.x.values) # {id(p): 20} still there
```

- **Why**: `id(instance)` is always hashable.
- **Issues**:
  - Dead entries remain.
  - ⚠️ **Warning**: **ID reuse risks confusion becuase memory is limited and python might use this memory address for storing something else later ** (rare).
- **Takeaway**: **IDs work but need cleanup**.

## Adding Weak Ref Callbacks

Clean up with callbacks:

```python
import weakref

class IntegerValue:
    def __init__(self):
        self.values = {}
    def remove_object(self, weakref):
        key = [k for k, (wr, _) in self.values.items() if wr is weakref][0]
        del self.values[key]
    def __set__(self, instance, value):
        self.values[id(instance)] = (weakref.ref(instance, self.remove_object), int(value))
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return self.values[id(instance)][1]

class Point:
    x = IntegerValue()

p1 = Point()
p2 = Point()
p1.x = 10.1
p2.x = 100.1
print(p1.x, p2.x)     # 10 100
print(Point.x.values) # {140709859826176: (<weakref at 0x7ff98cdaf290; to 'Point' at 0x7ff991361a00>, 10), 140709786942880: (<weakref at 0x7ff98cdaf380; to 'Point' at 0x7ff98cddfda0>, 100)}
del p1                # Callback removes entry
print(Point.x.values) # {140709786942880: (<weakref at 0x7ff98cdaf380; to 'Point' at 0x7ff98cddfda0>, 100)} Only p2’s entry remains
del p2
print(Point.x.values)  # {}
```

- **How**: Store `(weakref, value)` tuples; callback deletes dead entries.
- **Benefits**: Handles unhashable objects, cleans up.
- **Key Point**: **Callbacks ensure no stale data**.

> [!question]- Why does remove_object(self, weakref) needs a second parameter weakref and from where it is getting this weakref becuase i do not see it gettting called anywhere ?
> The `remove_object` method in your code has a `weakref` parameter because it is used as a callback function by the `weakref.ref` object to handle cleanup when a `Point` instance is garbage-collected. Let’s break this down step-by-step to explain why `remove_object` needs the `weakref` parameter and where it comes from, addressing the confusion about it not being explicitly called in your code.
> 
> ### Understanding the Code
> Your `IntegerValue` descriptor uses a dictionary (`self.values`) to store values associated with `Point` instances, using the instance’s `id` as the key and a tuple of `(weakref.ref(instance, self.remove_object), int(value))` as the value. The `weakref.ref` creates a weak reference to the `Point` instance, and `self.remove_object` is passed as a callback to be invoked when the referenced instance is garbage-collected. The `weakref` parameter in `remove_object` is automatically provided by Python’s `weakref` module when the callback is triggered.
> 
> ### Why `remove_object` Needs a `weakref` Parameter
> The `weakref.ref` constructor allows you to specify a callback function that is called when the referenced object (e.g., a `Point` instance) is garbage-collected. The signature of this callback function must accept a single argument: the weak reference object (`weakref.ref`) that points to the now-garbage-collected object. This is why `remove_object` is defined as:
> 
> ```python
> def remove_object(self, weakref):
>     key = [k for k, (wr, _) in self.values.items() if wr is weakref][0]
>     del self.values[key]
> ```
> 
> - **Purpose of `weakref` parameter**: The `weakref` parameter is the weak reference object (created by `weakref.ref`) that was associated with the garbage-collected `Point` instance. The callback uses this to identify which entry in `self.values` corresponds to the deleted instance, allowing it to remove that entry from the dictionary.
> - **Why it’s needed**: When the `Point` instance is garbage-collected, Python calls the callback (`remove_object`) and passes the `weakref.ref` object as an argument. This allows `remove_object` to locate and remove the correct key-value pair from `self.values` by matching the weak reference.
> 
> ### Where Does the `weakref` Parameter Come From?
> You’re correct that `remove_object` is not explicitly called in your code. Instead, it is automatically invoked by Python’s `weakref` module when a `Point` instance is garbage-collected. Here’s how it works:
> 
> 1. **Creating the Weak Reference**:
>    In the `__set__` method, you create a weak reference to the `Point` instance:
>    ```python
>    self.values[id(instance)] = (weakref.ref(instance, self.remove_object), int(value))
>    ```
>    - `weakref.ref(instance, self.remove_object)` creates a weak reference to `instance` (a `Point` object) and registers `self.remove_object` as the callback to be called when `instance` is garbage-collected.
>    - The `weakref.ref` object is stored in `self.values` alongside the `int(value)`.
> 
> 2. **Garbage Collection Triggers the Callback**:
>    When a `Point` instance (e.g., `p1`) is deleted (via `del p1`) and becomes eligible for garbage collection, Python’s garbage collector detects that the object is no longer referenced (except by the weak reference). At this point:
>    - The `weakref.ref` object associated with `p1` becomes “dead” (i.e., it points to `None` because the object is gone).
>    - Python automatically calls the callback function (`remove_object`) registered with the weak reference, passing the `weakref.ref` object as the argument.
> 
> 3. **How `remove_object` Uses the `weakref` Parameter**:
>    Inside `remove_object`, you use the `weakref` parameter to find the corresponding entry in `self.values`:
>    ```python
>    key = [k for k, (wr, _) in self.values.items() if wr is weakref][0]
>    ```
>    - The `weakref` parameter is the same `weakref.ref` object that was stored in `self.values[id(instance)]`.
>    - The list comprehension searches `self.values` for the key where the stored weak reference (`wr`) matches the `weakref` parameter.
>    - Once the key is found, the corresponding entry is deleted from `self.values`.
> 
> ### Why You Don’t See `remove_object` Being Called
> You don’t explicitly call `remove_object` in your code because it’s a callback triggered by Python’s garbage collection mechanism. The `weakref.ref` object is responsible for invoking `remove_object` when the referenced `Point` instance is garbage-collected. This happens automatically when:
> - You execute `del p1`, which removes the reference to `p1`.
> - The Python garbage collector runs and determines that `p1` is no longer reachable (i.e., it has no strong references).
> - The garbage collector then calls `remove_object`, passing the `weakref.ref` object that was created for `p1`.
> 
> ### Example of the Process
> Let’s trace the execution of your code:
> 
> ```python
> p1 = Point()
> p1.x = 10.1  # Stores (weakref.ref(p1, remove_object), 10) in IntegerValue.values[id(p1)]
> p2 = Point()
> p2.x = 100.1  # Stores (weakref.ref(p2, remove_object), 100) in IntegerValue.values[id(p2)]
> print(p1.x, p2.x)  # Outputs: 10 100 (fetches values from IntegerValue.values)
> print(Point.x.values)  # Shows dictionary with entries for p1 and p2
> del p1  # Removes strong reference to p1
> # Garbage collector runs (may not be immediate, but `del` enables it)
> # weakref.ref(p1) detects p1 is gone, calls remove_object(weakref_ref_p1)
> # remove_object finds id(p1) in IntegerValue.values and deletes the entry
> print(Point.x.values)  # Shows only p2’s entry
> del p2  # Same process for p2
> print(Point.x.values)  # Shows empty dictionary {}
> ```
> 
> ### Why Use a Weak Reference and Callback?
> The use of `weakref.ref` and the `remove_object` callback ensures that `IntegerValue.values` doesn’t prevent `Point` instances from being garbage-collected (which would happen if you stored strong references to the instances). The callback cleans up `self.values` when instances are deleted, preventing memory leaks by removing stale entries.
> 
> ### Potential Issues and Notes
> 1. **Non-Immediate Garbage Collection**:
>    Python’s garbage collection may not run immediately after `del p1`, so `remove_object` might not be called instantly. In your example, the output matches the expectation because Python’s garbage collector likely runs before the next `print(Point.x.values)`. You can force garbage collection with `import gc; gc.collect()` to ensure immediate cleanup, but this is usually unnecessary.
> 
> 2. **Thread Safety**:
>    The `remove_object` method assumes that `self.values` isn’t modified concurrently. If your code runs in a multithreaded environment, you might need to add synchronization (e.g., a lock) to avoid race conditions when accessing `self.values`.
> 
> 3. **List Comprehension in `remove_object`**:
>    The line `key = [k for k, (wr, _) in self.values.items() if wr is weakref][0]` assumes there’s exactly one matching key. If `self.values` is empty or corrupted, this could raise an `IndexError`. A safer approach might be:
>    ```python
>    def remove_object(self, weakref):
>        for key, (wr, _) in list(self.values.items()):  # Use list to avoid runtime modification
>            if wr is weakref:
>                del self.values[key]
>                break
>    ```
> 
> ### Summary
> - The `weakref` parameter in `remove_object` is required because it’s the argument passed by Python’s `weakref.ref` when the callback is triggered during garbage collection.
> - It’s not explicitly called in your code; Python’s garbage collector calls it automatically when a `Point` instance is deleted and collected.
> - The `weakref` parameter is the `weakref.ref` object created in `__set__`, allowing `remove_object` to identify and remove the corresponding entry from `self.values`.
>

## Slots and Weak Refs

```python
class Person:
    pass

p = Person()
print(Person.__dict__) # {'__module__': '__main__', '__dict__': <attribute '__dict__' of 'Person' objects>, '__weakref__': <attribute '__weakref__' of 'Person' objects>, '__doc__': None}
print(hasattr(Person.__weakref__, '__get__'), hasattr(Person.__weakref__, '__set__')) # True, True

print(p.__weakref__) # None
wr = weakref.ref(p)
print(p.__weakref__) # <weakref at 0x7ff98cdaf7e0; to 'Person' at 0x7ff98cddf7a0>
```
We see that in a classes dictionary there is an attribute called `__weakref__` and it is a data descriptor because it has `__get__` and `__set__` attributes.

**Problem with slots**: If we add `__slots__` to the class, it is added to the classes dictionary `Person.__dict__` then both `__weakref__` and Person objects dictionary `'__dict__': <attribute '__dict__' of 'Person' objects>` is removed from the dictionary ( as seen below)
```python
import weakref

class Person:
    __slots__ = 'name'

p = Person()
print(Person.__dict__) # {'__module__': '__main__', '__slots__': 'name', 'name': <member 'name' of 'Person' objects>, '__doc__': None}


print(p.__weakref__) # type object 'Person' has no attribute '__weakref__'
print(p.__dict__) # 'Person' object has no attribute '__dict__'
wr = weakref.ref(p) # cannot create weak reference to 'Person' object
```

> [!error] `wr = weakref.ref(p)` gives error <font color="#ff0000">cannot create weak reference to 'Person' object</font> because class `Person` does not have `__weakref__` attribute.

Solution : Add `__weakref__` to the slots:

```python
class Person:
    __slots__ = ('name', '__weakref__',)  # Enable weak refs
    x = IntegerValue()

p = Person()
print(Person.__dict__) # {'__module__': '__main__', '__slots__': ('name', '__weakref__'), 'x': <__main__.IntegerValue object at 0x7ff98cddf500>, 'name': <member 'name' of 'Person' objects>, '__weakref__': <attribute '__weakref__' of 'Person' objects>, '__doc__': None}
print(p.__dict__) # 'Person' object has no attribute '__dict__'

p.x = 10
del p  # Works with callback
```

- **Issue**: Slots remove `__weakref__` unless specified.
- **Fix**: Add `__weakref__` to `__slots__`.
- **Takeaway**: **Slots need explicit weak ref support**.

## Practical Example: ValidString

Reusable descriptor with validation:

```python
class ValidString:
    def __init__(self, min_length=0, max_length=255):
        self.data = {}
        self.min_length = min_length
        self.max_length = max_length
    def finalize_instance(self, weakref):
        key = [k for k, (wr, _) in self.data.items() if wr is weakref][0]
        del self.data[key]
    def __set__(self, instance, value):
        if not isinstance(value, str):
            raise ValueError("Value must be a string")
        if len(value) < self.min_length:
            raise ValueError(f"Value should be at least {self.min_length} characters")
        if len(value) > self.max_length:
            raise ValueError(f"Value cannot exceed {self.max_length} characters")
        self.data[id(instance)] = (weakref.ref(instance, self.finalize_instance), value)
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return self.data[id(instance)][1] if id(instance) in self.data else None

class Person:
    __slots__ = ('__weakref__',)
    first_name = ValidString(1, 100)
    last_name = ValidString(1, 100)


    def __eq__(self, other):
        return (
            isinstance(other, Person) and
            self.first_name == other.first_name and
            self.last_name == other.last_name
        )

class BankAccount:
    __slots__ = '__weakref__',

    account_number = ValidString(5, 255)

    def __eq__(self, other):
        return isinstance(other, BankAccount) and self.account_number == other.account_number
```

- **Usage**:
```python
p = Person()
try:
	p.first_name = "" # Value should be at least 1 characters
except Exception as e:
	print(e)          # Value should be at least 1 characters

p.first_name = "Guido"
p.last_name = "van Rossum"
print(p.first_name, p.last_name)  # Guido van Rossum

print(Person.first_name.data) # {140709786940000: (<weakref at 0x7ff98bb007c0; to 'Person' at 0x7ff98cddf260>, 'Guido')}
print(Person.last_name.data) # {140709786940000: (<weakref at 0x7ff98bb004f0; to 'Person' at 0x7ff98cddf260>, 'van Rossum')}
del p  # Dictionary cleans up
print(Person.first_name.data) # {}
print(Person.last_name.data)  # {}

b1, b2 = BankAccount(), BankAccount()
b1.account_number, b2.account_number = 'Savings', 'Checking'
print(b1.account_number, b2.account_number) # Savings Checking
```

- **Why**: Reusable, validates, works with slots.
- **Key Point**: **Descriptors reduce code duplication**.


> [!question]- Does the statement `p.first_name = "Guido"` invoke only `__set__`, or does it also invoke `__get__` due to the presence of `p.first_name` before the assignment?
> 
> The statement `p.first_name = "Guido"` **only invokes `__set__`**, not `__get__`. Here's why:
> 
> - In Python, an assignment like `p.first_name = "Guido"` directly calls the `__set__` method of the descriptor (`ValidString` in this case) because it’s a data descriptor (it defines both `__get__` and `__set__`). The descriptor protocol prioritizes `__set__` for assignments.
> - The presence of `p.first_name` before the assignment doesn’t trigger `__get__`. Python doesn’t evaluate the left-hand side of an assignment as an expression that needs to be "fetched" first—it’s simply the target of the assignment.
> 
> To confirm:
> 
> ```python
> class ValidString:
>     def __init__(self):
>         self.data = {}
>     def __set__(self, instance, value):
>         print("Calling __set__")
>         self.data[id(instance)] = value
>     def __get__(self, instance, owner):
>         print("Calling __get__")
>         return self.data.get(id(instance))
> 
> class Person:
>     name = ValidString()
> 
> p = Person()
> p.name = "Guido"  # Output: Calling __set__
> ```
> 
> - **Output**: Only `"Calling __set__"` appears.
> - **Key Point**: **Assignment triggers `__set__` alone; `__get__` is called only when retrieving the value (e.g., `print(p.name)`)**.


> [!question]- Does python has order or direction of executing a statement like `p.first_name = "Guido"`  as right to left or left to right ?
> Python executes a statement like `p.first_name = "Guido"` **left to right**, but with a specific twist for assignments due to how the language processes them. Here's the breakdown:
> 
> - **General Execution**: Python typically evaluates expressions from left to right (e.g., in `a + b * c`, it resolves `+` and `*` based on operator precedence, but still processes the tokens left to right).
> - **Assignment Specifics**: For an assignment like `p.first_name = "Guido"`, Python doesn’t "evaluate" the left-hand side (`p.first_name`) as an expression to fetch a value first. Instead:
>   1. **Target Identification**: Python identifies `p.first_name` as the assignment target (a "binding" operation). This involves resolving `p` (the instance) and `first_name` (the attribute name) to locate the descriptor or slot for assignment.
>   2. **Right-Hand Side Evaluation**: Python then evaluates the right-hand side (`"Guido"`), which is a simple string literal here, so it’s immediately available.
>   3. **Assignment Execution**: Finally, it invokes the `__set__` method of the descriptor (if present) or updates the instance’s `__dict__` (if no descriptor), passing the instance (`p`), attribute name (`first_name`), and value (`"Guido"`).
> 
> ### Why It Feels Left-to-Right
> - The left-hand side is "processed" first to determine where the value will go, but it’s not "evaluated" like an expression (no `__get__` is called). Only after that does Python handle the right-hand side and complete the assignment.
> - Example with a descriptor:
> 
> ```python
> class ValidString:
>     def __set__(self, instance, value):
>         print(f"Setting {value} on {instance}")
>         # Store value logic here
> 
> class Person:
>     name = ValidString()
> 
> p = Person()
> p.name = "Guido"  # Output: Setting Guido on <__main__.Person object...>
> ```
> 
> - **Order**: Python resolves `p.name` (finds the descriptor `ValidString`) before dealing with `"Guido"`, then calls `__set__`.
> 
> ### Contrast with Other Languages
> - In some languages (e.g., C), assignment might seem "right to left" because the right-hand value is computed and then stored into the left-hand location. Python’s model is more about **target binding followed by value assignment**, aligning with a left-to-right flow in terms of syntax processing.
> 
> ### Key Point
> - **Python processes `p.first_name = "Guido"` left to right**: It identifies the target (`p.first_name`) first, then evaluates the value (`"Guido"`), and finally performs the assignment. No right-to-left evaluation occurs here.
