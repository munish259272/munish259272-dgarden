# Getters and Setters in Descriptors

## Exploring `__get__`

Start with a non-data descriptor for UTC time:

```python
from datetime import datetime, timezone

class TimeUTC:
    def __get__(self, instance, owner):
        if instance is None:
            return self  # Called from class
        return datetime.now(timezone.utc).isoformat()  # Called from instance

class Logger:
    current_time = TimeUTC()

l = Logger()
print(l.current_time)      # e.g., 2025-03-30T12:34:56.789012
print(Logger.current_time) # <__main__.TimeUTC object at 0x...>
```

- **Signature**: `__get__(self, instance, owner)`
  - `self`: The `TimeUTC` instance.
  - `instance`: The calling instance (`l`) or `None` (from `Logger`).
  - `owner`: The class (`Logger`).
- **Behavior**:
  - **Class access**(`Logger.current_time`): Returns the descriptor itself (`self`).
  - **Instance access**(`l.current_time`): Returns the computed value (UTC time).
- **Key Point**: **`__get__` adapts based on caller**.

## Understanding `__set__`

Now look at the setter:

```python
class IntegerValue:
    def __set__(self, instance, value):
        instance.__dict__["_value"] = int(value)
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return instance.__dict__["_value"]
```

- **Signature**: `__set__(self, instance, value)`
  - `self`: The descriptor instance.
  - `instance`: The calling instance (never `None` becuase typically setters and deleters are always called from instances becuase we want to set property `x` on the instance and not on the class).
  - `value`: The value to set.
- **Difference**: No `owner`—setters are instance-only.
- **Why**: **Setters target instance data**, not class data.

## The Shared Descriptor Pitfall

![[91. Getters and Setters - Lecture.pdf#page=5]]

Class attributes are shared:

```python
class Point2D:
    x = IntegerValue()
    y = IntegerValue()

p1 = Point2D()
p2 = Point2D()
p1.x = 100
print(p1.x, p2.x)  # 100 100 (shared!)
```

- **Issue**: `x` is a class attribute—one `IntegerValue` instance for all `Point2D` objects and that is why `__get__` and `__set__` have instance as one of the parameters so that they know they are being called from which instance.
- **Problem**: Setting `p1.x` affects `p2.x`—not what we want!
- **Key Point**: **Class-level descriptors don’t store instance data**.


> [!question]- Isn't `p1.x=100` supposed to create instance variable x. Why does it affects `p2.x`?
> 
> When you assign `p1.x = 100`, it does not directly create an instance variable `x` on `p1`. Instead, it invokes the `__set__` method of the `IntegerValue` descriptor, which is defined at the class level. This descriptor modifies the instance's dictionary (`instance.__dict__`) to store the value using a key (typically prefixed with an underscore to avoid conflicts with other attributes). 
> 
> Here’s what happens step-by-step:
> 
> 1. **Descriptor Access**: When you set `p1.x = 100`, Python looks up `x` in `Point2D`, finds the descriptor (`IntegerValue`), and calls its `__set__` method.
> 
> 2. **Instance-specific Storage**: Inside `__set__`, the value is stored in `p1`'s dictionary (`p1.__dict__`) under a key like `_value`. This ensures each instance independently maintains its own state.
> 
> > [!question]- Why do we need to use `_value` instead of `value` in `instance.__dict__["_value"] = int(value)` ?
> > 
> > Using `_value` instead of `value` in `instance.__dict__["_value"] = int(value)` is a common practice to avoid conflicts and potential issues with attribute names. Here are the key reasons:
> > 
> > 1. **Name Collisions**: By using a name prefixed with an underscore (e.g., `_value`), you minimize the risk of accidentally overwriting or conflicting with other attributes that might be defined directly on the instance or elsewhere in its class hierarchy.
> > 
> > 2. **Encapsulation**: The underscore prefix is a convention in Python that indicates an attribute is intended for internal use within a class. It signals to developers that this attribute should not be accessed or modified directly outside of its intended context, promoting better encapsulation.
> > 
> > 3. **Avoiding Shadowing**: If you used `value` instead of `_value`, there could be unintended consequences if `value` were defined elsewhere, either as an instance variable or through inheritance. Using `_value` helps ensure that the descriptor's storage mechanism doesn't interfere with other parts of the class or its instances.
> > 
> > 4. **Descriptor Pattern**: This naming convention aligns with how descriptors are typically implemented in Python, where they manage their internal state separately from the main attributes users interact with.
> > 
> > By using a distinct key like `_value`, you ensure that each instance manages its state without interfering with other attributes, while also adhering to Python's conventions for indicating private or protected data.
> 
> 3. **Shared Descriptor Instance**: The descriptor itself (`IntegerValue`) is shared across all instances since it’s defined at the class level. However, because it uses individual instance dictionaries for storage, each instance manages its data separately.
> 
> The confusion arises from thinking that setting a class-level attribute would directly create an instance attribute; instead, descriptors manage this process by delegating storage to each instance's dictionary. Therefore, while descriptors are shared, they manipulate data on a per-instance basis due to how they interact with each instance's dictionary.

## Fixing with Instance Storage

Use the instance’s `__dict__`:

```python
class IntegerValue:
    def __set__(self, instance, value):
        instance.__dict__["_x"] = int(value)
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return instance.__dict__["_x"]

class Point2D:
    x = IntegerValue()

p1 = Point2D()
p2 = Point2D()
p1.x = 100
p2.x = 200
print(p1.x, p2.x)  # 100 200
```

- **Solution**: Store data in `instance.__dict__`, not the descriptor.
- **Takeaway**: **`instance` parameter ensures per-instance values**.
