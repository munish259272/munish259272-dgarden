# Properties and Descriptors Deep Dive

## Property with Decorators

Start with the decorator syntax for a validated `age` property:

```python
from numbers import Integral

class Person:
    @property
    def age(self):
        return getattr(self, "_age", None)
    @age.setter
    def age(self, value):
        if not isinstance(value, Integral):
            raise ValueError("age must be an integer")
        if value < 0:
            raise ValueError("age must be non-negative")
        self._age = value

p = Person()
try:
    p.age = -5  # ValueError: age must be non-negative
except ValueError as e:
    print(e)
p.age = 10
print(p.age)        # 10
print(p.__dict__)   # {'_age': 10}
```

- **How**: `@property` defines the getter; `@age.setter` defines the setter.
- **Key Point**: **Decorators wrap methods into a property object**.

## Without Decorators

Rewrite the same property explicitly:

```python
class Person:
    def get_age(self):
        return getattr(self, "_age", None)
    def set_age(self, value):
        if not isinstance(value, Integral):
            raise ValueError("age must be an integer")
        if value < 0:
            raise ValueError("age must be non-negative")
        self._age = value
    age = property(fget=get_age, fset=set_age)

p = Person()
p.age = 10
print(p.age)  # 10
```

- **Equivalence**: `property(fget, fset)` mirrors the decorator version.
- **Key Point**: **Properties are data descriptors under the hood**.

## Property as Descriptor

Properties have `__get__`, `__set__`, and `__delete__`:

```python
prop = Person.age
print(hasattr(prop, "__get__"))    # True
print(hasattr(prop, "__set__"))    # True
print(hasattr(prop, "__delete__")) # True
```

- **Behavior**: Even without `fset`, `__set__` exists but raises an error if `fset` is `None`.
- **Key Point**: **All properties are data descriptors**.

## Read-Only Property

Omit `fset` for read-only:

```python
class TimeUTC:
    @property
    def current_time(self):
        return "current_time"

t = TimeUTC()
print(t.current_time)  # current_time
try:
    t.current_time = "new_time"  # AttributeError
except AttributeError as e:
    print(e)  # can't set attribute
```

- **Why**: `__set__` exists but fails due to missing `fset`.
- **Key Point**: **Read-only depends on `fset` absence**.

## Custom Property Class

Build a `MakeProperty` descriptor:

```python
class MakeProperty:
    def __init__(self, fget=None, fset=None):
        self.fget = fget
        self.fset = fset
    def __set_name__(self, owner, name):
        self.prop_name = name
    def __get__(self, instance, owner):
        print("Get called")
        if instance is None:
            return self
        if self.fget is None:
            raise AttributeError(f"{self.prop_name} is not readable")
        return self.fget(instance)
    def __set__(self, instance, value):
        print("Set called")
        if self.fset is None:
            raise AttributeError(f"{self.prop_name} is not writeable")
        self.fset(instance, value)

class Person:
    def get_name(self):
        print("Get name called")
        return getattr(self, "_name", None)
    def set_name(self, value):
        print("Set name called")
        self._name = value
    name = MakeProperty(fget=get_name, fset=set_name)

p = Person()
p.name = "Guido"  # Set called, Set name called
print(p.name)     # Get called, Get name called, Guido
p.__dict__["name"] = "Alex"
print(p.name)     # Get called, Get name called, Guido (descriptor wins)
```

- **How**: Mimics `property` with custom `__get__` and `__set__`.
- **Key Point**: **Data descriptors ensure control over `__dict__`**.

## Adding Decorator Support

Extend `MakeProperty` for decorators:

```python
class MakeProperty:
    def __init__(self, fget=None, fset=None):
        self.fget = fget
        self.fset = fset
    def __set_name__(self, owner, name):
        self.prop_name = name
    def __get__(self, instance, owner):
        if instance is None:
            return self
        if self.fget is None:
            raise AttributeError(f"{self.prop_name} is not readable")
        return self.fget(instance)
    def __set__(self, instance, value):
        if self.fset is None:
            raise AttributeError(f"{self.prop_name} is not writeable")
        self.fset(instance, value)
    def setter(self, fset):
        self.fset = fset
        return self

class Person:
    @MakeProperty
    def age(self):
        return getattr(self, "_age", None)
    @age.setter
    def age(self, value):
        self._age = value

p = Person()
print(Person.age) # <__main__.MakeProperty object at 0x7ff4b15d7050>
p.age = 100
print(p.age)  # 100
print(p.__dict__)  # {'_name': 100}
```

- **How**: `@MakeProperty` wraps the getter; `.setter` adds the setter.
- **Key Point**: **Descriptors enable decorator syntax too**.

> [!question]- `print(Person.age)` returns `<__main__.MakeProperty object at 0x7ff4b15d7050>` because it called `__get__` and `instance` was `None`. Why is `instance` None until `p.age = 100` ?
> 
> 
> When you access `Person.age`, you're accessing the descriptor object itself, not an instance of `Person`. In this context, the `__get__` method is called with `instance` set to `None` because there isn't an instance involved in this particular access. This is because:
> 
> 1. **Class-level Access**: When you access a property directly from the class (e.g., `Person.age`), Python calls the descriptor's `__get__` method with `instance=None`. This indicates that the access is happening at the class level, not through an instance.
> 
> 2. **Descriptor Protocol**: According to Python's descriptor protocol, if you call a descriptor via a class rather than an instance, the first argument (`instance`) passed to the descriptor's methods (`__get__`, `__set__`, etc.) will be `None`.
> 
> 3. **Instance-level Access**: Conversely, when you access or modify a property through an instance (e.g., `p.age = 100`), Python calls the descriptor's methods with that specific instance as the first argument.
> 
> So, when you do something like:
> 
> ```python
> print(Person.age)
> ```
> 
> You're seeing `<__main__.MakeProperty object at 0x7ff4b15d7050>` because it's returning the descriptor object itself, and since it's accessed through the class and not an instance, it receives `None` for its `instance` parameter. The actual manipulation of data happens only when accessed through an instance like:
> 
> ```python
> p = Person()
> p.age = 100
> print(p.age)
> ```
> 
> Here, because you're interacting with an instance (`p`) of the class (`Person`), Python passes this specific instance as the first argument (`instance`) to any corresponding descriptor method calls (like `__get__`, `__set__`).

> 3. **Why `__call__` is Not Needed**
>    - In typical class-based decorators, `__call__` is used when the class instance needs to act as a callable, wrapping a function and executing it when invoked. For example:
>      ```python
>      class MyDecorator:
>          def __init__(self, func):
>              self.func = func
>          def __call__(self, *args, **kwargs):
>              print("Before calling the function")
>              result = self.func(*args, **kwargs)
>              print("After calling the function")
>              return result
>      ```
>      Here, `__call__` makes the decorator instance callable, allowing it to wrap and modify the execution of the decorated function.
> 
>    - However, `MakeProperty` doesn't need to be called like a function. Instead, it transforms the decorated method into a **property descriptor**, which is managed by Python's descriptor protocol. The `MakeProperty` instance itself is stored as the attribute (`Person.age`), and Python uses `__get__` and `__set__` to handle attribute access, not direct function calls. Thus, `__call__` is unnecessary because the class isn't acting as a callable wrapper but as a descriptor.

The decorator syntax (`@MakeProperty`) simply instantiates the class with the decorated function as an argument.

> ### Example to Clarify
