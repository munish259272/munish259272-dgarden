# Exploring Python’s `object` Class

## `object` as the Universal Base

Every class in Python inherits from the built-in `object` class:

```python
class Person:
    pass

print(issubclass(Person, object))  # True
print(issubclass(int, object))     # True
print(issubclass(dict, object))    # True
```

- **Implicit**: No explicit parent? It’s still `object`.
- **Type**: `type(object)` → `<class 'type'>` (it’s a class, lowercase due to C-style naming).
- **Key Point**: **All types are subclasses of `object`**.

---

## Modules as Objects

Even modules inherit from `object`:

```python
import math
import types

print(type(math))                # <class 'module'>
t = type(math)                   # Store module type
print(issubclass(t, object))     # True
print(types.ModuleType == t)     # True
```

- **Source**: `types.ModuleType` defines the module class.
- **Takeaway**: **Everything in Python is an object**.

---

## Functions and Types

Functions too:

```python
def myfunc(): pass

print(type(myfunc))               # <class 'function'>
print(types.FunctionType == type(myfunc))  # True
print(isinstance(myfunc, object)) # True
```

- **Types Module**: Hosts `FunctionType`, `GeneratorType`, etc.
- **Key Point**: **Instances of any type are instances of `object`**.

---

## `object`’s Default Behaviors

`object` provides core methods:

```python
o1 = object()
print(repr(o1))       # <object object at 0x...>
print(o1 == o1)       # True (compares id())
print(hash(o1))       # Some int (based on id())
```

- **Methods**: `__eq__` (id-based equality), `__hash__`, `__repr__`, `__init__`, `__new__`.
- **Fallback**: `str(o1)` calls `__repr__` if `__str__` isn’t defined.

---

## Inheriting Defaults

Custom classes get these too:

```python
p1 = Person()
p2 = Person()
print(repr(p1))       # < __main__.Person object at 0x... >
print(p1 == p2)       # False (different objects)
print(p1 == p1)       # True (same object)
```

- **Proof**: `id(Person.__eq__)` matches `id(object.__eq__)`—it’s inherited.
- **Key Point**: **Default behaviors come from `object`**.

---

## Overriding `object` Methods

Override when needed:

```python
class Person:
    def __init__(self):
        pass

p = Person()
print(id(Person.__init__) != id(object.__init__))  # True (overridden)
```

- **Default `__init__`**: No arguments, just initializes. (==i.e `id(Person.__init__)` is same as `id(object.__init__)` if we do not define it in our class==)
- **Custom**: Overrides `object`’s version.
- **Creation**: `__new__` (from `object`) still runs first.
> [!question]- "**Creation**: `__new__` (from `object`) still runs first." - You mean `__new__` from object is run first whether we provide our own `__init__` in our class or not ?
> 
> Yes, exactly! `__new__` from `object` runs first whether you provide your own `__init__` in your class or not. Let me clarify:
> 
> - **`__new__` Role**: It’s the constructor—it creates the instance (allocates memory) and returns it. It’s a static method inherited from `object` unless overridden.
> - **`__init__` Role**: It’s the initializer—it sets up the instance after `__new__` creates it.
> - **Order**: For any class, Python always calls `__new__` first to make the object, then calls `__init__` to initialize it, regardless of whether you define `__init__`.
> 
> ### Examples
> 
> 1. **No `__init__` Defined**:
>    ```python
>    class Person:
>        pass
> 
>    p = Person()
>    ```
>    - `__new__` (from `object`) creates the `Person` instance.
>    - `__init__` (default from `object`, a no-op) runs next.
> 
> 2. **Custom `__init__`**:
>    ```python
>    class Person:
>        def __init__(self):
>            print("Initializing")
> 
>    p = Person()  # Prints "Initializing"
>    ```
>    - `__new__` (still from `object`) creates the instance.
>    - Your `__init__` runs after, printing the message.
> 
> 3. **Overriding `__new__`**:
>    ```python
>    class Person:
>        def __new__(cls):
>            print("Creating")
>            return super().__new__(cls)
>        def __init__(self):
>            print("Initializing")
> 
>    p = Person()  # Prints "Creating" then "Initializing"
>    ```
>    - Your `__new__` runs first, delegates to `object.__new__` to create the instance.
>    - Then `__init__` runs.
> 
> ### Why `__new__` from `object`?
> - If you don’t define `__new__`, Python uses `object.__new__` because it’s the base implementation that knows how to allocate memory for any object.
> - It’s always called first to produce the instance that `__init__` then works on.
> 
> So, **yes**, `__new__` from `object` runs first in all cases unless you override it, and your `__init__` (custom or default) follows.
- **Takeaway**: **Override to customize, else use `object`’s defaults**.
