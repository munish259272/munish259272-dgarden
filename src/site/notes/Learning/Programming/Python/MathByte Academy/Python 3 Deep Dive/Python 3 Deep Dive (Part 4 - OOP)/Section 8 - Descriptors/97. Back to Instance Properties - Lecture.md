# Data Descriptors as Instance Properties

## WeakKeyDictionary Approach

Use `WeakKeyDictionary` for clean storage:

```python
from weakref import WeakKeyDictionary

class IntegerValue:
    def __init__(self):
        self.values = WeakKeyDictionary()
    def __set__(self, instance, value):
        self.values[instance] = int(value)
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return self.values.get(instance)
```

- **Pros**: Auto-cleans when instances die.
- **Limit**: **Requires hashable instances**.
- **Key Point**: **Simple but restrictive**.

---

## Using Instance IDs

Switch to IDs for unhashable objects:

```python
class IntegerValue:
    def __init__(self):
        self.values = {}
    def __set__(self, instance, value):
        self.values[id(instance)] = int(value)
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return self.values.get(id(instance))
```

- **Advantage**: IDs are always hashable.
- **Drawbacks**:
  - Entries(ids) linger after garbage collection.
  - ⚠️ **Warning**: **ID reuse risks wrong lookups** (rare).
- **Takeaway**: **IDs avoid hashability but clutter persists**.

## Final Solution: Weak Refs with Callbacks

Combine weak refs and callbacks:

```python
import weakref

class IntegerValue:
    def __init__(self):
        self.values = {}
        self._refs = {}  # Track weak refs
    def _cleanup(self, ref):
        # Callback: remove entry when object dies
        instance_id = ref._id
        self.values.pop(instance_id, None)
        self._refs.pop(instance_id, None)
    def __set__(self, instance, value):
        instance_id = id(instance)
        self.values[instance_id] = int(value)
        if instance_id not in self._refs:
            ref = weakref.ref(instance, self._cleanup)
            ref._id = instance_id  # Store ID in ref
            self._refs[instance_id] = ref
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return self.values.get(id(instance))

class Point2D:
    __slots__ = ()  # Works with slots
    x = IntegerValue()

p = Point2D()
p.x = 10
print(p.x)  # 10
del p       # Auto-cleaned via callback
```

- **How**:
  - Use `id(instance)` as key.
  - Store weak ref with callback to remove dead entries.
- **Benefits**:
  - Works with slots (no `__dict__` needed) since we are not using instance itself to store the values but still achieving instance specific storage.
  - Handles unhashable objects.
  - Keeps dictionary clean.
- **Key Point**: **Callbacks ensure no leaks or clutter**.

> [!question]- While doing cleanup in `_cleanup` why are doing `instance_id = ref._id` instead of `instance_id = None` ?
> 
> 
> In the `_cleanup` method, we are using `instance_id = ref._id` instead of `instance_id = None` because we need to identify which specific entry in the `self.values` and `self._refs` dictionaries should be removed when the reference to an instance is garbage collected.
> 
> Here's why this is important:
> 
> 1. **Tracking Instances**: When we set a value on an instance using the descriptor, we store that value in a dictionary (`self.values`) with the key being the `id(instance)`. This allows us to associate each instance with its corresponding value.
> 
> 2. **Weak References**: We also store a weak reference to each instance in another dictionary (`self._refs`). The weak reference includes a callback function (`_cleanup`) that is triggered when the instance is garbage collected. This callback needs to know which entry in our dictionaries corresponds to the now-deleted instance.
> 
> 3. **Identifying Entries**: By storing `instance_id` (which is essentially `id(instance)`) as an attribute of the weak reference object (`ref._id`), we can retrieve it within the `_cleanup` method. This allows us to precisely remove only those entries associated with that particular instance, rather than indiscriminately removing entries or setting them to `None`.
> 
> 4. **Preventing Leaks and Clutter**: Using this mechanism ensures that once an object (or its reference) no longer exists, its associated data is cleaned up from both dictionaries, preventing memory leaks and keeping our data structure clean.
> 
> Therefore, retrieving and using `instance_id = ref._id` is crucial for accurately cleaning up resources tied to specific instances without affecting other instances or leaving orphaned data behind.


> [!question]- Why is there a second parameter `None` for `pop` ?
> 
> The `pop` method in Python dictionaries is used to remove a specific key and return its associated value. It has the following signature:
> 
> ```python
> dict.pop(key[, default])
> ```
> 
> - `key`: The key you want to remove from the dictionary.
> - `default`: An optional parameter. If the key is not found, `pop` will return this default value instead of raising a `KeyError`. By default, if you don't specify this parameter and the key is not found, a `KeyError` will be raised.
> 
> In the context of your code:
> 
> ```python
> def _cleanup(self, ref):
>     instance_id = ref._id
>     self.values.pop(instance_id, None)
>     self._refs.pop(instance_id, None)
> ```
> 
> The use of `None` as the second parameter in `pop` serves as a safety mechanism. It ensures that if for some reason the `instance_id` is not found in the dictionary (perhaps due to concurrent modifications or other logic errors), no exception will be raised. Instead, it will simply return `None`, allowing your program to continue executing without interruption.
> 
> This approach is particularly useful when you're dealing with potentially unpredictable states or when it's important to ensure cleanup happens smoothly without additional error handling for missing keys.


> [!question]- What is a Callback ?
> A **callback** is a function passed as an argument to another function or method, which is then executed at a specific time or event. In Python, it’s often used to customize behavior or handle events asynchronously.
> 
> For example, in the context of weak references (like in the last rewrite), a callback is a function that gets called automatically when an object is garbage collected. Here’s a simple illustration:
> 
> ```python
> import weakref
> 
> def callback(ref):
>     print("Object was garbage collected!")
> 
> class MyClass:
>     pass
> 
> obj = MyClass()
> ref = weakref.ref(obj, callback)  # callback is called when obj is collected
> del obj                           # Triggers garbage collection
> # Output: "Object was garbage collected!"
> ```
> 
> - **Key Idea**: The callback (`callback` here) runs when the weak reference detects the object (`obj`) is no longer alive, letting you react (e.g., clean up resources).
> - **Use Case**: In descriptors, it removes stale dictionary entries when instances die.
> 
> > [!question]- What is a callback in general ( not specific to weak references above) ?
> > 
> > 
> > A **callback**, in general programming terms, is a function that is passed as an argument to another function and is intended to be "called back" at some appropriate point in the execution of that function or application. This allows the behavior of the calling function to be customized or extended without modifying its code.
> > 
> > Here are some key points about callbacks:
> > 
> > 1. **Asynchronous Programming**: Callbacks are commonly used in asynchronous programming to handle events or data that become available at a later time. For example, when making network requests, a callback function might be used to handle the response once it arrives.
> > 
> > 2. **Event Handling**: In event-driven programming (such as GUI applications), callbacks are often used to define what should happen when specific events occur, like button clicks or mouse movements.
> > 
> > 3. **Customization and Extensibility**: By passing different callback functions into a generic piece of code, you can customize its behavior for different scenarios without changing its core logic. This is useful for creating flexible and reusable components.
> > 
> > 4. **Functional Programming**: In functional programming paradigms, higher-order functions (functions that take other functions as arguments) often utilize callbacks as a way to apply operations over data collections or manage control flow.
> > 
> > Here’s a simple example of a callback in Python:
> > 
> > ```python
> > def greet(name):
> >     return f"Hello, {name}!"
> > 
> > def process_name(name, callback):
> >     # Some processing on name
> >     processed_name = name.strip().capitalize()
> >     # Use the callback with processed name
> >     return callback(processed_name)
> > 
> > print(process_name(" alice ", greet))  # Output: "Hello, Alice!"
> > ```
> > 
> > In this example:
> > - `greet` is a simple function that formats a greeting.
> > - `process_name` takes a `name` and a `callback` function as arguments.
> > - After processing the name (e.g., trimming whitespace and capitalizing), it calls the `callback` with the processed name.
> >   
> > This pattern allows `process_name` to remain flexible; you could pass different functions as callbacks for different behaviors.


> [!question]- Does `_cleanup` automatically gets called on `del p` but how does it recieves the parameter `ref` ?
> 
> Yes, `_cleanup` is automatically called, and it’s actually a clever feature of how Python’s `weakref` module handles the lifecycle of an object.
> 
> Here is the breakdown of how that `ref` parameter finds its way into your method.
> 
> ---
> 
> ### 1. The Callback Mechanism
> 
> When you create a weak reference using `weakref.ref(obj, callback)`, you are telling Python: _"Keep an eye on `obj`. If its reference count hits zero, please execute `callback`."_
> 
> The crucial part is the signature of that callback. Python is hard-coded to pass the **weak reference object itself** as the first (and only) argument to the callback function once the referent (the object it points to) is garbage collected.
> 
> ### 2. The Step-by-Step Execution
> 
> 1. **Creation:** When you run `ref = weakref.ref(instance, self._cleanup)`, the `ref` object is "registered" with the Python garbage collector.
>     
> 2. **Death of p:** When you call `del p` (and no other references to that instance exist), the object is destroyed.
>     
> 3. **The Trigger:** Immediately after the object is cleared from memory, the weak reference object "wakes up" and triggers the callback you provided (`self._cleanup`).
>     
> 4. **The Injection:** Python executes `self._cleanup(ref)`. It passes the `ref` object back to you so you can identify which specific reference just "expired."
>     
> 
> ---
> 
> ### 3. Why did you attach `._id` to the ref?
> 
> You’ve used a very smart pattern here. Since the `instance` is already dead by the time `_cleanup` runs, you can't call `id(instance)` inside the cleanup function.
> 
> By doing this:
> 
> 
> ```python
> ref = weakref.ref(instance, self._cleanup)
> ref._id = instance_id  # Monkey-patching the ID onto the ref object
> ```
> 
> You are "tagging" the weak reference. When Python passes that `ref` back into `_cleanup(self, ref)`, the ID is still attached to it, allowing you to successfully find and `pop` the correct entry from your `self.values` dictionary.
> 
> ### Important Note on `__slots__`
> 
> In your code, `Point2D` has `__slots__ = ()`. For `weakref` to work with classes using slots, you usually must explicitly include `'__weakref__'` in the slots list:
> 
> Python
> 
> ```
> class Point2D:
>     __slots__ = ('__weakref__',) # Required for weakref support in slotted classes
>     x = IntegerValue()
> ```
> 
> Without `__weakref__` in the slots, `weakref.ref(instance)` would typically raise a `TypeError` because the instance wouldn't have the necessary internal machinery to be tracked.
