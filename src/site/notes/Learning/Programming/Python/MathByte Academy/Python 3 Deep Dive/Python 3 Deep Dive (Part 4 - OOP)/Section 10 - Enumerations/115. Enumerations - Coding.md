# Using Python Enumerations

## Setting Up Enums

Start with the `enum` module and subclass `Enum`:

```run-python
from enum import Enum

class Color(Enum):
    RED = 1
    GREEN = 2
    BLUE = 3

class Status(Enum):
    PENDING = "pending"
    RUNNING = "running"
    COMPLETED = "completed"

class UnitVector(Enum):
    V1D = (1,)
    V2D = (1, 1)
    V3D = (1, 1, 1)
```

- **Flexibility**: Values can be **any object**—ints, strings, tuples, etc.
- **Key Point**: **Enums adapt to your needs**.

---

## Member Types and Properties

Members are instances of their enum class:

```run-python
print(type(Status.PENDING))  # <enum 'Status'>
print(isinstance(Status.PENDING, Status))  # True
print(Status.PENDING.name)   # "PENDING"
print(Status.PENDING.value)  # "pending"
print(UnitVector.V3D.value)  # (1, 1, 1)
```

- **Note**: `Status.PENDING` isn’t a string—it’s a `Status` object.
- **Key Point**: **Members carry `name` and `value` attributes**.

---

## Comparing Members

Use `is` for **identity checks**:

```run-python
print(Status.PENDING is Status.PENDING)  # True
print(Status.PENDING == Status.PENDING)  # True (but avoid)
print(Status.PENDING in Status)          # True
```

- **Why `is`?**: Faster, and $==$ can be overridden.
- **Key Point**: **Stick to `is` for member equality**.

---

## No Default Comparisons

Rich comparisons aren’t built-in:

```run-python
class Constants(Enum):
    ONE = 1
    TWO = 2
    THREE = 3

# Constants.ONE < Constants.TWO  # TypeError: '<' not supported
```

- **Benefit**: Prevents meaningless operations (e.g., `ONE * 2`).
- **Key Point**: **Enums avoid accidental comparisons**.

> [!question]- What are rich comparisons ?
> 
> Rich comparisons refer to the set of comparison operations that can be performed between objects, typically including less than (`<`), less than or equal to (`<=`), greater than (`>`), and greater than or equal to (`>=`). In Python, these are implemented using special methods such as `__lt__`, `__le__`, `__gt__`, and `__ge__`. 
> 
> Enums do not support these rich comparisons by default because the concept of ordering doesn't inherently apply to all sets of enum members. This design choice helps avoid accidental and often meaningless comparisons between enum members that might not have a logical ordering. However, if needed, you can define custom behavior for these operators by implementing the aforementioned special methods within your enum class.

## Lookup by Value or Name

Access members dynamically:

```run-python
print(Status("pending"))      # Status.PENDING
print(UnitVector((1, 1, 1)))  # UnitVector.V3D
print(Status["PENDING"])      # Status.PENDING
print(getattr(Status, "PENDING"))  # Status.PENDING
```

- **By Value**: Call the enum (e.g., `Status("pending")`).
- **By Name**: Use brackets (e.g., `Status["PENDING"]`) or `getattr`.
- **Key Point**: **Callables and `__getitem__` enable lookups**.

---

## Handling Non-Existent Values

Lookups fail gracefully:

```run-python
try:
    Status("invalid")  # ValueError: 'invalid' is not a valid Status
except ValueError as e:
    print(e)
print(getattr(Status, "INVALID", None))  # None
```

- **Options**: Catch exceptions or use `getattr` with a default.
- **Key Point**: **Safe lookups prevent crashes**.

---

## Hashability

Members are **always hashable**, even with unhashable values:

```run-python
class Person:
    __hash__ = None  # Unhashable

class Family(Enum):
    PERSON1 = Person()
    PERSON2 = Person()

d = {Family.PERSON1: "p1", Family.PERSON2: "p2"}
print(d[Family.PERSON1])  # "p1"
```

- **Why**: Hashing uses the member, not its value.
- **Key Point**: **Enums work in dicts and sets**.

---

## Iteration

Enums are **iterable** in definition order:

```run-python
print(list(Status))  # [<Status.PENDING: 'pending'>, ...]
class Numbers(Enum):
    THREE = 3
    TWO = 2
    ONE = 1
print(list(Numbers))  # [<Numbers.THREE: 3>, <Numbers.TWO: 2>, <Numbers.ONE: 1>]
```

- **Order**: Follows declaration, not value.
- **Key Point**: **Preserved order aids consistency**.

---

## Immutability

Enums are **fixed after creation**:

```run-python
# Status.PENDING.value = 10  # AttributeError
# Status.NEW = 100  # TypeError: Cannot extend enumerations
```

- **Limits**: No adding, removing, or reassigning values.
- **Key Point**: **Immutability ensures reliability**.

---

## Subclassing Rules

Empty enums can be subclassed:

```run-python
class EnumBase(Enum):
    pass
class EnumExt(EnumBase):
    ONE = 1  # Works
    
##------------------------------------------------------##

class Base(Enum):
    ONE = 1
# class Ext(Base):  # TypeError: Cannot extend enumerations
#     TWO = 2
```

- **Why**: Prevents member changes via inheritance.
- **Key Point**: **Subclassing is for empty bases only**.

## Checking Membership

> [!question]- Why do we have two ways to reference something by name? Like, we can use `Status.PENDING` or `Status['PENDING']`.
> The two ways to reference enumeration members by name—`Status.PENDING` (dot notation) and `Status["PENDING"]` (bracket notation)—exist to offer **flexibility based on context**:
> 
> ### 1. **`Status.PENDING` (Dot Notation)**  
> - **Purpose**: Direct, declarative access to a member when you know the name at coding time.
> - **Use Case**: Hardcoding a specific member in your logic.
> - **Example**:
>   ```python
>   from enum import Enum
>   
>   class Status(Enum):
>       PENDING = "pending"
>   
>   print(Status.PENDING.value)  # "pending"
>   ```
> - **Why**: It’s **readable and static**, ideal for fixed references.
> 
> ### 2. **`Status["PENDING"]` (Bracket Notation)**  
> - **Purpose**: Dynamic lookup when the name comes as a string (e.g., from user input, APIs, or variables).
> - **Use Case**: When the member name isn’t known until runtime.
> - **Example**:
>   ```python
>   import json
>   
>   payload = '{"status": "PENDING"}'
>   data = json.loads(payload)
>   status_name = data["status"]  # "PENDING" (string)
>   member = Status[status_name]  # Status.PENDING
>   print(member.value)  # "pending"
>   ```
> - **How**: Uses `__getitem__` to map the string to the member.
> - **Why**: It’s **programmatic**, handling runtime data.
> 
> ### Key Differences
> - **Static vs. Dynamic**: Dot notation is compile-time; bracket notation is runtime.
> - **Source**: Dot notation uses the attribute directly; bracket notation uses a string key.
> 
> ### Why Both?
> - **Convenience**: `Status.PENDING` is cleaner for known values.
> - **Versatility**: `Status["PENDING"]` integrates with external systems (e.g., JSON, configs) where names are strings.
> - **Analogy**: Like `dict["key"]` vs. `obj.key`—same idea, different needs.
> 
> ### Key Point
> - **Two methods suit different scenarios**: Use dot for static code, brackets for dynamic lookups.

Test if a string is a member:

```run-python
def is_member(enum, name):
    return getattr(enum, name, None) is not None

print(is_member(Status, "PENDING"))  # True
print(Status.__members__) # {'PENDING': <Status.PENDING: 'pending'>, 'RUNNING': <Status.RUNNING: 'running'>, 'COMPLETED': <Status.COMPLETED: 'completed'>}
print("pending" in Status.__members__)  # True
```

- **Methods**: `getattr`, `__members__` (a read-only dict/mapping proxy).
- **Key Point**: **Multiple ways to validate names**.
