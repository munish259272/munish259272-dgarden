# Diving into Properties in Python

## Bare Attributes: The Simple Way

We’ve been using **bare attributes** in Python:
```python
class Person:
    def __init__(self, name):
        self.name = name

p = Person("Alex")
print(p.__dict__)  # {'name': 'Alex'}
p.name = 100  # No restrictions
print(p.name)  # 100
```

- **Direct Access**: `p.name` gets or sets the value in `p.__dict__`.
- **Issue**: No control—`p.name = 100` or `p.name = ""` works, even if we want only non-empty strings.

**Key Point**: **Bare attributes** are easy but lack validation.

---

## Getter/Setter Approach (Java Style)

In Java, we’d use **private attributes** with methods:
```python
class Person:
    def __init__(self, name):
        self._name = name  # Pseudo-private
    
    def get_name(self):
        return self._name
    
    def set_name(self, value):
        if isinstance(value, str) and len(value.strip()) > 0:
            self._name = value.strip()
        else:
            raise ValueError("Name must be a non-empty string")

p = Person("Alex")
print(p.get_name())  # "Alex"
p.set_name("Eric")  # Works
# p.set_name("")  # Raises ValueError
```

- **`_name`**: The underscore signals “treat this as private”—a convention, not enforcement.
- **Python’s Twist**: `p._name` is still accessible, unlike Java’s true `private`.

**Takeaway**: **Getters and setters** add control, but `p.get_name()` is clunky.

---

## Validation in `__init__`

To enforce rules from the start:
```python
class Person:
    def __init__(self, name):
        self.set_name(name)  # Use setter
    
    def get_name(self):
        return self._name
    
    def set_name(self, value):
        if isinstance(value, str) and len(value.strip()) > 0:
            self._name = value.strip()
        else:
            raise ValueError("Name must be a non-empty string")

p = Person("Alex")  # Works
# p = Person("")  # Raises ValueError
```

- **Reuse**: Call `set_name` in `__init__` for consistent validation.
- **Contrast**: Setting `self._name = name` directly skips checks.

**Key Point**: **Setters in `__init__`** ensure initial state is valid.

---

## Why Not Always Use Methods?

- **Syntax**: `p.get_name()` and `p.set_name("Eric")` vs. `p.name`—the latter is cleaner.
- **Python’s Way**: Start with bare attributes (`self.name`) and switch to methods later if needed, without breaking code.

**Takeaway**: **Bare attributes** are Python’s default for simplicity.

---

## Properties: Best of Both Worlds

Use the `property` class to keep clean syntax with control:
```python
class Person:
    def __init__(self, name):
        self._name = name  # Temporary bare attribute
    
    def get_name(self):
        print("Getter called")
        return self._name
    
    def set_name(self, value):
        print("Setter called")
        self._name = value
    
    name = property(fget=get_name, fset=set_name)

p = Person("Alex")
print(p.name)  # Getter called, "Alex"
print(type(Person.name)) # <class 'property'>
p.name = "Eric"  # Setter called
print(type(p.name)) # <class 'str'>
print(p.__dict__)  # {'_name': 'Eric'}
```

- **`name`**: A class-level `property` object, not in `p.__dict__`.
- **How It Works**:
  - `p.name` triggers `get_name`.
  - `p.name = "Eric"` triggers `set_name`.

**Key Point**: **Properties maintain `p.name` syntax** while using methods.

---

## Adding a Deleter

Extend with a deleter:
```python
class Person:
    def __init__(self, name):
        self._name = name
    
    def get_name(self):
        print("Getter called")
        return self._name
    
    def set_name(self, value):
        print("Setter called")
        self._name = value
    
    def del_name(self):
        print("Deleter called")
        del self._name
    
    name = property(fget=get_name, fset=set_name, fdel=del_name)

p = Person("Alex")
del p.name  # Deleter called
print(p.__dict__)  # {}
p.name = "Eric"  # Setter called, restores _name
```

- **`fdel`**: Runs when `del p.name` is called, removing `_name`.
- **Persistent Property**: `name` stays in the class, allowing reuse.

**Takeaway**: **Properties support get, set, and delete** operations.

---

## Property Lookup and Masking

- **Lookup**:
  - `p.name` checks `p.__dict__` (no `name`), then `Person.__dict__` (finds `property`), triggering the getter.
- **Masking**:
```python
p.__dict__["name"] = "John"  # Instance attribute
print(p.name)  # Still calls getter, "Eric" (not "John")
```

- **Why?**: Python prioritizes the class-level `property` over instance attributes with the same name.

> You’re absolutely right to question this—it seems to contradict the usual Python attribute lookup order! Let’s dive into why **Python prioritizes the class-level `property` over an instance attribute with the same name**, even though the standard lookup order checks the instance’s namespace first. This is a special behavior tied to how `property` objects work as **descriptors**, and I’ll explain it step-by-step.
> 
> ---
> 
> ### Standard Attribute Lookup Order
> 
> Normally, when you access `p.name`, Python follows this order:
> 1. **Instance Namespace**: Checks `p.__dict__` for `"name"`.
> 2. **Class Namespace**: If not found, looks in `Person.__dict__` (and up the inheritance chain).
> 3. **AttributeError**: If still not found, raises an error.
> 
> So, if `p.__dict__` has a `"name"` key, you’d expect `p.name` to return that value, bypassing anything in the class. For regular attributes, that’s true:
> 
> ```python
> class Person:
>     name = "Class-level"
> 
> p = Person()
> print(p.name)  # "Class-level"
> p.name = "Instance-level"
> print(p.__dict__)  # {'name': 'Instance-level'}
> print(p.name)  # "Instance-level"
> ```
> 
> - Here, the instance attribute `"name"` in `p.__dict__` **masks** the class attribute `name`.
> 
> ---
> 
> ### Properties Behave Differently
> 
> Now, let’s use a `property`:
> ```python
> class Person:
>     def __init__(self, name):
>         self._name = name
>     def get_name(self):
>         return self._name
>     def set_name(self, value):
>         self._name = value
>     name = property(get_name, set_name)
> 
> p = Person("Alex")
> print(p.name)  # "Alex" (via getter)
> p.__dict__["name"] = "John"  # Force instance attribute
> print(p.__dict__)  # {'_name': 'Alex', 'name': 'John'}
> print(p.name)  # Still "Alex" (via getter, not "John")
> ```
> 
> - **Surprise**: Even though `"name": "John"` is in `p.__dict__`, `p.name` returns `"Alex"` from the getter. Why?
> 
> ---
> 
> ### The Secret: Descriptors
> 
> The key lies in **descriptors**, a protocol Python uses for special attribute handling. A `property` object is a **data descriptor**, which changes the lookup rules. Here’s how:
> 
> #### Descriptor Basics
> - A **descriptor** is an object (typically at the class level) that defines one or more of these methods:
>   - `__get__`: Controls getting the attribute.
>   - `__set__`: Controls setting the attribute.
>   - `__delete__`: Controls deleting the attribute.
> - `property` implements `__get__` and `__set__` (and `__delete__` if `fdel` is provided), making it a **data descriptor**.
> 
> #### Lookup with Descriptors
> When Python sees a descriptor in the class namespace:
> 1. **Class Check First**: Before looking at `p.__dict__`, Python checks `Person.__dict__` for `"name"`.
> 2. **Descriptor Detection**: If `"name"` is a descriptor with `__get__` or `__set__`, Python calls its methods **instead of using the instance’s value**.
> 3. **Priority**: Data descriptors (with both `__get__` and `__set__`) **override instance attributes**, even if they exist in `p.__dict__`.
> 
> - **Proof**: `property` is a descriptor:
> ```python
> print(Person.name.__get__)  # <method-wrapper '__get__' ...>
> print(Person.name.__set__)  # <method-wrapper '__set__' ...>
> ```
> 
> ---
> 
> ### How It Works in Your Example
> 
> - **Setup**: `Person.name` is a `property` object in `Person.__dict__`.
> - **Access `p.name`**:
>   - Python looks in `Person.__dict__` and finds `name` as a `property`.
>   - Since it’s a data descriptor, Python calls `name.__get__(p, Person)`, which runs `get_name(self)` and returns `self._name` ("Alex").
>   - The `"name": "John"` in `p.__dict__` is **ignored** because the descriptor takes precedence.
> 
> - **Setting `p.name = "Eric"`**:
>   - Python calls `name.__set__(p, "Eric")`, which runs `set_name(self, "Eric")`, updating `self._name`.
>   - The instance’s `"name": "John"` isn’t touched.
> 
> ---
> 
> ### Why This Exception to the Rule?
> 
> - **Control**: Descriptors like `property` are designed to **enforce access rules**. If instance attributes could override them, you’d lose the validation or logic in `get_name`/`set_name`.
> - **Consistency**: `p.name` should always behave as defined by the `property`, not be bypassed by a rogue instance attribute.
> 
> #### Non-Descriptor Contrast
> If `name` were a regular function or non-data descriptor (only `__get__`):
> ```python
> class Person:
>     def name(self):
>         return "Class-level"
> p = Person()
> p.name = "Instance-level"
> print(p.name)  # "Instance-level" (instance wins)
> ```
> 
> - No `__set__` means it’s not a data descriptor, so the instance attribute wins.
> 
> ---
> 
> ### Conclusion
> 
> Python **prioritizes the class-level `property`** because it’s a **data descriptor**, which overrides the standard lookup order. Normally, `p.__dict__` trumps `Person.__dict__`, but descriptors flip this for attributes they manage, ensuring their `__get__`/`__set__` logic runs instead of using raw instance values.
> 
> **Key Point**: **Descriptors like `property` hijack attribute access**, bypassing `p.__dict__` to maintain control. We’ll explore descriptors more later!

> [!important] **Key Point**: **Properties override instance attributes** of the same name.

## Adding Documentation

Add a docstring:
```python
class Person:
    """This is a person object."""
    def __init__(self, name):
        self._name = name
    def get_name(self):
        return self._name
    def set_name(self, value):
        self._name = value
    name = property(get_name, set_name, doc="The person's name")

help(Person)  # Shows class docstring and "name: The person's name"
```

- **`doc`**: Describes the property in `help()` output.

**Takeaway**: **Properties can be documented** for clarity.
