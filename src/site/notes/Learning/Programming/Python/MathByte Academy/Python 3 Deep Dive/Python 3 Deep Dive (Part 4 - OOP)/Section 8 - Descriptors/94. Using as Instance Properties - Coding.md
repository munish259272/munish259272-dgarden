# Descriptors as Instance Properties

## Basic Storage in Instance Dictionary

Use the instance’s `__dict__` for storage:

```python
class IntegerValue:
    def __set__(self, instance, value):
        instance.__dict__['stored_value'] = int(value)
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return getattr(instance, 'stored_value', None)

class Point1D:
    x = IntegerValue()

p1 = Point1D()
p2 = Point1D()
p1.x = 10.1
p2.x = 20.2
print(p1.x, p2.x)  # 10 20
print(p1.__dict__) # {'stored_value': 10}
```

- **How**: Hardcode a key (`stored_value`) in `__dict__`.
- **Key Point**: **Works until naming conflicts arise**.

---

## Naming Conflicts

Multiple descriptors clash:

```python
class Point2D:
    x = IntegerValue()
    y = IntegerValue()

p = Point2D()
p.x = 10.1
print(p.x)         # 10
p.y = 20.2
print(p.y, p.x)    # 20 20 (y overwrote x!)
print(p.__dict__)  # {'stored_value': 20}
```

- **Issue**: Both `x` and `y` use `stored_value`—last write wins.
- **Takeaway**: **Hardcoding keys fails with multiple properties**.

---

## Unique Names via Init

Pass a name to the descriptor:

```python
class IntegerValue:
    def __init__(self, name):
        self.storage_name = f"_{name}"
    def __set__(self, instance, value):
        setattr(instance, self.storage_name, int(value))
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return getattr(instance, self.storage_name, None)

class Point2D:
    x = IntegerValue('x')
    y = IntegerValue('y')

p1 = Point2D()
p2 = Point2D()
p1.x = 10.1
p1.y = 20.2
p2.x = 100.1
print(p1.x, p1.y)  # 10 20
print(p2.x)        # 100
print(p1.__dict__) # {'_x': 10, '_y': 20}
```

- **Fix**: Unique keys (`_x`, `_y`) per descriptor.
- **Drawbacks**:
  - User must specify names twice.
  - Risk overwriting existing keys (e.g., `_x`).
  - Fails with slots (no `__dict__`).
- **Key Point**: **Manual naming is fragile**.

## Descriptor Storage Solution

Store in the descriptor instead:

```python
class IntegerValue:
    def __init__(self):
        self.values = {}
    def __set__(self, instance, value):
        self.values[instance] = int(value)
    def __get__(self, instance, owner):
        if instance is None:
            return self
        return self.values.get(instance)

class Point2D:
    x = IntegerValue()
    y = IntegerValue()

p1 = Point2D()
p2 = Point2D()
p1.x = 10.1
p1.y = 20.2
p2.x = 100.1
print(p1.x, p1.y)         # 10 20
print(p2.x)               # 100
print(Point2D.x.values)   # {<p1>: 10, <p2>: 100}
print(Point2D.y.values)   # {<p1>: 20}
```

- **How**: Use `instance` as a key in `self.values`.
- **Pros**: Works with slots, no naming conflicts.
- **Requirement**: Instances must be hashable.
- **Key Point**: **Descriptor storage is robust**.

## Memory Leak Problem

References persist:

```python
p1 = Point2D()
id_p1 = id(p1)
p1.x = 10.1
del p1
print(Point2D.x.values)  # Still holds <p1>: 10
```

- **Issue**: `self.values` keeps `p1` alive—ref count stays 1.
- ⚠️ **Warning**: **Memory leak without cleanup**.
- **Next**: Weak references to fix this.

> [!question]- "<font color="#000000" style="font-weight:normal;font-style:italic;">self.values keeps p1 alive</font>" -- How ?
> 
> The dictionary `self.values` in the `IntegerValue` descriptor keeps `p1` alive because **standard Python dictionaries hold strong references to their keys and values**.
> 
> Here is a breakdown of how this causes a memory leak:
> 
> ---
> 
> ## 🔑 Strong References and the Memory Leak
> 
> ### 1. Strong Reference Creation
> 
> When you execute the line `p1.x = 10.1`, the descriptor's `__set__` method is called:
> 
> Python
> 
> ```
> # Inside IntegerValue.__set__(self, instance, value):
> self.values[instance] = int(value) 
> # Equivalent to: Point2D.x.values[p1] = 10
> ```
> 
> - The dictionary `Point2D.x.values` (which is stored on the descriptor object itself) uses the object referred to by `p1` as a **key**.
>     
> - A standard Python dictionary maintains a **strong reference** to every object used as a key. This means the object's reference count is incremented.
>     
> 
> ### 2. Preventing Garbage Collection
> 
> An object is only garbage collected (its memory released) when its **reference count** drops to zero.
> 
> |**Step**|**Action**|**Reference Count for p1**|**Result**|
> |---|---|---|---|
> |Initial|`p1 = Point2D()`|1 (from the variable `p1`)||
> |Set|`p1.x = 10.1`|2 (from `p1` variable + **strong reference** in `self.values` dictionary)||
> |Delete|`del p1`|1 (the `p1` variable reference is removed)|**Object `p1` is NOT collected.**|
> 
> Even after you use `del p1` (which removes the name `p1` from the local namespace and decrements the count by one), the object `p1` still has a reference count of 1 because it's being held as a key in the `self.values` dictionary.
> 
> ### 3. The Leak
> 
> The object `p1` remains allocated in memory indefinitely, accessible only through the descriptor's internal dictionary, `Point2D.x.values`. If the application creates and deletes many instances of `Point2D`, all of them will persist in the descriptor's memory, leading to a **memory leak**.
> 
> ---
> 
> ## ✅ The Solution: Weak References
> 
> To fix this, you must use a data structure that does **not** hold strong references to the keys, allowing them to be garbage collected when no other strong references exist.
> 
> This is precisely the purpose of the **`weakref.WeakKeyDictionary`**:
> 
> 
> ```python
> import weakref
> 
> class IntegerValue:
>     def __init__(self):
>         # self.values now uses WeakKeyDictionary
>         # It holds a weak reference to instance (the key)
>         self.values = weakref.WeakKeyDictionary() 
>     # ... (rest of __set__ and __get__ are the same)
> ```
> 
> With a `WeakKeyDictionary`, when the last external strong reference to `p1` is removed (i.e., when `del p1` is called), the `WeakKeyDictionary` automatically removes the key/value pair corresponding to `p1` from its internal storage, preventing the memory leak.
