Let’s explore **delegating to parent classes with `super()`**, a concept you’ll often see in the `__init__` method of custom classes, though it’s not limited to that. You can use `super()` anywhere you need to **call a method higher up the inheritance hierarchy**. But here’s the catch: you don’t always need it. If a method isn’t overridden in the current class, Python will automatically find it in the parent—no `super()` required.

Take this example where we *do* need `super()` because of an override:

```run-python
class Person:
    def work(self):
        return "Person works"

class Student(Person):
    def work(self):
        result = super().work()  # Calls Person's work
        return f"Student studies and {result}"

s = Student()
s.work()
```

For a `Student` instance calling `work()`, using `self.work()` would trigger **infinite recursion**—it keeps calling `Student`’s `work`, crashing with an exception. Instead, `super().work()` explicitly tells Python to **fetch the parent’s version**, giving us "Student studies and Person works." The `super()` function climbs the hierarchy, finding the first `work` it encounters, even if it’s not directly in `Person` but inherited from further up.

Here’s a deeper chain:

```run-python
class Person:
    def work(self):
        return "Person works"

class Student(Person):
    pass

class PythonStudent(Student):
    def work(self):
        result = super().work()
        return f"PythonStudent codes and {result}"
```

A `PythonStudent` instance calling `work()` outputs "PythonStudent codes and Person works." `super().work()` looks in `Student`, finds nothing, then hits `Person`’s version. The method stays **bound to the `PythonStudent` instance**, so `self` refers to it throughout.

Now, watch delegation ripple up:

```run-python
class Person:
    def work(self):
        return "Person works"

class Student(Person):
    def work(self):
        result = super().work()
        return f"Student studies and {result}"

class PythonStudent(Student):
    def work(self):
        result = super().work()
        return f"PythonStudent codes and {result}"

ps = PythonStudent()
ps.work() # PythonStudent codes and Student studies and Person works.
```

A `PythonStudent`’s `work()` call yields "PythonStudent codes and Student studies and Person works." ==Each `super()` **hops up one level**==, with every method bound to the original `PythonStudent` instance.

But **no ambiguity, no `super()`**. Look at this:

```run-python
class Person:
    def work(self):
        return "Person works"

class Student(Person):
    def study(self):
        return "Student studies"

class PythonStudent(Student):
    def code(self):
        result1 = self.work()  # From Person
        result2 = self.study()  # From Student
        return f"{result1} and {result2} and PythonStudent codes"

ps = PythonStudent()
ps.code()
```

Calling `code()` on a `PythonStudent` instance gives "Person works and Student studies and PythonStudent codes." Since `work` and `study` aren’t overridden in `PythonStudent`, `self` finds them automatically—no `super()` needed.

**Binding matters**: The instance calling the method dictates `self`. Even with `super()`, the original instance persists:

```run-python
class Person:
    def work(self):
        return f'{self} works...'

class Student(Person):
    def work(self):
        result = super().work()
        return f'{self} studies... and {result}'

class PythonStudent(Student):
    def work(self):
        result = super().work()
        return f'{self} codes... and {result}'

ps = PythonStudent() 
print(hex(id(ps))) # 🔥0x7fb628aff230
print(ps.work()) # <__main__.PythonStudent object at 🔥0x7fb628aff230> codes... and <__main__.PythonStudent object at 🔥0x7fb628aff230> studies... and <__main__.PythonStudent object at 🔥0x7fb628aff230> works...
```

A `Student` instance’s `work()` shows the same memory address in both outputs, proving `self` remains the `Student` instance across the delegation.

This applies to attributes too. Consider:

```run-python
class Person:
    def set_name(self, value):
        print("Setting name in Person")
        self.name = value

class Student(Person):
    def set_name(self, value):
        print("Student delegating to parent")
        super().set_name(value)

s = Student()  
s.set_name('Alex') # Student delegating to parent
				   # Setting name in Person
print(s.__dict__)  # {'name': 'Alex'}
```

A `Student` instance calling `set_name("Alex")` sets `name` in its own namespace, not `Person`’s, because `self` is the `Student` instance. Check its `__dict__`: `{'name': 'Alex'}`.

In `__init__`, this is super useful:

```run-python
class Person:
    def __init__(self, name):
        self.name = name

class Student(Person):
    def __init__(self, name, student_number):
        super().__init__(name)
        self.student_number = student_number

s = Student('Python', 30)  
print(s.__dict__)  # {'name': 'Python', 'student_number': 30}
```

A `Student("Python", 30)` instance has `__dict__` as `{'name': 'Python', 'student_number': 30}`. **==Without an override, the parent’s `__init__` runs automatically==**, but if you override it, you must use `super()` to include the parent’s logic.

Now, a practical example with properties:([This is tricky](https://www.udemy.com/course/python-3-deep-dive-part-4/learn/lecture/14192392?start=995#notes))

```run-python
from math import pi
from numbers import Real

class Circle:
    def __init__(self, r):
        self._set_radius(r)
        self._area = None
        self._perimeter = None

    def _set_radius(self, r):
        if isinstance(r, Real) and r > 0:
            self._r = r
            self._area = None
            self._perimeter = None
        else:
            raise ValueError("Radius must be a positive real number")

    @property
    def radius(self):
        return self._r

    @radius.setter
    def radius(self, r):
        self._set_radius(r)

    @property
    def area(self):
        if self._area is None:
            self._area = pi * self.radius ** 2
        return self._area

    @property
    def perimeter(self):
        if self._perimeter is None:
            self._perimeter = 2 * pi * self.radius
        return self._perimeter

class UnitCircle(Circle):
    def __init__(self):
        super().__init__(1)

    @property
    def radius(self):
        return super().radius  # Read-only, uses Circle's getter

u = UnitCircle()
print(u.radius)
```

A `UnitCircle` starts with radius 1. You can read `radius`, `area`, or `perimeter`, but setting `radius` fails because `UnitCircle` overrides it as read-only. Using `super().radius` fetches `Circle`’s getter, avoiding recursion. The `_set_radius` method ensures `__init__` validates the radius without clashing with the property.

That’s the tutorial—**`super()` delegates up the chain**, binding to the calling instance, and shines in `__init__` and beyond. Next, we’ll test it out!
