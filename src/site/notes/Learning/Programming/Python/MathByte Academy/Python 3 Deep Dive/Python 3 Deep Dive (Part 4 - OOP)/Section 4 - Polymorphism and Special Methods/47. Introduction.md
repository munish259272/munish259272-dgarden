# Exploring Polymorphism in Python

## What is Polymorphism?

**Polymorphism** means defining behavior that adapts to different object types. Python embraces this with **duck typing**: "If it walks like a duck and quacks like a duck, it’s a duck." Translation: If an object has the right behaviors, we can use it—type doesn’t matter.

---

## Duck Typing in Action

Python’s iteration is a classic example:

```python
for item in [1, 2, 3]:  # List
    print(item)
for item in (1, 2, 3):  # Tuple
    print(item)
```

- **Requirement**: The object must support the **iterable protocol** (has `__iter__` or `__getitem__`).
- **Flexibility**: Works with lists, tuples, dictionaries, generators, or custom classes—type is irrelevant.

Next step: `iter()` needs an iterator (with `__next__`):

```python
class MyRange:
    def __init__(self, end):
        self._end = end
        self._current = 0
    
    def __iter__(self):
        return self
    
    def __next__(self):
        if self._current >= self._end:
            raise StopIteration
        value = self._current
        self._current += 1
        return value

for i in MyRange(3):  # Custom class
    print(i)  # 0, 1, 2
```

- **Key Point**: **Behavior matters, not type**.

> [!question]- What is `__iter__` and `__next__` ? is the presence of `__iter__` causes it to be called automatically when called with `iter()` ?
>  Let’s break down `__iter__` and `__next__` in your `MyRange` class, along with your question about `iter()`.
> 
> ### What is `__iter__`?
> - **Definition**: `__iter__` is a special (dunder) method that makes an object **iterable**. It’s part of Python’s **iterable protocol**.
> - **Purpose**: It must return an **iterator**—an object with a `__next__` method.
> - **How It Works**: When you use `for`, `iter()`, or anything expecting an iterable, Python calls `__iter__` to get an iterator.
> - **In Your Code**: 
>   ```python
>   def __iter__(self):
>       return self
>   ```
>   - Here, `MyRange` returns itself as the iterator, meaning it’s both the iterable and the iterator (a common pattern for simple cases).
> 
> ### What is `__next__`?
> - **Definition**: `__next__` is a special method that defines how an **iterator** produces values. It’s part of the **iterator protocol**.
> - **Purpose**: It returns the next value in the sequence or raises `StopIteration` when exhausted.
> - **How It Works**: Python calls `__next__` repeatedly on the iterator until `StopIteration` is raised.
> - **In Your Code**:
>   ```python
>   def __next__(self):
>       if self._current >= self._end:
>           raise StopIteration
>       value = self._current
>       self._current += 1
>       return value
>   ```
>   - Returns `0`, `1`, `2` incrementally, stopping at `3` by raising `StopIteration`.
> 
> ### How They Work Together
> - **For Loop**: When you write `for i in MyRange(3)`:
>   1. Python calls `iter(MyRange(3))`, which invokes `MyRange(3).__iter__()` → returns the object itself.
>   2. Python then calls `__next__()` on that object repeatedly:
>      - First: `0`, `_current` becomes `1`.
>      - Second: `1`, `_current` becomes `2`.
>      - Third: `2`, `_current` becomes `3`.
>      - Fourth: `_current >= 3`, raises `StopIteration`, loop ends.
> - **Output**: Prints `0`, `1`, `2`.
> 
> ### Does `__iter__` Get Called Automatically with `iter()`?
> - **Answer**: Yes! The presence of `__iter__` is exactly why `iter()` works. 
> - **Details**: 
>   - `iter(obj)` looks for `__iter__` on `obj` and calls it. If found, it expects the result to be an iterator (with `__next__`).
>   - If `__iter__` isn’t defined, `iter()` tries `__getitem__` (for sequence-like access), but that’s not your case.
> - **Proof**:
>   ```python
>   mr = MyRange(3)
>   iterator = iter(mr)  # Calls mr.__iter__(), returns mr
>   print(next(iterator))  # Calls mr.__next__(), prints 0
>   print(next(iterator))  # Calls mr.__next__(), prints 1
>   ```
> - **Key Point**: **Defining `__iter__` hooks into `iter()`**, enabling iteration.
> 
> ### Summary
> - **`__iter__`**: Makes `MyRange` iterable, returning an iterator (here, itself).
> - **`__next__`**: Makes it an iterator, yielding values one by one.
> - **Automatic?**: Yes, `iter()` triggers `__iter__` automatically when present.

## Polymorphic Operators

Operators like `+`, `-`, `*`, `/` adapt to types:

```python
print(1 + 2)          # 3 (integers)
print(1.5 + 2.5)      # 4.0 (floats)
print([1, 2] + [3])   # [1, 2, 3] (lists)
print("a" * 3)        # "aaa" (strings)
```

- **Support**: Works with ints, floats, lists, tuples, and more.
- **Custom Types**: Add `__add__` to make your class support `+`.

**Takeaway**: **Operators are inherently polymorphic**.

## Special (Dunder) Methods

Python uses **special methods** (e.g., `__method__`) to enable this flexibility:

```python
class Vector:
    def __init__(self, x, y):
        self.x = x
        self.y = y
    
    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)

v1 = Vector(1, 2)
v2 = Vector(3, 4)
print((v1 + v2).x, (v1 + v2).y)  # 4 6
```

- **Examples**:
  - `__init__`: Object creation.
  - `__enter__`, `__exit__`: Context managers (`with` statement).
  - `__getitem__`, `__setitem__`: Sequence access (`obj[0]`).
  - `__iter__`, `__next__`: Iteration.
  - `__len__`: `len(obj)`.
  - `__contains__`: `in` operator.
- **Result**: Python uses these if they’re there—type doesn’t matter.

**Key Point**: **Dunder methods unlock polymorphic behavior**.

## ⚠️ Don’t Invent Dunder Names

Avoid creating custom `__myfunc__`-style names:

```python
# Bad idea
class BadIdea:
    def __myfunc__(self):  # Don’t do this
        pass
```

- **Why?**:
  - **Clashes**: Future Python updates might use your name, breaking code.
  - **Readability**: Developers expect `__xx__` to be Python’s, not yours.
- **Fix**: Use `_myfunc` (private convention) or plain `myfunc`.

**Takeaway**: **Leave dunder naming to Python**.
