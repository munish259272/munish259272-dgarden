# Diving Deeper into Exceptions

## What Are Exceptions?

Exceptions are **objects**—instances of classes:

- **Nature**: Not always errors, but signals of special conditions.
- **Example**: `StopIteration`—expected when iteration ends, not a failure.
- **Key Point**: **Exceptions are instances, not just errors**.

---


### Call Stack Explained

![[125. Python Exceptions - Lecture.pdf#page=3]]

- **Start**: `main → func1` (stack: `[func1]`).
- **Next**: `func1 → func2` (stack: `[func1, func2]`).
- **Then**: `func2 → func3` (stack: `[func1, func2, func3]`).
- **Read**: Bottom (origin) to top (latest).

### Exception Propagation

![[125. Python Exceptions - Lecture.pdf#page=4]]

> [!note] 
> Execution of function 3 is stopped at this time when there is an exception( It is basically an exit). It passes the exception to function 2 and if it is unhandled there it also stops the execution and passes it further to function 1 and at last the program terminate in function 1 if unhandled.
> Stack trace contains the information about this call stack.

Exceptions **move up the call stack** if unhandled:

```python
def func3():
    raise ValueError("Oops")
def func2():
    func3()
def func1():
    func2()

func1()  # Propagates: func3 → func2 → func1 → crash
```

- **Flow**: From `func3` to `func2`, then `func1`, until handled or program exits.
- **Stack Trace**: Logs the path (e.g., "ValueError in func3, called by func2, called by func1").
- **Key Point**: **Unhandled exceptions escalate**.

## Handling Exceptions

Use `try` to **intercept propagation**:

```python
try:
    raise ValueError("Bad value")
except ValueError:
    print("Caught it")
```

- **Purpose**: Prevent crashes, manage flow.
- **Key Point**: **Handle to recover or redirect**.

### Try Statement Structure
- **Clauses**: `try`, `except`, `else`, `finally`.
- **Like**: `if`/`elif`/`else`—a compound statement.
- **Key Point**: **Multi-part for flexibility**.

---

## Raising Exceptions

Manually trigger with `raise`:

```python
raise ValueError("Invalid input")
```

- **Options**: Catch and re-raise, or raise a new one.
- **Key Point**: **Control flow with `raise`**.

---

## Exception Categories

Two main types:

1. **Compilation**: `SyntaxError`—caught when loading code.
2. **Runtime**: `ValueError`, `KeyError`, `StopIteration`—during execution.
- **Key Point**: **Timing defines exception type**.

---

## Python’s Exception Hierarchy

Built-ins form a **class tree**:

- **Root**: `BaseException`.
- **Top-Level**:
  - `SystemExit` (e.g., `sys.exit()`).
  - `KeyboardInterrupt` (Ctrl+C).
  - `GeneratorExit` (generator close).
  - `Exception` (most others).
- **Under `Exception`**:
  - `ArithmeticError` → `ZeroDivisionError`.
  - `LookupError` → `IndexError`, `KeyError`.
  - `ValueError`, `TypeError`, etc.
- **Key Point**: **Hierarchy groups related exceptions**.

### Why Not `BaseException`?
- **Broad**: Catches *everything* (`KeyboardInterrupt` too).
- **Usual**: Inherit from `Exception` for user exceptions.
- **Key Point**: **`Exception` is the practical base**.

---

## Inheritance Benefits

Subclasses **inherit catchability**:

```python
try:
    raise IndexError("Bad index")
except LookupError:
    print("Caught")  # Works: IndexError is a LookupError
```

- **Chain**: `IndexError` < `LookupError` < `Exception`.
- **Broad Catch**: `except Exception` grabs all subclasses.
- ⚠️ **Caution**: Too broad (`Exception`) can mask issues.
- **Key Point**: **Hierarchy enables flexible catching**.

---

## Basic `try`/`except`

Minimal handling setup:

```python
try:
    x = 1 / 0
except ZeroDivisionError as e:
    print(f"Error: {e}")  # Error: division by zero
```

- **Scope**: Keep `try` small for clarity.
- **Handle**: `as e` gives the exception object.
- **Key Point**: **Targeted `try` aids recovery**.
