So far, we’ve explored **inheriting and overriding methods** from a parent class. Overriding can be tricky because we need to understand how method resolution works for each call. On the other hand, **extending** is straightforward—we’re simply adding new functionality that doesn’t exist in the inheritance chain by defining new methods or attributes in the child class. This applies to attributes too, but it’s more engaging with methods. Let’s dive into an example.

We’ll start with a `Person` class that inherits basic functionality from the `object` class, with no special methods yet. Next, we’ll create a `Student` class that inherits from `Person` and add a `study` method:

```python
class Person:
    pass

class Student(Person):
    def study(self):
        return "study " * 3
```

If we instantiate `Person`, like `p = Person()`, and try `p.study()`, we’ll get an `AttributeError` because `study` doesn’t exist in `Person`. But with `s = Student()`, calling `s.study()` works fine—it returns `"study study study"`. This shows **extension in action**: `Student` inherits from `Person` and adds new functionality.

Now, let’s revisit overriding. Suppose we redefine `Person` with a `routine` method that calls `eat`, `study`, and `sleep`:

```python
class Person:
    def routine(self):
        return self.eat() + self.study() + self.sleep()
    
    def eat(self):
        return "person eats "
    
    def sleep(self):
        return "person sleeps "

class Student(Person):
    def study(self):
        return "student studies "
```

If we create `p = Person()` and call `p.routine()`, it fails with an `AttributeError` because `study` isn’t defined in `Person`. But with `s = Student()`, `s.routine()` works: it uses `eat` and `sleep` from `Person` and `study` from `Student`, returning `"person eats student studies person sleeps"`. This highlights how **methods bind to the instance**—`self` is the `Student` instance, so it finds `study` in `Student`.

To make `Person.routine()` more robust, we can check for `study`’s existence:

```python
class Person:
    def routine(self):
        result = self.eat()
        if hasattr(self, "study") and callable(self.study):
            result += self.study()
        result += self.sleep()
        return result
    
    def eat(self):
        return "person eats "
    
    def sleep(self):
        return "person sleeps "

class Student(Person):
    def study(self):
        return "student studies "
```

Now, `p = Person(); p.routine()` returns `"person eats person sleeps"`, while `s = Student(); s.routine()` returns `"person eats student studies person sleeps"`. This avoids errors and shows **flexible extension**.

Sometimes, we want `Person` to be a base class only, not instantiated directly. In languages like Java, this is handled with *abstract classes*. Python has this too via the `abc` module, where a base class defines required methods (like `study`) that subclasses must implement. We won’t cover that now, but it’s a way to enforce structure.

This binding logic applies to attributes too, not just methods. Consider an `Account` class with a class attribute `apr` and instance attributes:

```python
class Account:
    apr = 3.0
    
    def __init__(self, account_number, balance):
        self.account_number = account_number
        self.balance = balance
        self.account_type = "generic account"
    
    def calc_interest(self):
        return f"calc interest on {self.account_type} with APR = {self.apr}"
```

For `a = Account("123", 100)`, `a.apr` is `3.0` (from the class), `a.account_type` is `"generic account"`, and `a.calc_interest()` returns `"calc interest on generic account with APR = 3.0"`. Now, a `Savings` subclass:

```python
class Savings(Account):
    apr = 5.0
    
    def __init__(self, account_number, balance):
        self.account_number = account_number
        self.balance = balance
        self.account_type = "savings account"
```

With `s = Savings("234", 200)`, `s.apr` is `5.0`, `s.account_type` is `"savings account"`, and `s.calc_interest()` returns `"calc interest on savings account with APR = 5.0"`. Even though `calc_interest` is from `Account`, `self` is a `Savings` instance, so it uses `Savings`’s attributes.

Be cautious with class vs. instance attributes. If `calc_interest` used `Account.apr` instead of `self.apr`, it’d always return `3.0`, ignoring `Savings.apr`. A better approach is:

```python
def calc_interest(self):
    return f"calc interest on {self.account_type} with APR = {type(self).apr}"
```

This uses the instance’s class (`Savings` or `Account`) to fetch `apr`, ensuring correctness. If someone sets `s.apr = 10` on the instance, `type(self).apr` still uses the class’s `5.0`, avoiding unintended overrides.

That wraps up our look at inheritance, overriding, and extension—key concepts made clear through this tutorial!
