# Raising and Handling Exceptions

## The `raise` Keyword

Trigger exceptions with **`raise`**:

```python
raise ValueError("Custom exception")
```

- **Options**: Direct or instantiate first:
  ```python
  e = ValueError("Custom exception")
  raise e
  ```
- **Result**: Unhandled, it crashes (or Jupyter catches it).
- **Key Point**: **`raise` starts propagation**.

---

## Propagation Basics

Unhandled exceptions **bubble up**:

```python
raise ValueError("Error")
# Jupyter catches it, prints trace, continues
```

- **Outside Jupyter**: Program exits with error.
- **Key Point**: **Handling stops crashes**.

---

## Handling with `try`

Catch specific exceptions:

```python
try:
    raise ValueError("Custom message")
except ValueError as e:
    print(str(e))  # Custom message
```

- **Multiple Args**:
  ```python
  raise ValueError("Main", "Secondary")
  # Prints: ('Main', 'Secondary')
  ```
- **Key Point**: **Specific catches control flow**.

### Multiple Handlers
```python
def func1():
    raise IndexError("Bad index")

try:
    func1()
except ValueError:
    print(f"ValueError: {e}")
except IndexError as e:
    print(f"IndexError: {e}")
```

- **Order**: Specific to broad (e.g., `ValueError` before `Exception`).
- **Key Point**: **Order respects hierarchy**.

---

## `finally` Clause

Runs **always**:

```python
try:
    raise ValueError
except ValueError:
    print("Handled")
finally:
    print("Finally runs")
```

- **Even Unhandled**:
  ```python
  try:
    raise KeyError
  finally:
    print("Still runs")  # Before propagation
  ```
- **Use**: Cleanup (e.g., closing files).
- **Key Point**: **`finally` ensures execution**.

---

## `else` Clause

Runs if **no exceptions**:

```python
try:
    x = 10
except ValueError:
    print("Error")
else:
    print("No exception")  # Only if try succeeds
```

- **Requires `except`**: No standalone `try`/`else`.
- ⚠️ **Pitfall**: Not the same as post-`except` code:
  ```python
  try:
    raise ValueError
  except ValueError:
    print("Error")
  print("Not equivalent")  # Runs anyway
  ```
- **Key Point**: **`else` isolates success**.

## Nested `try` Statements

Handle complex logic:

```python
import json

json_data = '''
{
    "Alex": {"age": 25},
    "Brian": {"age": 21, "city": "London"},
    "Guido": {"age": "fifty"}
}
'''
data = json.loads(json_data)

class Person:
    __slots__ = ("name", "_age")
    
    def __init__(self, name): 
	    self.name = name; 
	    self._age = None
	    
    @property
    def age(self): 
	    return self._age
    
    @age.setter
    def age(self, value):
        if isinstance(value, int) and value >= 0:
            self._age = value
        else:
            raise ValueError("Invalid age")
            
    def __repr__(self): 
	    return f"Person(name={self.name}, age={self._age})"

persons = []
for name, attrs in data.items():
    try:
        p = Person(name)
        for attr_name, attr_value in attrs.items():
            try:
                setattr(p, attr_name, attr_value)
            except AttributeError:
                print(f"Ignoring attribute {attr_name} for {name}")
        else:
            persons.append(p)
    except ValueError as e:
        print(f"Data for {name} has invalid value: {e}")
```

- **Output**: Ignores `city`, skips `Guido`, adds `Alex` and `Brian`.
- **Key Point**: **Nesting refines handling**.


> [!question]- Why we cannot use `p.attr_name = attr_value` instead of `setattr(p, attr_name, attr_value)` in the above code ?
> 
> You cannot use `p.attr_name = attr_value` instead of `setattr(p, attr_name, attr_value)` in the provided code. The reason is that Python interprets `p.attr_name` as trying to access an attribute literally named "attr_name" on the object `p`, which does not exist. 
> 
> In contrast, `setattr(p, attr_name, attr_value)` uses the value of the variable `attr_name` as the name of the attribute to set on the object `p`. This means if `attr_name` is `"age"`, it effectively executes `p.age = attr_value`. 
> 
> Using `setattr` allows you to dynamically set attributes using their names stored in variables, which is essential when iterating over dictionary keys or similar use cases where attribute names are determined at runtime.

### Alternative (No Nesting)
```python
persons = []
for name, attrs in data.items():
    p = Person(name)
    skip = False
    for attr_name, attr_value in attrs.items():
        try:
            setattr(p, attr_name, attr_value)
        except AttributeError:
            print(f"Ignoring attribute {attr_name} for {name}")
        except ValueError as e:
            print(f"Data for {name} has invalid value: {e}")
            skip = True
            break
    if not skip:
        persons.append(p)
```

- **Drawback**: More flags, less elegant.
- **Key Point**: **Nesting often cleaner**.

## EAFP in Action

**EAFP** (Easier to Ask Forgiveness):

```python
class ConversionError(Exception):  
    pass  
  
  
def convert_int(val):  
    if not isinstance(val, int):  
        raise TypeError  
    if val not in (0, 1):  
        raise ValueError("Integer values 0 or 1 only")  
    return bool(val)  
  
  
def convert_str(val):  
    if not isinstance(val, str):  
        raise TypeError  
  
    v = val.casefold()  
    if v in ("0", "f", "false"):  
        return False  
    if v in ("1", "t", "true"):  
        return True  
    raise ValueError("Valid strings: 0, 1, t, f, true, false")  
  
  
def make_bool(val):  
    try:  
        try:  
            b = convert_int(val)  
        except TypeError:  
            try:  
                b = convert_str(val)  
            except TypeError:  
                raise ConversionError(f"Type {type(val)} inadmissible")  
  
    except ValueError as e:  
        raise ConversionError(f"Value {val} cannot be converted: {e}")  
    else:  
        return b
```

- **Test**:
  ```python
  for v in (True, 0, "t", "False", 10, "abc", 1.0):
      try:
          print(v, make_bool(v))
      except ConversionError as e:
          print(v, e)
  ```
- **Key Point**: **EAFP simplifies logic**.

> [!question]- Does nested `try` introduces a scope ?
> 
> Nested `try` blocks do not introduce a new scope in Python. The variables defined within a `try`, `except`, `else`, or `finally` block are accessible outside of that block, provided they are defined before any potential exception that could jump execution out of the block. This behavior is consistent with the general scoping rules in Python, where variables defined inside a function are accessible throughout the entire function.
> 
> For example:
> 
> ```python
> try:
>     x = 10
>     raise ValueError("Example")
> except ValueError as e:
>     print(f"Caught an exception: {e}")
> 
> print(x)  # x is still accessible here
> ```
> 
> In this example, the variable `x` remains accessible after the `try-except` block because it was successfully assigned before the exception was raised. Similarly, variables initialized in nested `try` blocks remain accessible within their parent blocks, as long as they were assigned before any exceptions that could disrupt control flow.
> 
> > [!question]- What about `with`, `if else` etc. Do they introduce scope ?
> > 
> > In Python, constructs like `with`, `if-else`, and loops (such as `for` and `while`) do not introduce a new scope. This means that variables defined within these blocks are accessible outside of them, provided they are defined before any potential disruption in the control flow.
> > 
> > Here’s how each construct behaves with respect to variable scope:
> > 
> > ### `with` Statement:
> > The `with` statement is used for resource management, often with context managers. It does not create a new scope.
> > 
> > ```python
> > with open('file.txt', 'r') as file:
> >     content = file.read()
> > 
> > print(content)  # `content` is accessible here.
> > ```
> > 
> > ### `if-else` Statement:
> > Conditionals like `if`, `elif`, and `else` also do not introduce a new scope.
> > 
> > ```python
> > if True:
> >     message = "Hello, World!"
> > else:
> >     message = "Goodbye!"
> > 
> > print(message)  # `message` is accessible here.
> > ```
> > 
> > ### Loops (`for`, `while`):
> > Loops do not create a new scope either. Variables defined or modified within a loop are available after the loop ends.
> > 
> > ```python
> > for i in range(3):
> >     number = i
> > 
> > print(number)  # Last value assigned to number in the loop is accessible here.
> > ```
> > 
> > ### Functions:
> > Unlike the constructs mentioned above, functions do create a new local scope. Variables defined inside a function are local to that function unless explicitly declared as global or nonlocal.
> > 
> > ```python
> > def my_function():
> >     local_var = "I'm local to this function."
> > 
> > # print(local_var)  # This would raise an error because local_var is not accessible here.
> > ```
> > 
> > In summary, most control flow structures in Python share the same scope with their surrounding code block, while functions define their own local scopes. This behavior promotes flexibility when writing code, allowing variables to be reused across different constructs without re-declaration outside of functions.

- **Works**: But more checks, less flexible.
- **Key Point**: **LBYL grows complex**.

## Real-World EAFP

Flexible indexing:

```python
def get_item(seq, idx, default=None):
    try:
        return seq[idx]
    except (IndexError, TypeError, KeyError):
        return default
```

- **Handles**: Lists, dicts, slices—effortlessly.
- **Test**:
  ```python
  print(get_item([1, 2, 3], 0))          # 1
  print(get_item([1, 2, 3], 10, "Nope")) # Nope
  print(get_item({"a": 100}, "a"))       # 100
  print(get_item([1, 2, 3, 4], slice(1, 3))) # [2, 3]
  ```

### LBYL Struggle
```python
def get_item_ask(seq, idx, default=None):
    if hasattr(seq, "__getitem__"):
        if isinstance(seq, dict):
            return seq.get(idx, default)
        if isinstance(idx, int) and idx < len(seq):
            return seq[idx]
    return default
```

- **Fails**: Infinite sequences, slices, etc.
- **Key Point**: **EAFP adapts better**.
