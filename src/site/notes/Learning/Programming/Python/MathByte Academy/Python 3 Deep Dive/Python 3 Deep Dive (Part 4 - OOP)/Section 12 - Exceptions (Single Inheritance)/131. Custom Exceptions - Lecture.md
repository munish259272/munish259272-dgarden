# Crafting Custom Exceptions

## Defining Custom Exceptions

Create exceptions by **subclassing**:

```python
class WidgetException(Exception):
    "Base exception for Widget library"
```

- **Best Practice**: Inherit from `Exception`, not `BaseException`.

> [!question]- "<font color="#000000" style="font-weight:normal;font-style:italic;">Inherit from `Exception`, not `BaseException`</font>" - Why ?
> Inheriting from `Exception` instead of `BaseException` is recommended because of how Python's exception handling works and what each class represents in the hierarchy:
> 
> - **`BaseException`**: This is the root of all exceptions in Python. It includes not just "normal" exceptions (like `ValueError` or `TypeError`) but also special system-level exceptions like `SystemExit`, `KeyboardInterrupt`, and `GeneratorExit`. These are typically used for control flow or program termination, not for typical error handling. If you inherit directly from `BaseException`, your custom exception becomes a sibling to these, which can lead to unexpected behavior.
> 
> - **`Exception`**: This is a subclass of `BaseException` and serves as the base for almost all standard error-type exceptions (e.g., `ValueError`, `IndexError`). It excludes those special system exceptions. Most code uses `except Exception` to catch "regular" errors, assuming it won’t catch things like `SystemExit` or `KeyboardInterrupt`, which are meant to bypass normal error handling.
> 
> ### Why It Matters
> If you inherit from `BaseException`:
> - A broad `except Exception` clause (common in code) **won’t catch your custom exception**, because it’s not under `Exception`. This can confuse users or break expected handling.
> - Your exception might unintentionally get grouped with system-level exceptions, which are usually handled separately (or not at all) by design.
> 
> By inheriting from `Exception`:
> - Your custom exception fits into the standard error-handling flow, caught by `except Exception` when needed.
> - It avoids interference with system-level exceptions, keeping their special behavior intact.
> 
> ### Example
> ```python
> class MyBaseError(BaseException): pass
> class MyNormalError(Exception): pass
> 
> try:
>     raise MyBaseError("Oops")
> except Exception:
>     print("Caught!")  # Won’t run—MyBaseError isn’t under Exception
> 
> try:
>     raise MyNormalError("Oops")
> except Exception:
>     print("Caught!")  # Runs—MyNormalError is under Exception
> ```
> 
> **Key Point**: Inherit from `Exception` to ensure **compatibility with typical error handling** and avoid surprises with system exceptions.

- **Why**: Broad `except Exception` misses `BaseException` subclasses (e.g., `SystemExit`).
- **Key Point**: **`Exception` aligns with standard handling**.

## Exception Hierarchy

Python’s hierarchy:

- **`BaseException`**: Root (includes `SystemExit`, `KeyboardInterrupt`).
- **`Exception`**: Most exceptions live here.

Custom hierarchy example:

```python
class WidgetException(Exception): "Base for Widget library"
class StockException(WidgetException): "Stock-related issues"
class OutOfStockException(StockException): "Item unavailable"
class OverInventoryLimit(StockException): "Too much stock"
class SalesException(WidgetException): "Sales-related issues"
class OverOrderLimit(SalesException): "Order exceeds limit"
class InvalidCouponCode(SalesException): "Bad coupon"
```

- **Benefit**: Trap at any level:
  ```python
  try:
      raise OutOfStockException
  except StockException:
      print("Stock issue")  # Catches OutOfStockException
  except WidgetException:
      print("Widget issue")  # Catches all
  ```
- **Key Point**: **Hierarchy enables flexible catching**.

## Extending Functionality

Exceptions are **classes**, so enhance them:

```python
class WidgetException(Exception):
    def __init__(self, *args, code=500):
        super().__init__(*args)
        self.code = code
        print(f"Logging: {self}")  # Auto-logging

    def to_json(self):
        return {"code": self.code, "message": str(self)}

class OutOfStockException(WidgetException):
    def __str__(self):
        return f"Out of stock: {self.args}"
```

- **Test**:
  ```python
  e = OutOfStockException("Item X")
  print(e)           # Logging: Out of stock: ('Item X',)
                     # Out of stock: ('Item X',)
  print(e.to_json()) # {'code': 500, 'message': "Out of stock: ('Item X',)"}
  ```
- **Use**: Consistent output (e.g., APIs), auto-logging.
- **Key Point**: **Custom methods add power**.

## Multiple Inheritance

Exceptions can inherit from **multiple parents**:

```python
class SalesException(WidgetException): "Sales issues"
class InvalidSalePrice(SalesException, ValueError): "Bad price"

e = InvalidSalePrice("Price too high")
print(isinstance(e, SalesException))  # True
print(isinstance(e, ValueError))      # True
```

- **Handling**:
  ```python
  try:
      raise InvalidSalePrice("Price too high")
  except ValueError:
      print("Caught as ValueError")  # Works
  except SalesException:
      print("Caught as SalesException")  # Also works
  ```
- **Why**: Flexibility for users (trap as `ValueError` or specific type).
- **Key Point**: **Multiple inheritance broadens catchability**.
