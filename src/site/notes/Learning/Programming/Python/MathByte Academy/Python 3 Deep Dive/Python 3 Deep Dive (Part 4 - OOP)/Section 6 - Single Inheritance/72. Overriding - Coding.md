Let’s explore how **inheritance** and **method overriding** work in Python, focusing on the `__str__` and `__repr__` methods, and then extend that understanding to a custom inheritance chain.

Every custom class in Python implicitly inherits from the `object` class, bringing along methods like `__str__`. Start with a basic `Person` class:
```python
class Person:
    pass

p = Person()
print(str(p))  # Outputs something like "<__main__.Person object at 0x...>"
```
This default string representation comes from `object`’s `__str__` implementation.

Now, **override** `__str__`:
```python
class Person:
    def __str__(self):
        return "Person class"

p = Person()
print(str(p))  # Outputs: "Person class"
```
The `str()` function calls our `__str__`, overriding `object`’s version because Python prioritizes the closest implementation in the inheritance chain.

What if we only define `__repr__`?
```python
class Person:
    def __repr__(self):
        return "Person with debugging info"

p = Person()
print(str(p))  # Outputs: "Person with debugging info"
```
Here’s why: `object`’s `__str__` doesn’t return a string directly—it calls `__repr__` as a fallback. Since we overrode `__repr__`, `str(p)` triggers `object.__str__` ( bound to `p`), which then calls our `__repr__`. The **binding** to `p` ensures our version runs, not `object`’s.

### Understanding Binding in Inheritance 📌
When a method is called, Python starts at the instance’s class and moves up the hierarchy, using the first matching method it finds—**bound to the instance**. Let’s test this with a chain:
```python
class Shape:
    def __init__(self, name):
        self.name = name
    
    def info(self):
        return f"Shape.info called for {self.name}"
    
    def extended_info(self):
        return f"Shape.extended_info called for {self.name}, plus {self.info()}"

class Polygon(Shape):
    def __init__(self, name):
        self.name = name  # Should use super().__init__(name) in practice
    
    def info(self):
        return f"Polygon.info called for {self.name}"

p = Polygon("square")
print(p.info())          # "Polygon.info called for square"
print(p.extended_info()) # "Shape.extended_info called for square, plus Polygon.info called for square"
```
- `p.info()` uses `Polygon`’s `info` because `p` is a `Polygon` instance.
- `p.extended_info()` uses `Shape`’s version (not defined in `Polygon`), but inside it, `self.info()` calls `Polygon`’s `info` since `self` is bound to `p`, a `Polygon`.

### `__str__` and `__repr__` in Action
Mimic this with `__str__` and `__repr__`:
```python
class Person:
    def __str__(self):
        print("Person.__str__ called")
        return self.__repr__()

class Student(Person):
    def __repr__(self):
        return f"Student at {hex(id(self))}"

s = Student()
print(str(s))  # Outputs: "Person.__str__ called" then "Student at 0x..."
```
- `str(s)` finds `__str__` in `Person`, which calls `self.__repr__`.
- `self` is `s` (a `Student`), so it uses `Student`’s `__repr__`, not `object`’s.

For a `Person` instance:
```python
p = Person()
print(str(p))  # "Person.__str__ called" then "<__main__.Person object at 0x...>"
```
Here, `__repr__` falls back to `object`’s since `Person` doesn’t define it.

### Key Takeaway 🌟
When analyzing method calls:
1. Identify the **instance’s class**.
2. Look for the method starting there, moving up the inheritance chain.
3. The method executed is **bound to the instance**, so overrides in the instance’s class take precedence, even if called indirectly.

This tutorial shows how Python’s inheritance and binding mechanics shape method resolution—crucial for mastering object-oriented programming!
