# 🎯 Query Parameters in FastAPI

In the previous tutorial, we learned about **Path Parameters**, which allow us to identify a specific resource using the URL path.

In this tutorial, we'll learn about **Query Parameters**, which are commonly used to **filter data**.

Think of a query parameter as asking:

> "I already know where the data is. Now I want only a specific subset of that data."

Query parameters are placed **after a question mark (`?`) in the URL**.

---

# 📌 What is a Query Parameter?

Suppose we have the URL:

```text
/books?category=science
```

Here:

|Part|Meaning|
|---|---|
|`/books`|Path|
|`?`|Start of query parameters|
|`category`|Parameter name|
|`science`|Parameter value|

This tells FastAPI:

> "Give me books, but only those whose category is Science."

---

# 🧠 Real-World Analogy

Imagine a library.

### Without Query Parameters

```text
/books
```

means:

> Show me all books.

### With Query Parameters

```text
/books?category=science
```

means:

> Show me only Science books.

The query parameter acts like a **filter**.

---

# 🎯 Creating a Query Parameter Endpoint

Let's create an endpoint that filters books by category.

```python
@app.get("/books/")
async def read_category_by_query(category: str):
    books_to_return = []

    for book in books:
        if book["category"].casefold() == category.casefold():
            books_to_return.append(book)

    return books_to_return
```

---

# 📌 Understanding the Code

## Function Parameter

```python
category: str
```

This parameter automatically receives the value from the URL.

For example:

```text
/books?category=science
```

becomes:

```python
category = "science"
```

FastAPI does this automatically.

---

## Create an Empty List

```python
books_to_return = []
```

This list will store matching books.

---

## Loop Through All Books

```python
for book in books:
```

We examine each book one at a time.

---

## Check for a Matching Category

```python
if book["category"].casefold() == category.casefold():
```

This comparison ignores uppercase/lowercase differences.

Example:

```python
"Science".casefold()
```

becomes

```python
"science"
```

---

## Add Matching Books

```python
books_to_return.append(book)
```

Every matching book is added to the result list.

---

## Return Results

```python
return books_to_return
```

Possible outcomes:

```python
[]
```

No matches found.

or

```python
[
    {...},
    {...}
]
```

Matching books found.

Suppose the user requests:

```python
category = "science"
```

```python

```

For every matching Science book:


# 📌 How Swagger Knows It Is a Query Parameter

Notice that our route is:

```python
@app.get("/books/")
```

There is **no path parameter**:

```python
/books/{something}
```

Instead, FastAPI sees a function parameter:

```python
category: str
```

and automatically treats it as a **query parameter**.

That's why Swagger generates:

```text
/books?category=science
```

instead of:

```text
/books/science
```

# 🎯 Combining Path Parameters and Query Parameters

In real-world applications, we often use **both together**.

Suppose our books are:

|Title|Author|Category|
|---|---|---|
|Book 2|Author Two|Science|
|Book 6|Author Two|Math|

We can already:

### Search by title

```text
/books/Title%20Two
```

or

### Filter by category

```text
/books?category=science
```

But what if we want:

> Show me books written by Author Two that belong to the Science category.

For this, we combine a path parameter and a query parameter.

---

# 📌 Endpoint

```python
@app.get("/books/{book_author}")
async def read_author_category_by_query(
    book_author: str,
    category: str
):
```

---

# How FastAPI Maps Values

Request:

```text
/books/Author%20Two?category=science
```

FastAPI automatically converts:

```python
book_author = "Author Two"
```

from the path parameter and

```python
category = "science"
```

from the query parameter.

---

# Full Example

```python
@app.get("/books/{book_author}")
async def read_author_category_by_query(
    book_author: str,
    category: str
):

    books_to_return = []

    for book in books:
        if (
            book["author"].casefold() == book_author.casefold()
            and
            book["category"].casefold() == category.casefold()
        ):
            books_to_return.append(book)

    return books_to_return
```

---

# 📌 Why Parentheses Are Better Than Backslashes

The instructor uses a line continuation character:

```python
if condition1 and \
   condition2:
```

A more Pythonic approach is:

```python
if (
    condition1
    and condition2
):
```

✅ Easier to read

✅ Less error-prone

✅ Preferred in modern Python

---

# 🎯 Path Parameters vs Query Parameters

|Feature|Path Parameter|Query Parameter|
|---|---|---|
|Purpose|Identify a resource|Filter a resource|
|Location|Inside URL path|After `?`|
|Example|`/books/Author Two`|`?category=science`|

---

> [!important]- Key Takeaways
> 
> ✅ **Path Parameters identify which resource you want**
> 
> ```text
> /books/Author%20Two
> ```
> 
> ✅ **Query Parameters filter the returned data**
> 
> ```text
> ?category=science
> ```
> 
> ✅ Query parameters automatically map to function parameters.
> 
> ```python
> category: str
> ```
> 
> ✅ Multiple filters can be combined.
> 
> ```text
> /books?category=science
> ```
> 
> ✅ Path parameters and query parameters are commonly used together.
> 
> ```text
> /books/Author%20Two?category=science
> ```

> [!tip]- Easy Way to Remember
> 
> Think of a query parameter as a **WHERE clause in SQL**.
> 
> ```sql
> SELECT *
> FROM books
> WHERE category = 'science'
> ```
> 
> URL equivalent:
> 
> ```text
> /books?category=science
> ```
> 
> **Path Parameter → Find the location/resource**
> 
> **Query Parameter → Filter the data returned from that resource**
