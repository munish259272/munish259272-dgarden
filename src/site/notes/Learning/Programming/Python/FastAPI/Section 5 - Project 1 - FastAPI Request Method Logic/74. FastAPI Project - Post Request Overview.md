# 📮 POST Request in FastAPI

So far, we have worked with **GET requests** using:

- **Path Parameters** → Values passed as part of the URL path.
- **Query Parameters** → Values passed after `?` in the URL.

Now we'll learn about another important HTTP method: **POST Request**.

---

# 🎯 What is a POST Request?

A **POST request** is used to **create new data** on the server.

Unlike a GET request, a POST request can send additional data inside a **Request Body**.

📌 **Key Difference**

|GET Request|POST Request|
|---|---|
|Used to retrieve data|Used to create data|
|Data usually comes from URL|Data comes from Request Body|
|No request body is typically used|Request body is commonly used|

---

# 📦 What is a Request Body?

A **Request Body** is a section of the HTTP request that contains data being sent to the server.

Instead of placing information in the URL, we send the entire object inside the body.

For example, suppose we want to add a new book:

```json
{
    "title": "Title Seven",
    "author": 2,
    "category": "Math"
}
```

This book information is sent inside the **body** of the POST request.

The FastAPI application receives this data and can store it in a database, list, file, etc.

---

# 📚 Example Scenario

Imagine Author 2 has written a new book:

|Field|Value|
|---|---|
|Title|Title Seven|
|Author|2|
|Category|Math|

Instead of putting all this information in the URL, we send it as the **body** of a POST request.

### Request URL

```text
http://localhost:8000/books/create_book
```

Notice that:

✅ No query parameters are used.

✅ No dynamic path parameters are used.

✅ The actual book data is sent in the request body.

---

# 🔄 How FastAPI Handles This Request

The endpoint will use `@app.post()` instead of `@app.get()`.

```python
@app.post("/books/create_book")
async def create_book(new_book):
    ...
```

Here:

- `new_book` represents the data received in the request body.
- FastAPI automatically reads the body and assigns it to `new_book`.

> [!question]- I only see a string `"create_book"` in `@app.post("/books/create_book")` just like in the GET request. How does it have a body?
> 
> This is a very important observation. The **body is NOT specified in the URL path**.
> 
> The line:
> 
> ```python
> @app.post("/books/create_book")
> ```
> 
> only tells FastAPI:
> 
> > "If a POST request arrives at `/books/create_book`, run this function."
> 
> It says **nothing about the body itself**.
> 
> ---
> 
> ### 📌 Think of an HTTP Request as Two Separate Parts
> 
> ```text
> POST /books/create_book HTTP/1.1
> Host: localhost:8000
> 
> {
>     "title": "Title Seven",
>     "author": 2,
>     "category": "Math"
> }
> ```
> 
> The request consists of:
> 
> ```text
> ┌────────────────────────────┐
> │ URL / Route                │
> │ /books/create_book         │
> └────────────────────────────┘
> 
> ┌────────────────────────────┐
> │ HTTP Method                │
> │ POST                       │
> └────────────────────────────┘
> 
> ┌────────────────────────────┐
> │ Request Body               │
> │ {                          │
> │   "title":"Title Seven"    │
> │   "author":2               │
> │   "category":"Math"        │
> │ }                          │
> └────────────────────────────┘
> ```
> 
> The route and the body are completely separate things.
> 
> ---
> 
> ### 🎯 Where Does FastAPI Get The Body From?
> 
> Consider:
> 
> ```python
> @app.post("/books/create_book")
> async def create_book(new_book):
>     return new_book
> ```
> 
> When FastAPI sees a POST request arrive, it looks at:
> 
> ```python
> new_book
> ```
> 
> and says:
> 
> > "I need a value for `new_book`. Let me check the request body."
> 
> So it automatically reads the body and assigns it to `new_book`.
> 
> Conceptually:
> 
> ```python
> Request Body
>       │
>       ▼
> {
>     "title": "Title Seven",
>     "author": 2
> }
>       │
>       ▼
> new_book
> ```
> 
> ---
> 
> ### 📌 Compare GET vs POST
> 
> #### GET Request
> 
> ```python
> @app.get("/books/{book_id}")
> async def get_book(book_id: int):
>     return book_id
> ```
> 
> Request:
> 
> ```text
> GET /books/5
> ```
> 
> FastAPI extracts:
> 
> ```python
> book_id = 5
> ```
> 
> from the URL.
> 
> ---
> 
> #### POST Request
> 
> ```python
> @app.post("/books/create_book")
> async def create_book(new_book: dict):
>     return new_book
> ```
> 
> Request:
> 
> ```http
> POST /books/create_book
> 
> {
>     "title": "Python",
>     "author": 1
> }
> ```
> 
> FastAPI extracts:
> 
> ```python
> new_book = {
>     "title": "Python",
>     "author": 1
> }
> ```
> 
> from the body.
> 
> ---
> 
> ### 📌 How Does the Client Send the Body?
> 
> The body is sent by the client (browser, Postman, curl, frontend app, mobile app, etc.).
> 
> Example with `curl`:
> 
> ```bash
> curl -X POST \
> http://localhost:8000/books/create_book \
> -H "Content-Type: application/json" \
> -d '{
>     "title":"Python",
>     "author":1
> }'
> ```
> 
> Notice:
> 
> ```bash
> -d '{
>     "title":"Python",
>     "author":1
> }'
> ```
> 
> This `-d` (data) part becomes the request body.
> 
> > [!question]- You mean every HTTP request has different parts like the address (URL), content type, request body, etc. which are sent to and fro, and it is part of the HTTP protocol, and the information is stacked and sent over the internet wires just like the TCP/IP protocol stack?
> > 
> > **Yes, exactly.** You're thinking about it in the correct way.
> > 
> > Think of the protocols as layers stacked on top of each other:
> > 
> > ```text
> > Application Layer
> > ┌─────────────────────────┐
> > │ HTTP                    │
> > │ GET /books              │
> > │ POST /books/create_book │
> > │ Headers                 │
> > │ Body                    │
> > └─────────────────────────┘
> >             │
> >             ▼
> > Transport Layer
> > ┌─────────────────────────┐
> > │ TCP                     │
> > │ Ports                   │
> > │ Sequence Numbers        │
> > │ Reliability             │
> > └─────────────────────────┘
> >             │
> >             ▼
> > Internet Layer
> > ┌─────────────────────────┐
> > │ IP                      │
> > │ Source IP               │
> > │ Destination IP          │
> > └─────────────────────────┘
> >             │
> >             ▼
> > Link Layer
> > ┌─────────────────────────┐
> > │ Ethernet / WiFi         │
> > │ MAC Addresses           │
> > └─────────────────────────┘
> > ```
> > 
> > ---
> > 
> > ## 📌 HTTP Creates a Message
> > 
> > When you write:
> > 
> > ```python
> > @app.post("/books/create_book")
> > ```
> > 
> > and a client sends:
> > 
> > ```http
> > POST /books/create_book HTTP/1.1
> > Host: localhost:8000
> > Content-Type: application/json
> > 
> > {
> >     "title": "Python",
> >     "author": 1
> > }
> > ```
> > 
> > this is an **HTTP message**.
> > 
> > The HTTP protocol defines its structure:
> > 
> > ### 1. Request Line
> > 
> > ```http
> > POST /books/create_book HTTP/1.1
> > ```
> > 
> > Contains:
> > 
> > - Method (`POST`)
> >     
> > - URL path (`/books/create_book`)
> >     
> > - HTTP version
> >     
> > 
> > ---
> > 
> > ### 2. Headers
> > 
> > ```http
> > Host: localhost:8000
> > Content-Type: application/json
> > ```
> > 
> > Contains metadata about the request.
> > 
> > Examples:
> > 
> > ```http
> > Content-Type: application/json
> > Authorization: Bearer xxxxx
> > User-Agent: Chrome
> > Accept: application/json
> > ```
> > 
> > ---
> > 
> > ### 3. Body
> > 
> > ```json
> > {
> >     "title": "Python",
> >     "author": 1
> > }
> > ```
> > 
> > Contains the actual data being sent.
> > 
> > ---
> > 
> > ## 🎯 Then TCP Takes The HTTP Message
> > 
> > HTTP itself doesn't know how to move bits across the network.
> > 
> > It hands the entire message to TCP:
> > 
> > ```text
> > HTTP Request
> >      │
> >      ▼
> > TCP
> > ```
> > 
> > TCP then breaks it into segments:
> > 
> > ```text
> > HTTP Message
> >      │
> >      ▼
> > ┌─────────┐
> > │ Segment │
> > ├─────────┤
> > │ Segment │
> > ├─────────┤
> > │ Segment │
> > └─────────┘
> > ```
> > 
> > and adds TCP headers:
> > 
> > ```text
> > [TCP Header][HTTP Data]
> > ```
> > 
> > ---
> > 
> > ## 🎯 Then IP Takes The TCP Segment
> > 
> > IP adds its own header:
> > 
> > ```text
> > [IP Header]
> >     +
> > [TCP Header]
> >     +
> > [HTTP Data]
> > ```
> > 
> > Now routers know:
> > 
> > - Source IP
> >     
> > - Destination IP
> >     
> > 
> > and can forward the packet across the Internet.
> > 
> > ---
> > 
> > ## 🎯 Then Ethernet/WiFi Wraps Everything Again
> > 
> > Finally:
> > 
> > ```text
> > [Ethernet Header]
> >       +
> > [IP Header]
> >       +
> > [TCP Header]
> >       +
> > [HTTP Data]
> > ```
> > 
> > This becomes actual electrical signals, radio waves, or optical pulses travelling across cables/fiber/WiFi.
> > 
> > ---
> > 
> > ## 📌 Real Example
> > 
> > Suppose you click:
> > 
> > ```text
> > http://localhost:8000/books/create_book
> > ```
> > 
> > with a POST body:
> > 
> > ```json
> > {
> >     "title": "Python"
> > }
> > ```
> > 
> > The stack roughly looks like:
> > 
> > ```text
> > Ethernet Frame
> > └── IP Packet
> >     └── TCP Segment
> >         └── HTTP Request
> >             ├── POST
> >             ├── /books/create_book
> >             ├── Content-Type: application/json
> >             └── {"title":"Python"}
> > ```
> > 
> > At the server:
> > 
> > ```text
> > Ethernet
> >     ↓
> > IP
> >     ↓
> > TCP
> >     ↓
> > HTTP
> >     ↓
> > FastAPI
> > ```
> > 
> > Each layer removes its own header and passes the remaining data upward.
> > 
> > This process is called **encapsulation** on the sender side and **decapsulation** on the receiver side.
> > 
> > ---
> > 
> > > [!tip]  
> > > 🎯 A useful mental model:
> > > 
> > > - **IP** answers: _"Which machine should receive this?"_
> > >     
> > > - **TCP** answers: _"How do I reliably deliver this stream of bytes?"_
> > >     
> > > - **HTTP** answers: _"What does this data mean? Is it a GET, POST, URL, headers, body, etc.?"_
> > >     
> > > - **FastAPI** answers: _"Which Python function should handle this HTTP request?"_
> > >     
> > 
> > So when FastAPI receives a POST request, the request body is not something FastAPI invented. It is a standard part of the **HTTP protocol message format**, which itself is carried inside TCP, which is carried inside IP, which is carried inside Ethernet/WiFi.
> 
> ---
> 
> ### 🎯 Key Insight
> 
> The route:
> 
> ```python
> "/books/create_book"
> ```
> 
> does **not contain the body**.
> 
> It is merely an **address**.
> 
> Think of it like mailing a package:
> 
> ```text
> Address (Route)
> ↓
> 123 Main Street
> 
> Package Contents (Body)
> ↓
> Book information
> ```
> 
> The address tells the package **where to go**.
> 
> The package contents are the **body**.
> 
> Similarly:
> 
> ```python
> @app.post("/books/create_book")
> ```
> 
> tells FastAPI where the request should go, while the actual book data arrives separately inside the HTTP request body.

# 🧠 Simple Analogy

Think of a **GET request** like asking a librarian:

> "Can you show me Book #5?"

You are only requesting information.

A **POST request** is like handing the librarian a completely new book and saying:

> "Please add this book to the library."

The entire book's information is provided, and the library stores it.

# 💻 Example Code

```python
from fastapi import FastAPI, Body

app = FastAPI()

books = []

@app.post("/books/create_book")
async def create_book(new_book=Body()):
    books.append(new_book)
    return {"message": "Book created successfully"}
```

### Example Request Body

```json
{
    "title": "Title Seven",
    "author": 2,
    "category": "Math"
}
```

> [!tip]  
> 🎯 **Remember:**
> 
> - **GET** → Retrieve existing data.
>     
> - **POST** → Create new data.
>     
> - **POST requests can contain a Request Body.**
>     
> - The Request Body is commonly used to send complex objects such as books, users, products, orders, etc.
>     
> - FastAPI can automatically read the body and make it available inside your function parameters.

## 📌 Summary

- **POST requests are used to create new resources/data.**
- **Request Body** contains the data being sent to the server.
- The URL remains clean because the data is not placed in query parameters.
- FastAPI uses `@app.post()` to define POST endpoints.
- The received body can be stored directly using operations such as:

```python
books.append(new_book)
```

This allows clients to send complete objects (such as books) to the FastAPI server and have them created or stored by the application.
