# ➕ Creating a POST Endpoint in FastAPI

In the previous lesson, we created a **GET endpoint** that returns all books stored in our `books` list.

Each item in this list is a **Book object**, created using our custom `Book` class.

```python
class Book:
    id: int
    title: str
    author: str
    description: str
    rating: int
```

The constructor (`__init__`) initializes these values whenever a new book object is created.

---

# 🎯 Goal of This Lesson

Create a **POST endpoint** that allows users to add new books to the application.

Current API functionality:

```text
GET /books
```

New functionality:

```text
POST /create-book
```

This endpoint will receive book information from the request body and add it to the `books` list.

---

# 🏗️ Creating the POST Endpoint

Add the following code:

```python
from fastapi import FastAPI, Body

@app.post("/create-book")
async def create_book(
    book_request = Body()
):
    books.append(book_request)
```

---

# 📌 Understanding `Body()`

`Body()` tells FastAPI:

> "Read the data from the HTTP request body and store it in this variable."

```python
book_request = Body()
```

When a client sends:

```json
{
    "id": 7,
    "title": "HP 4",
    "author": "Author 4",
    "description": "Book Description",
    "rating": 4
}
```

FastAPI places that JSON into:

```python
book_request
```

# 🚨 The Validation Problem

Let's try sending invalid data:

```json
{
    "id": -90,
    "title": "Bad Book",
    "author": "Unknown",
    "description": "Invalid Data",
    "rating": 1999
}
```

---

## What Happens?

FastAPI accepts it without complaint.

Response:

```http
200 OK
```

And the book gets stored.

---

## Result

```json
{
    "id": -90,
    "title": "Bad Book",
    "rating": 1999
}
```

This is clearly invalid.

---

# Why Is This Happening?

Because:

```python
book_request = Body()
```

only reads incoming data.

It does **not validate** it.

Think of `Body()` as a mailbox.

📬 Mailbox Analogy:

```text
Incoming Request
        │
        ▼
      Body()
        │
        ▼
 Accepts Everything
```

Whether the data is good or bad:

```json
{
    "rating": 5
}
```

or

```json
{
    "rating": 999999
}
```

`Body()` accepts both.

---

# 🎯 What Validation Should We Have?

Examples:

### Rating

Should only allow:

```text
1 → 5
```

Valid:

```json
{
    "rating": 1
}
```

```json
{
    "rating": 5
}
```

Invalid:

```json
{
    "rating": 999
}
```

---

### ID

Instead of allowing:

```json
{
    "id": -90
}
```

the API should automatically generate the next ID.

Example:

```text
Current Highest ID = 6
```

New book:

```text
ID = 7
```

Automatically.

---

# 📌 Current State of the Endpoint

### What It Can Do

✅ Accept request body
✅ Read JSON data
✅ Append data to list
✅ Return status code 200

---

### What It Cannot Do Yet

❌ Validate ratings
❌ Prevent negative IDs
❌ Ensure required fields exist
❌ Verify data types
❌ Auto-generate IDs
❌ Reject bad input

---

> [!tip]  
> 🎯 **`Body()` only extracts data from the request body. It does not validate that the data is correct.**
> 
> To build professional APIs, we need validation rules that ensure incoming data follows our business requirements.

# Key Takeaways

- Created a **POST endpoint** using `@app.post()`.
- Used `Body()` to read incoming request data.
- Added new books using:

```python
books.append(book_request)
```

- Verified the endpoint through Swagger.
- Learned that data currently exists only in memory.
- Discovered that `Body()` performs **no validation**.

    - ID validation

    - Required field validation
