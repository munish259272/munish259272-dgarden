# ✅ Data Validation in FastAPI Using Pydantic

In this lesson, we will learn how to perform **data validation** in FastAPI using **Pydantic**.

Until now, our API could accept almost any data sent by the user. For example:

```json
{
    "id": -90,
    "title": "Bad Book",
    "author": "Unknown",
    "description": "Invalid Book",
    "rating": 9999
}
```

FastAPI would happily accept this request because we were using:

```python
book_request = Body()
```

which simply reads the incoming data but does **not validate** it.

To solve this problem, FastAPI uses **Pydantic**.

---

# 📖 What is Pydantic?

**Pydantic** is a Python library used for:

- ✅ Data Validation
- ✅ Data Parsing
- ✅ Data Modeling
- ✅ Error Handling
- ✅ Type Checking

FastAPI relies heavily on Pydantic to validate incoming request data.

---

## 🎯 Why Do We Need Pydantic?

Without validation:

```json
{
    "id": -500,
    "rating": 99999
}
```

could enter our application.

With Pydantic:

```text
Incoming Request
        │
        ▼
 Pydantic Validation
        │
        ▼
 Valid? ── Yes ──► Continue
        │
        └── No ──► Return Error
```

This prevents invalid data from reaching our application logic.

---

# 🏗️ Creating a Request Model

To validate incoming requests, we create a **separate Pydantic model**.

This model represents the data we expect from the client.

```python
from pydantic import BaseModel

class BookRequest(BaseModel):
    id: int
    title: str
    author: str
    description: str
    rating: int
```

---

# 📌 Why Create a Separate Request Model?

You may notice that this looks very similar to our existing `Book` class.

```python
class Book:
    id: int
    title: str
    author: str
    description: str
    rating: int
```

and

```python
class BookRequest(BaseModel):
    id: int
    title: str
    author: str
    description: str
    rating: int
```

look almost identical.

---

## Why Not Use `Book` Directly?

Because these classes serve different purposes.

|Class|Purpose|
|---|---|
|`Book`|Internal Python object used by the application|
|`BookRequest`|Incoming request validation|

Think of it like this:

```text
Client
  │
  ▼
BookRequest
(Pydantic Validation)
  │
  ▼
Book
(Application Object)
```

---

# 🧠 What is `BaseModel`?

The most important part is:

```python
class BookRequest(BaseModel):
```

Notice the inheritance:

```python
(BaseModel)
```

This comes from Pydantic.

```python
from pydantic import BaseModel
```

By inheriting from `BaseModel`, we automatically get:

- Type validation
- Data parsing
- Error handling
- JSON conversion
- Serialization

---

## Example

Suppose we define:

```python
class BookRequest(BaseModel):
    rating: int
```

and receive:

```json
{
    "rating": "5"
}
```

Pydantic can automatically convert:

```text
"5"
```

into:

```text
5
```

if possible.

---

# 🔍 Adding Field Validation

Besides checking data types, Pydantic allows validation rules using `Field()`.

Example:

```python
from pydantic import BaseModel, Field

class BookRequest(BaseModel):
    title: str = Field(min_length=3)
    rating: int = Field(gt=0, lt=6)
```

---

## What Does `Field()` Do?

`Field()` allows us to add constraints.

Example:

```python
rating: int = Field(
    gt=0,
    lt=6
)
```

means:

$$\large{0 < rating < 6}$$

Valid ratings:

```text
1
2
3
4
5
```

Invalid ratings:

```text
0
6
100
-1
```

---

## Example Validation Flow

```text
Request
   │
   ▼
{
   "rating": 4
}
   │
   ▼
Field Validation
   │
   ▼
Pass ✅
```

---

```text
Request
   │
   ▼
{
   "rating": 99
}
   │
   ▼
Field Validation
   │
   ▼
Fail ❌
```

Once validation succeeds, we still need to create a real `Book` object.

The process looks like:

```text
JSON Request
      │
      ▼
BookRequest
(Pydantic)
      │
      ▼
Validation
      │
      ▼
Book Object
      │
      ▼
books List
```

```python
@app.post("/create-book")
async def create_book(
    book_request: BookRequest

    ...
```

Notice:

```python
book_request: BookRequest
```

FastAPI now knows:

1. Receive JSON data
2. Validate it using `BookRequest`
3. Reject invalid requests automatically

---

# Creating a New Book Object

After validation succeeds:

```python
new_book = Book(
    **book_request.dict()
)
```

---

# 🤔 What Does `dict()` Do?

Pydantic objects can be converted into dictionaries.

Example:

```python
book_request
```

contains:

```python
BookRequest(
    id=1,
    title="Python",
    author="Mukesh",
    description="Learning Python",
    rating=5
)
```

Calling:

```python
book_request.dict()
```

returns:

```python
{
    "id": 1,
    "title": "Python",
    "author": "Mukesh",
    "description": "Learning Python",
    "rating": 5
}
```

---

# 🤔 What Does `**` Mean?

The double-asterisk operator (`**`) is called **dictionary unpacking**.

Given:

```python
{
    "id": 1,
    "title": "Python",
    "author": "Mukesh",
    "description": "Learning Python",
    "rating": 5
}
```

This:

```python
Book(
    **book_request.dict()
)
```

becomes:

```python
Book(
    id=1,
    title="Python",
    author="Mukesh",
    description="Learning Python",
    rating=5
)
```

---

## Visual Representation

Without `**`

```python
Book(
    id=data["id"],
    title=data["title"],
    author=data["author"],
    description=data["description"],
    rating=data["rating"]
)
```

With `**`

```python
Book(
    **data
)
```

Much cleaner and easier to maintain.

# Example Endpoint

```python
@app.post("/create-book")
async def create_book(
    book_request: BookRequest
):
    new_book = Book(
        **book_request.dict()

```

# 📌 Key Takeaways

- **Pydantic** is FastAPI's primary tool for request validation.
- `BaseModel` provides automatic validation and parsing.
- `Field()` allows additional validation rules.
- `BookRequest` is used to validate incoming client data.
- `Book` remains the application's internal object.
- `book_request.dict()` converts a Pydantic object into a dictionary.
- `**` unpacks dictionary key-value pairs into function arguments.
- Validation happens **before** the endpoint logic executes.

> [!important]  
> 🎯 **A common FastAPI pattern is:**
> 
> ```text
> Request JSON
>      │
>      ▼
> Pydantic Model
>      │
>      ▼
> Validation
>      │
>      ▼
> Application Object
>      │
>      ▼
> Business Logic
> ```
> 
> This separation keeps APIs clean, safe, and easy to maintain.
