# Implementing HTTP Exceptions in FastAPI

So far, we have added validation for:

✅ Request bodies using `Field()`

✅ Path parameters using `Path()`

✅ Query parameters using `Query()`

However, there is still a problem.

A user may pass validation but request something that does not exist.

For example:

```http
GET /books/3000
```

The value `3000` is:

```python
3000 > 0
```

✅ Valid according to our path validation.

But there is no book with ID `3000`.

In this situation, validation succeeds, but the requested resource does not exist.

This is where **HTTP Exceptions** become useful.

---

# Import `HTTPException`

At the top of the file:

```python
from fastapi import FastAPI, Path, Query, HTTPException
```

---

# What is an HTTP Exception?

An HTTP Exception allows us to:

- Stop execution of the current function
    
- Return a specific HTTP status code
    
- Return a custom error message
    

General syntax:

```python
raise HTTPException(
    status_code=404,
    detail="Item not found"
)
```

When FastAPI encounters:

```python
raise HTTPException(...)
```

it immediately stops executing the endpoint and sends the response back to the client.

---

# Why Use HTTP Exceptions?

Consider the endpoint:

```python
@app.get("/books/{book_id}")
async def read_book(book_id: int):
```

Current logic:

```python
for book in books:
    if book.id == book_id:
        return book
```

Problem:

If no matching book is found:

```python
for book in books:
    if book.id == book_id:
        return book
```

the loop ends and nothing is returned.

The client receives an unclear response.

Instead, we should explicitly tell the client:

> "The requested book does not exist."

---

# Adding a 404 Not Found Exception

Update the endpoint:

```python
@app.get("/books/{book_id}")
async def read_book(
    book_id: int = Path(gt=0)
):
    for book in books:
        if book.id == book_id:
            return book

    raise HTTPException(
        status_code=404,
        detail="Item not found"
    )
```

# Adding HTTP Exceptions to PUT Requests

Consider the update endpoint.

Current logic:

```python
for i in range(len(books)):
    if books[i].id == book.id:
        books[i] = book
```

Problem:

If the requested ID does not exist:

- Loop completes
    
- Nothing is updated
    
- User still receives a success response
    

⚠️ This is misleading because no update actually occurred.

---

# Solution: Track Whether a Change Happened

Create a flag variable:

```python
book_changed = False
```

---

## Before Update

```python
book_changed = False
```

Meaning:

```text
No update has occurred yet.
```

---

## When Update Succeeds

```python
books[i] = book
book_changed = True
```

Meaning:

```text
A matching book was found and updated.
```

---

## Check After Loop

```python
if not book_changed:
    raise HTTPException(
        status_code=404,
        detail="Item not found"
    )
```

---

# Complete Example

```python
@app.put("/books/update_book")
async def update_book(book: BookRequest):

    book_changed = False

    for i in range(len(books)):
        if books[i].id == book.id:
            books[i] = book
            book_changed = True

    if not book_changed:
        raise HTTPException(
            status_code=404,
            detail="Item not found"
        )
```

---

# Why Use a Boolean Flag?

Think of it like a checklist:

```text
Book Found?
    │
    ├─ Yes → book_changed = True
    │
    └─ No  → book_changed remains False
```

After processing:

```python
if not book_changed:
```

we know that no matching record existed.

---

# Adding HTTP Exceptions to DELETE Requests

The same problem exists with delete operations.

Current logic:

```python
for i in range(len(books)):
    if books[i].id == book_id:
        books.pop(i)
```

If the book does not exist:

- Nothing is deleted
    
- User still receives success
    

❌ Incorrect behavior.

---

# Improved Delete Logic

Start with:

```python
book_changed = False
```

---

## When Deletion Occurs

```python
books.pop(i)
book_changed = True
```

---

## After Loop

```python
if not book_changed:
    raise HTTPException(
        status_code=404,
        detail="Item not found"
    )
```

---

# Complete Delete Example

```python
@app.delete("/books/{book_id}")
async def delete_book(
    book_id: int = Path(gt=0)
):

    book_changed = False

    for i in range(len(books)):
        if books[i].id == book_id:
            books.pop(i)
            book_changed = True
            break

    if not book_changed:
        raise HTTPException(
            status_code=404,
            detail="Item not found"
        )
```

# Validation vs HTTP Exceptions

These solve different problems.

|Tool|Purpose|
|---|---|
|`Field()`|Validate request body|
|`Path()`|Validate path parameters|
|`Query()`|Validate query parameters|
|`HTTPException`|Handle valid requests that cannot be completed|
