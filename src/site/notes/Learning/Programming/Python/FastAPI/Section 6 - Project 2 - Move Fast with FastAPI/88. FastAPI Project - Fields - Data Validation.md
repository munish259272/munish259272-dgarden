# Field Validation with Pydantic in FastAPI

## 🎯 Why Do We Need Validation?

In the previous section, we created a `BookRequest` model and used it as the request body for our **Create Book** endpoint.

The request was successfully transformed into a `Book` object and appended to the `books` list.

However, there is still a problem:

⚠️ Users can send invalid data.

For example:

- Empty titles
    
- Empty author names
    
- Descriptions that are too long
    
- Ratings outside the allowed range
    

Instead of manually checking every field, **Pydantic can automatically validate incoming data for us.**

---

# Importing `Field`

To add validation rules, import `Field` from Pydantic:

```python
from pydantic import BaseModel, Field
```

Since we are no longer using `Body`, we can remove it:

```python
from fastapi import FastAPI
```

---

# Adding Validation Rules

Our `BookRequest` model can be updated like this:

```python
class BookRequest(BaseModel):
    id: int
    title: str = Field(min_length=3)
    author: str = Field(min_length=1)
    description: str = Field(
        min_length=1,
        max_length=100
    )
    rating: int = Field(
        gt=0,
        lt=6
    )
```

---

## 📌 Understanding Each Validation Rule

### Title Validation

```python
title: str = Field(min_length=3)
```

The title must contain at least **3 characters**.

✅ Valid

```text
Python
```

❌ Invalid

```text
Hi
```

---

### Author Validation

```python
author: str = Field(min_length=1)
```

The author field cannot be empty.

---

### Description Validation

```python
description: str = Field(
    min_length=1,
    max_length=100
)
```

Requirements:

- At least 1 character
    
- Maximum 100 characters
    

Anything longer than 100 characters will cause validation to fail.

---

### Rating Validation

```python
rating: int = Field(
    gt=0,
    lt=6
)
```

Meaning:

$$\large{0 < Rating < 6}$$

Allowed values:

```text
1, 2, 3, 4, 5
```

Not allowed:

```text
0, 6, 7, ...
```

---

# What Happens When Validation Fails?

Suppose the user sends:

```json
{
  "title": "",
  "author": "John",
  "description": "Good book",
  "rating": 5
}
```

FastAPI automatically returns:

```json
{
  "detail": [
    {
      "msg": "ensure this value has at least 3 characters"
    }
  ]
}
```

HTTP Status:

```text
422 Unprocessable Entity
```

---

## 🎯 Important Benefit

Validation happens **before your endpoint code runs**.

### Request Flow

```text
Client Request
      │
      ▼
Pydantic Validation
      │
      ▼
Valid Request?
      │
 ┌────┴────┐
 │         │
Yes       No
 │         │
 ▼         ▼
Endpoint   422 Error
Runs
```

This prevents bad data from entering your application.

# Why Is This Useful?

Imagine someone submits:

```json
{
  "title": "",
  "author": "",
  "description": "",
  "rating": 500
}
```

Without validation:

- Invalid books could be stored.
    
- Extra checking code would be required everywhere.
    

With Pydantic:

✅ Validation is automatic.

✅ Cleaner code.

✅ Consistent API behavior.

# Auto-Generating Book IDs

## Problem

The client should **not** decide the book ID.

For example:

```json
{
  "id": 1000
}
```

A new book does not exist yet, so the client should not choose its ID.

Instead:

🎯 The server should generate IDs automatically.

---

# Creating a Helper Function

```python
def find_book_id(book):
    if len(books) > 0:
        book.id = books[-1].id + 1
    else:
        book.id = 1

    return book
```

---

## How It Works

### Case 1: Books Already Exist

Current IDs:

```text
1
2
3
4
5
6
```

Last book:

```python
books[-1]
```

returns:

```text
6
```

New ID becomes:

```text
6 + 1 = 7
```

---

### Case 2: First Book

If the list is empty:

```python
book.id = 1
```

The first book receives ID `1`.

---

# Using the Function

Before appending the book:

```python
books.append(
    find_book_id(new_book)
)
```

Execution order:

```text
Create Book
     │
     ▼
Generate ID
     │
     ▼
Append to List
```

---

# Ternary Operator Version

The same logic can be written in one line:

```python
book.id = (
    1
    if len(books) == 0
    else books[-1].id + 1
)
```

---

## Equivalent Logic

Traditional version:

```python
if len(books) == 0:
    book.id = 1
else:
    book.id = books[-1].id + 1
```

Ternary version:

```python
book.id = (
    1
    if len(books) == 0
    else books[-1].id + 1
)
```

Both produce the same result.

# Making ID Optional

## Problem

The request model still expects an ID:

```python
{
    "id": 0,
    "title": "Book",
    ...
}
```

But users should not provide IDs.

---

# Using `Optional`

Import:

```python
from typing import Optional
```

Update the model:

```python
id: Optional[int] = None
```

---

## What Does This Mean?

The field can contain:

```python
int
```

or

```python
None
```

Examples:

```python
id=5
```

✅ Valid

```python
id=None
```

✅ Valid

---

## Pydantic v2 Requirement

⚠️ When using `Optional`, you should also provide a default value:

```python
id: Optional[int] = None
```

Without `= None`, the field may still be treated as required.

---


Then:

```python

```


# ✅ Key Takeaways

- `Field()` allows **automatic validation** of request data.
    
- `min_length` and `max_length` validate string lengths.
    
- `gt` and `lt` validate numeric ranges.
    
- Invalid requests automatically return **422 Unprocessable Entity**.
    
- Validation occurs **before endpoint code executes**.
    
- Book IDs should be generated on the **server side**, not by the client.
    
- `books[-1]` retrieves the last item in a list.
    
- Ternary operators provide a shorter alternative to `if-else`.
    
- `Optional[int] = None` makes a field optional in **Pydantic v2**.
    
- The client only sends the data required to create the book, while the server handles ID generation.
