# Adding Validation to FastAPI Path Parameters

So far in our Books Project, we have added validation to our **Pydantic `BookRequest` model**, which validates data sent in **POST** and **PUT** request bodies.

However, we still have a gap in our validation logic:

1. **What if a book does not exist?**
    
2. **What if a user provides an invalid Book ID, such as `0` or `-1`?**
    

Currently, our API does not validate path parameters like `book_id`. Let's fix that. 🚀

---

# Current Validation Using Pydantic

Inside our `BookRequest` model, we already validate fields like `rating`.

Example:

```python
rating: int = Field(
    gt=0,
    lt=6
)
```

This means:

$$\large{0 < rating < 6}$$

Valid values:

```python
1
2
3
4
5
```

Invalid values:

```python
0
6
-1
```

✅ Pydantic automatically rejects invalid request bodies.

---

# The Problem with Path Parameters

Consider this endpoint:

```python
@app.get("/books/{book_id}")
async def read_book(book_id: int):
```

The `book_id` comes from the URL:

```http
/books/3
```

Currently, FastAPI converts it to an integer, but **does not enforce any additional constraints**.

This means requests like:

```http
/books/0
```

or

```http
/books/-1
```

are accepted and passed to our application.

⚠️ These are not valid book IDs.

---

# Using `Path` for Path Parameter Validation

FastAPI provides a special helper called `Path`.

Import it:

```python
from fastapi import FastAPI, Path
```

---

# Adding Validation to `book_id`

Update the endpoint:

```python
@app.get("/books/{book_id}")
async def read_book(
    book_id: int = Path(gt=0)
):
```

---

## What Does `gt=0` Mean?

```python
Path(gt=0)
```

means:

$$\large{\text{book\_id} > 0}$$

Only positive integers are accepted.

---

# Visual Representation

```text
Request
   │
   ▼
book_id received
   │
   ▼
Is book_id > 0 ?
   │
 ┌─┴─────┐
 │       │
Yes      No
 │       │
 ▼       ▼
Run API  Return Validation Error
```

🎯 Validation happens **before your endpoint function executes**.

---

# Testing Valid Input

Request:

```http
GET /books/3
```

Validation:

```python
3 > 0
```

✅ Passes validation.

Endpoint executes normally and returns Book 3.

---

# Testing Invalid Input

Request:

```http
GET /books/0
```

Validation:

```python
0 > 0
```

❌ False

FastAPI immediately returns:

```json
{
  "detail": [
    {
      "msg": "ensure this value is greater than 0"
    }
  ]
}
```

Status Code:

```http
422 Unprocessable Entity
```

---

# Understanding HTTP 422

> [!info]  
> **422 Unprocessable Entity** means:
> 
> - The request format is correct.
>     
> - FastAPI understands the request.
>     
> - The supplied data fails validation rules.
>     
> 
> Therefore, FastAPI refuses to execute the endpoint.

---

# Applying Validation to DELETE Endpoint

Our DELETE endpoint also uses a path parameter:

```python
@app.delete("/books/{book_id}")
async def delete_book(book_id: int):
```

We should apply the same validation:

```python
@app.delete("/books/{book_id}")
async def delete_book(
    book_id: int = Path(gt=0)
):
```

# Comparing Pydantic Validation and Path Validation

|Validation Type|Used For|Example|
|---|---|---|
|`Field()`|Request Body Validation|`rating`, `publish_date`|
|`Path()`|URL Path Parameter Validation|`book_id`|

Example:

```python
rating: int = Field(gt=0, lt=6)
```

Validates:

```json
{
  "rating": 5
}
```

---

```python
book_id: int = Path(gt=0)
```

Validates:

```http
/books/5
```

---

# Real-World Analogy

Imagine a library system:

- The **request body** is a form you fill out when adding a new book.
    
- The **path parameter** is the book number written on the library shelf.
    

Validation should occur in **both places**:

✅ Ensure the form contains valid information.

✅ Ensure the shelf number being requested actually follows valid rules.

Without path validation, someone could request:

```http
/books/-999
```

which makes no sense in a real library.

---

# Benefits of Path Validation

✅ Prevents invalid requests from reaching your business logic.

✅ Produces consistent API behavior.

✅ Automatically generates validation documentation in Swagger UI.

✅ Reduces bugs and defensive coding inside endpoint functions.

✅ Improves API reliability.
