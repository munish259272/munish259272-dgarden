# Updating Existing Data with a PUT Request in FastAPI

A **PUT request** is used to **update an existing object or piece of data** in a FastAPI application.

📌 In this example, we will update an existing book by sending new data in the request body.

---

## Creating the PUT Endpoint

```python
@app.put("/books/update_book")
async def update_book(book: BookRequest):
    for i in range(len(books)):
        if books[i].id == book.id:
            books[i] = book
```

---

## Understanding the Logic

### Endpoint Definition

```python
@app.put("/books/update_book")
```

This creates a **PUT API endpoint** that listens for requests at:

```http
/books/update_book
```

---

### Function Definition

```python
async def update_book(book: BookRequest):
```

The endpoint accepts a request body of type:

```python
BookRequest
```

which is a **Pydantic model**.

When a user sends JSON data, FastAPI automatically converts it into a `BookRequest` object.

---

### Loop Through Existing Books

```python
for i in range(len(books)):
```

This loops through every book in the `books` list.

---

### Find the Matching Book

```python
if books[i].id == book.id:
```

The goal is to find the book whose ID matches the ID sent in the request body.

---

### Replace the Existing Book

```python
books[i] = book
```

Once the matching book is found, the old book is replaced with the updated one.

✅ The book is effectively updated.

---

# Example Request

Suppose the current book list contains:

```python
Book(
    id=3,
    title="Master Endpoints",
    author="Coding with Ruby",
    description="This is an awesome book",

```

```json

    "rating": 5
}
```

---

## Result

The book:

```text
Master Endpoints
```

becomes:

```text
How to Use a Keyboard
```

# Why Is `id` Included Here?

At first, it might seem strange that the PUT request requires an ID even though the POST request did not.

The reason lies in the Pydantic model.

Example:

```python
class BookRequest(BaseModel):
    id: Optional[int] = None
    title: str
    author: str
    description: str
    rating: int
```

Notice:

```python
id: Optional[int] = None
```

This means:

- `id` can be provided.
    
- `id` can also be omitted.
    
- If omitted, its value becomes `None`.
    

---

## During POST Requests

When creating a new book:

```json
{
    "title": "New Book",
    "author": "Author",
    "description": "Description",
    "rating": 5
}
```

The ID is not required because the application may generate it automatically.

---

## During PUT Requests

When updating a book:

```json
{
    "id": 3,
    ...
}
```

The ID is necessary because the application must know **which book should be updated**.

🎯 Without an ID, FastAPI would not know which book to replace.

# Current Limitation of This Implementation

Consider this request:

```json
{
    "id": 100,
    "title": "Fake Book"
}
```

Suppose book ID 100 does not exist.

---

## What Happens?

The loop searches all books:

```python
for i in range(len(books)):
```

but never finds a match.

Since no match is found:

```python
books[i] = book
```

is never executed.

---

## Response Returned

FastAPI may still return:

```http
Status Code: 200 OK
```

with:

```json
null
```

Even though **nothing was updated**.

⚠️ This is misleading because the operation actually failed.

---

# Why Does This Happen?

Because the current function contains **no error handling**.

The function only says:

```python
Find matching book
If found, replace it
```

It never says:

```python
If book is not found,
return an error.
```

Therefore FastAPI assumes the request completed successfully.

---

# How This Should Ideally Work

A better implementation would:

1. Search for the book.
    
2. Update it if found.
    
3. Return an error if not found.
    

Example:

```python
if book_not_found:
    raise HTTPException(
        status_code=404,
        detail="Book not found"
    )
```

Then the response would be:

```http
404 Not Found
```

instead of:

```http
200 OK
```

✅ This makes the API more accurate and user-friendly.

---

> [!warning]  
> The current implementation updates books correctly when a valid ID is supplied, but it does **not** handle invalid IDs. Invalid requests may still return a successful status code until proper exception handling is added.

---

# PUT vs POST

|Feature|POST|PUT|
|---|---|---|
|Purpose|Create new data|Update existing data|
|Requires Existing Resource?|❌ No|✅ Yes|
|Typically Uses ID?|Usually No|Usually Yes|
|Example|Create a new book|Update Book ID 3|

---

> [!tip]  
> **Think of POST as "Add a new book to the shelf" and PUT as "Replace an existing book on the shelf."**
> 
> - POST creates something new.
>     
> - PUT modifies something that already exists.
>     

🎯 **Key Takeaways**

- **PUT requests are used to update existing resources.**
    
- **The request body is converted into a Pydantic object.**
    
- **The book ID is used to locate the book that needs updating.**
    
- **Once found, the old book is replaced with the new data.**
    
- **Without exception handling, invalid IDs may still return `200 OK`.**
    
- **Proper APIs should return meaningful status codes such as `404 Not Found` when an update cannot be performed.**
