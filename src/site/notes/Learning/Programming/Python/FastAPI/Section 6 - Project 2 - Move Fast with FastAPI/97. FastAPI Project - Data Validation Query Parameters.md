# Adding Validation to FastAPI Query Parameters

In the previous video, we added validation for **path parameters** using `Path`.

Now we will add validation for **query parameters** using `Query`.

---

# Import `Query`

At the top of the file, update the FastAPI import:

```python
from fastapi import FastAPI, Path, Query
```

Here:

|Import|Used For|
|---|---|
|`Path`|Validating path parameters|
|`Query`|Validating query parameters|

---

# What Are Query Parameters?

Query parameters are values passed after the `?` symbol in a URL.

Example:

```http
/books/?book_rating=5
```

Here:

```python
book_rating = 5
```

is a **query parameter**.

---

# Why Validate Query Parameters?

We already validate `rating` when creating or updating a book:

```python
rating: int = Field(gt=0, lt=6)
```

This means:

$$\large{0 < rating < 6}$$

So valid ratings are:

```text
1, 2, 3, 4, 5
```

If our API only allows books with ratings from **1 to 5**, then users should only be allowed to search using ratings from **1 to 5**.

✅ The validation rules should be consistent across the application.

---

# Add Validation to `book_rating`

Suppose we have this endpoint:

```python
@app.get("/books/")
async def read_books_by_rating(book_rating: int):
```

Update it like this:

```python
@app.get("/books/")
async def read_books_by_rating(
    book_rating: int = Query(gt=0, lt=6)
):
```

This means:

$$\large{0 < book\_rating < 6}$$

So only these values are valid:

```text
1, 2, 3, 4, 5
```

---

# Testing Rating Query Validation

## Valid Query

```http
GET /books/?book_rating=3
```

✅ This works because:

```python
3 > 0 and 3 < 6
```

The API returns all books with rating `3`.

---

## Invalid Query

```http
GET /books/?book_rating=8
```

❌ This fails because:

```python
8 < 6
```

is false.

FastAPI returns:

```http
422 Unprocessable Entity
```

With an error similar to:

```json
{
  "detail": [
    {
      "msg": "ensure this value is less than 6"
    }
  ]
}
```

# Add Validation to `publish_date`

We already added validation to `publish_date` in the `BookRequest` model:

```python
publish_date: int = Field(gt=1999, lt=2031)
```

This means:

$$\large{1999 < publish_date < 2031}$$

So valid years are:

```text
2000 to 2030
```

Now apply the same rule to the query parameter endpoint.

---

## Before Validation

```python
@app.get("/books/publish/")
async def read_books_by_publish_date(
    publish_date: int
):
```

---

## After Validation

```python
@app.get("/books/publish/")
async def read_books_by_publish_date(
    publish_date: int = Query(gt=1999, lt=2031)
):
```

This ensures the user can only search for publish dates between **2000 and 2030**.

# Validation Coverage So Far

At this point, our FastAPI application validates three important input areas:

|Input Area|FastAPI Tool|Example|

---

# Important Difference: Valid Format vs Existing Data

Even after validation, there is still one more problem.

Example:

```http
GET /books/3000
```

This request may pass validation because:

$$\large{3000 > 0}$$

But our application may not actually have any book with ID `3000`.

> [!warning]  
> `Path(gt=0)` only checks whether the ID is a **valid positive number**.  
> It does **not** check whether that book actually exists in the `books` list.

So we need another kind of validation:

✅ Is the input format valid?  
✅ Does the requested book actually exist?
