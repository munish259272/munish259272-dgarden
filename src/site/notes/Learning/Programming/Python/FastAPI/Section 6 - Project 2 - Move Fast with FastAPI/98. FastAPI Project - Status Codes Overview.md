# Understanding HTTP Status Codes in APIs

When working with APIs, every request sent from a client to a server returns a **status code**.

A **status code** is a standardized HTTP response that tells the client what happened on the server after processing the request.

📌 The client can be:

- A web browser
    
- A mobile application
    
- Another API
    
- A user interacting through Swagger UI
    

Status codes provide a universal way for clients and servers to communicate the outcome of a request.

---

# Why Are Status Codes Important?

Imagine ordering a product online.

Instead of silently accepting or rejecting your order, the store gives you a message:

|Situation|Real-Life Message|
|---|---|
|Order successful|"Your order has been placed."|
|Product not found|"Product does not exist."|
|Login required|"Please sign in first."|
|Store server crashed|"Something went wrong on our side."|

HTTP status codes serve the same purpose in APIs.

🎯 They tell the client:

- Was the request successful?
    
- Was there an error?
    
- Was the error caused by the client?
    
- Was the error caused by the server?
    

---

# HTTP Status Code Categories

Status codes are grouped into five major categories.

|Range|Category|Meaning|
|---|---|---|
|100–199|Informational|Request is being processed|
|200–299|Success|Request completed successfully|
|300–399|Redirection|Additional action is required|
|400–499|Client Errors|Problem caused by the client|
|500–599|Server Errors|Problem caused by the server|

---

# 1xx Informational Responses

These indicate that the request has been received and processing is continuing.

Examples:

```http
100 Continue
101 Switching Protocols
```

📌 These are mostly used internally by web servers and browsers.

You will rarely interact with them directly in FastAPI applications.

---

# 2xx Success Status Codes

These indicate that the request was successfully processed.

---

## 200 OK

```http
200 OK
```

Meaning:

✅ The request succeeded.

This is the most common success status code.

Typically used with:

```http
GET /books
```

Example:

```python
@app.get("/books")
async def get_books():
    return books
```

Response:

```http
200 OK
```

with data returned.

---

## 201 Created

```http
201 Created
```

Meaning:

✅ A new resource was successfully created.

Commonly used with:

```http
POST /books
```

Example:

```python
@app.post("/books")
async def create_book():
```

A new book is created in the system.

Response:

```http
201 Created
```

---

## 204 No Content

```http
204 No Content
```

Meaning:

✅ Request succeeded.

❌ No data is returned.

Commonly used when:

- Updating a resource
    
- Deleting a resource
    
- Performing an action without returning content
    

Example:

```http
PUT /books/5
```

The book is updated successfully, but no response body is sent.

---

# Success Status Code Summary

|Status Code|Meaning|
|---|---|
|200|Success with returned data|
|201|Resource created|
|204|Success with no response body|

# 4xx Client Error Status Codes

These errors occur because the client sent an invalid request.

📌 The server is working correctly.

The problem is with the request.

---

## 400 Bad Request

```http
400 Bad Request
```

Meaning:

❌ The server cannot process the request because the client sent invalid data.

Examples:

- Invalid request format
    
- Invalid request syntax
    
- Missing required data
    

---

## 401 Unauthorized

```http
401 Unauthorized
```

Meaning:

❌ Authentication is required.

Examples:

```text
Missing token
Invalid token
Expired token
```

The user must authenticate before accessing the resource.

---

## 404 Not Found

```http
404 Not Found
```

Meaning:

❌ The requested resource does not exist.

Example:

```http
GET /books/999
```

If Book 999 doesn't exist:

```http
404 Not Found
```

---

## 422 Unprocessable Entity

```http
422 Unprocessable Entity
```

Meaning:

❌ The request format is correct, but the data violates validation rules.

This is the error we have already seen in FastAPI.

---

### Example: Invalid Path Parameter

```python
book_id: int = Path(gt=0)
```

Request:

```http
GET /books/0
```

Validation:

```python
0 > 0
```

❌ False

Response:

```http
422 Unprocessable Entity
```

---

### Example: Invalid Query Parameter

```python
book_rating: int = Query(gt=0, lt=6)
```

Request:

```http
GET /books/?book_rating=10
```

Validation:

$$\large{10 < 6}$$

❌ False

Response:

```http
422 Unprocessable Entity
```

---

### Example: Invalid Request Body

```python
rating: int = Field(gt=0, lt=6)
```

Request:

```json
{
  "rating": 8
}
```

Response:

```http
422 Unprocessable Entity
```

> [!tip]  
> FastAPI heavily uses **422 Unprocessable Entity** whenever validation fails.

# 5xx Server Error Status Codes

These errors indicate that something went wrong on the server side.

📌 The client sent a valid request.

📌 The server failed while processing it.

---

## 500 Internal Server Error

```http
500 Internal Server Error
```

Meaning:

❌ An unexpected server-side error occurred.

Examples:

- Python exception
    
- Database failure
    
- Null reference error
    
- Unhandled exception
    
- Unexpected application crash
    

Example:

```python
number = 10 / 0
```

This causes:

```python
ZeroDivisionError
```

which may result in:

```http
500 Internal Server Error
```

---

# Client Error vs Server Error

## Client Error

Request:

```http
GET /books/0
```

Validation:

```python
book_id > 0
```

Fails.

Response:

```http
422 Unprocessable Entity
```

Problem:

❌ Client sent invalid data.

---

## Server Error

Request:

```http
GET /books/3
```

Valid request.

But:

```python
database_connection = None
database_connection.execute(...)
```

Application crashes.

Response:

```http
500 Internal Server Error
```

Problem:

❌ Server code failed.
