# Explicitly Returning HTTP Status Codes in FastAPI

So far, our FastAPI application has been automatically returning **`200 OK`** whenever a request succeeds.

While this works, professional APIs often return **more specific status codes** to better describe what happened on the server.

---

# Importing Status Codes

FastAPI is built on top of **Starlette**, and Starlette provides a collection of predefined HTTP status code constants.

Import them using:

```python
from starlette import status
```

> [!info]  
> FastAPI automatically installs Starlette, so no additional installation is required.

---

# Why Explicitly Define Status Codes?

Without explicitly setting a status code:

```python
@app.get("/books")
async def read_all_books():
    return books
```

FastAPI automatically returns:

```http
200 OK
```

However, we can make our intentions clearer:

```python
@app.get(
    "/books",
    status_code=status.HTTP_200_OK
)
async def read_all_books():
    return books
```

Now anyone reading the code immediately understands which status code is expected.

---

# Status Codes Used in This Project

|Operation|Status Code|Meaning|
|---|---|---|
|GET|200 OK|Success with returned data|
|POST|201 Created|New resource created|
|PUT|204 No Content|Resource updated successfully|
|DELETE|204 No Content|Resource deleted successfully|
|Missing Resource|404 Not Found|Item doesn't exist|
|Validation Failure|422 Unprocessable Entity|Invalid input|

---

# GET Endpoints → `200 OK`

A GET request retrieves data.

Since data is returned to the client, the standard response is:

```http
200 OK
```

---

## Read All Books

```python
@app.get(
    "/books",
    status_code=status.HTTP_200_OK
)
async def read_all_books():
    return books
```

---

## Read Book by ID

```python
@app.get(
    "/books/{book_id}",
    status_code=status.HTTP_200_OK
)
async def read_book(book_id: int):
```

---

## Read Books by Rating

```python
@app.get(
    "/books/",
    status_code=status.HTTP_200_OK
)
async def read_books_by_rating():
```

---

## Read Books by Publish Date

```python
@app.get(
    "/books/publish/",
    status_code=status.HTTP_200_OK
)
async def read_books_by_publish_date():
```

---

### Why Use `200 OK`?

A successful GET request usually means:

```text
Everything worked.
Here is the requested data.
```

Response:

```http
200 OK
```

---

# POST Endpoints → `201 Created`

POST requests typically create new resources.

Example:

```python
@app.post(
    "/books/create_book",
    status_code=status.HTTP_201_CREATED
)
async def create_book(book_request: BookRequest):
```

---

## Why `201 Created`?

When a client creates a new book:

```text
Book did not exist before.
Book exists now.
```

A new resource was created.

Therefore:

```http
201 Created
```

is more accurate than:

```http
200 OK
```

# PUT Endpoints → `204 No Content`

PUT requests usually update existing data.

Example:

```python
@app.put(
    "/books/update_book",
    status_code=status.HTTP_204_NO_CONTENT
)
async def update_book(book: BookRequest):
```

---

## Why Not Use `201 Created`?

Because:

```text
No new book was created.
```

We are simply modifying an existing one.

---

## Why Use `204 No Content`?

Because:

```text
Operation succeeded.
No response data is returned.
```

Response:

```http
204 No Content
```

---

### Example

Before:

```json
{
    "id": 3,
    "publish_date": 2020
}
```

Update:

```json
{
    "id": 3,
    "publish_date": 2021
}
```

After:

```json
{
    "id": 3,
    "publish_date": 2021
}
```

Response:

```http
204 No Content
```

No response body is returned.

---

# DELETE Endpoints → `204 No Content`

DELETE requests remove data.

Example:

```python
@app.delete(
    "/books/{book_id}",
    status_code=status.HTTP_204_NO_CONTENT
)
async def delete_book(book_id: int):
```

---

## Why Use `204 No Content`?

After deletion:

```text
Book removed successfully.
Nothing needs to be returned.
```

Response:

```http
204 No Content
```

---

### Example

Before:

```text
Book IDs:
1
2
3
```

Request:

```http
DELETE /books/2
```

After:

```text
Book IDs:
1
3
```

Response:

```http
204 No Content
```

---

# Difference Between `200 OK` and `204 No Content`

## Returning Data

```python
return books
```

Response:

```http
200 OK
```

with a response body.

---

## Returning Nothing

```python
# No return statement
```

Response:

```http
204 No Content
```

with **no response body at all**.

# HTTP Exceptions Still Override Success Codes

Even though we explicitly define success status codes:

```python
status_code=status.HTTP_200_OK
```

FastAPI will still return exceptions when necessary.

Example:

```python
raise HTTPException(
    status_code=404,
    detail="Item not found"
)
```

---

## Example

Endpoint:

```python
@app.get(
    "/books/{book_id}",
    status_code=status.HTTP_200_OK
)
```

Request:

```http
GET /books/9999
```

Book doesn't exist.

Response:

```http
404 Not Found
```

Body:

```json
{
    "detail": "Item not found"
}
```

⚠️ The exception overrides the default success status code.
