# Pydantic Validation for POST Requests in FastAPI

Until now, the `create_book()` endpoint was using `Body()`, but **`Body()` alone does not provide data validation**. We want to ensure that incoming data is valid **before** it is processed or stored.

To achieve this, FastAPI uses **Pydantic**, which is included automatically when FastAPI is installed.

---

# Why Use Pydantic?

🎯 **Pydantic validates incoming request data automatically.**

Without validation:

- Negative IDs could be accepted.
- Invalid data types could be submitted.
- Missing fields could cause errors later.

With Pydantic:

- Data types are checked automatically.
- Required fields are enforced.
- Additional validation rules can be added.

---

# Importing Pydantic

```python
from pydantic import BaseModel
```

- **Pydantic** is the validation framework.
- **BaseModel** is the parent class used to define request schemas.

---

# Creating a Request Model

Instead of accepting raw data from `Body()`, create a dedicated request model.

```python
class BookRequest(BaseModel):
    id: int
    title: str
    author: str
    description: str
    rating: int
```

This model defines exactly what data is expected from the client.

---

# Why Separate `Book` and `BookRequest`?

📌 A common design pattern is:

|Class|Purpose|
|---|---|
|`Book`|Internal application model (source of truth)|
|`BookRequest`|Incoming API request validation|

This separation ensures:

- External data is validated first.
- Internal data structure remains protected.
- Future request changes won't affect internal models.

```text
Client Request
       ↓
BookRequest (Validation)
       ↓
Book (Internal Model)
       ↓
Storage / Database
```

---

# Using the Request Model in POST Endpoint

Before:

```python
@app.post("/books")
def create_book(book_request=Body()):
    ...
```

After:

```python
@app.post("/books")
def create_book(book_request: BookRequest):
    ...
```

Now FastAPI automatically:

✅ Reads the JSON body
✅ Converts it into a `BookRequest` object
✅ Validates all fields
✅ Returns validation errors if data is invalid

---

# Automatic Swagger Documentation

After adding `BookRequest`, FastAPI automatically generates an example schema.

Swagger UI will show:

```json
{
  "id": 0,
  "title": "string",
  "author": "string",
  "description": "string",
  "rating": 0
}
```

This comes directly from the Pydantic model.

📌 **One of FastAPI's biggest advantages is that documentation is generated automatically from your models.**

---

# What Type is `book_request`?

If you print its type:

```python
print(type(book_request))
```

Output:

```python
<class 'BookRequest'>
```

This means FastAPI converts incoming JSON into a real Python object.

---

# Problem: Mixing Models

Suppose we do:

```python
books.append(book_request)
```

Then our list contains:

```text
Book
Book
BookRequest
Book
BookRequest
```

⚠️ This is not ideal because we want our application to store only `Book` objects.

---

# Converting `BookRequest` into `Book`

Instead of storing `BookRequest`, create a real `Book` object.

```python
new_book = Book(
    **book_request.dict()
)
```

Then:

```python
books.append(new_book)
```

---

# Pydantic v1 vs Pydantic v2

FastAPI supports both versions.

### Pydantic v1

```python
book_request.dict()
```

### Pydantic v2

```python
book_request.model_dump()
```

⚠️ If `.dict()` is deprecated or produces an error, use:

```python
book_request.model_dump()
```

instead.

---

# Understanding `**` (Double Asterisk)

Many beginners find this confusing.

Consider:

```python
book_request.dict()
```

Returns:

```python
{
    "id": 1,
    "title": "Python",
    "author": "Mukesh",
    "description": "Learning FastAPI",
    "rating": 5
}
```

Using:

```python
**book_request.dict()
```

expands the dictionary into keyword arguments.

Equivalent to:

```python
Book(
    id=1,
    title="Python",
    author="Mukesh",
    description="Learning FastAPI",
    rating=5
)
```

---

## Visual Explanation

```text
book_request
      ↓

{
  "id": 1,
  "title": "Python",
  "author": "Mukesh",
  "description": "Learning FastAPI",
  "rating": 5
}

      ↓

** expands dictionary

      ↓

Book(
  id=1,
  title="Python",
  author="Mukesh",
  description="Learning FastAPI",
  rating=5
)
```

🎯 The `**` operator converts dictionary key-value pairs into constructor arguments.

# Complete Example

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class Book:
    def __init__(
        self,
        id,
        title,
        author,
        description,
        rating
    ):
        self.id = id
        self.title = title
        self.author = author
        self.description = description
        self.rating = rating


class BookRequest(BaseModel):
    id: int
    title: str
    author: str
    description: str
    rating: int


@app.post("/books")
def create_book(book_request: BookRequest):

    new_book = Book(
        **book_request.model_dump()  # Pydantic v2
    )

    books.append(new_book)

    return {"message": "Book created successfully"}
```

---

# Current Validation Limitation

At this stage, Pydantic only validates:

```python
id: int
rating: int
title: str
author: str
description: str
```

Therefore these values are still accepted:

```json
{
  "id": -50,
  "title": "Python",
  "author": "Mukesh",
  "description": "Test",
  "rating": 100
}
```

because:

- `-50` is still an integer.
    
- `100` is still an integer.
    

So type validation passes successfully.

---

# Adding Stronger Validation

In the next step, we can add constraints such as:

```python
id: int = Field(gt=0)
rating: int = Field(ge=1, le=5)
```

This would enforce rules like:

- ID must be greater than 0.
- Rating must be between 1 and 5.

Then requests such as:

```json
{
  "id": -50,
  "rating": 100
}
```

would be rejected automatically.

---

> [!success]  
> ✅ **Summary**
> 
> - **Pydantic (`BaseModel`) is used for request validation in FastAPI.**
>     
> - **`BookRequest` validates incoming JSON data.**
>     
> - **FastAPI automatically converts JSON into a Pydantic object.**
>     
> - **Use `book_request: BookRequest` instead of `Body()`.**
>     
> - **Convert `BookRequest` into a `Book` object before storing it.**
>     
> - **`book_request.dict()` (v1) or `book_request.model_dump()` (v2) converts the model into a dictionary.**
>     
> - **`**` expands dictionary key-value pairs into constructor arguments.**
>     
> - **Current validation only checks data types.**
>     
> - **Field-level validation (ranges, limits, constraints) can be added later.**
