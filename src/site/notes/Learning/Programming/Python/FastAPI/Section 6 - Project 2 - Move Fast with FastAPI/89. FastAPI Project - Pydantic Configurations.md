# Improving FastAPI Request Documentation with `Field()` and `model_config`

In the previous section, we added validation rules to the fields of our `BookRequest` model:

- `title`
    
- `author`
    
- `description`
    
- `rating`
    

We also made `id` optional:

```python
id: Optional[int] = None
```

This is important because:

- A client **does not know the ID** of a new book being created.
    
- The server generates the ID automatically.
    
- However, the ID may still be needed later for operations such as:
    
    - Updating a book
        
    - Deleting a book

# Adding Descriptions to Fields

The schema is already useful, but we can make it even more informative.

Instead of:

```python
id: Optional[int] = None
```

use:

```python
id: Optional[int] = Field(
    description="ID is not needed on create",
    default=None
)
```

---

## Why Use a Description?

When users view the API documentation, they will immediately understand:

> The ID should not be provided when creating a new book.

This reduces confusion for API consumers.

---

# ⚠️ Important: Always Use `default=None`

When making a field optional in **Pydantic v2**, make sure you provide:

```python
default=None
```

Example:

```python
id: Optional[int] = Field(
    description="ID is not needed on create",
    default=None
)
```

If you forget `default=None`:

```python
id: Optional[int]
```

the field may still be treated as required.

---

> [!warning]  
> **`Optional[int]` does not automatically make a field optional.**
> 
> In Pydantic v2, you should also specify:
> 
> ```python
> default=None
> ```
> 
> Otherwise FastAPI may still expect the field in the request body.

# Customizing Example Values

By default, Swagger generates generic examples:

```json
{
  "title": "string",
  "author": "string",
  "description": "string",
  "rating": 0
}
```

These examples are technically valid but not very helpful.

It is better to provide realistic examples.

---

# Using `model_config`

Inside the Pydantic model:

```python
class BookRequest(BaseModel):

    id: Optional[int] = Field(
        description="ID is not needed on create",
        default=None
    )

    title: str = Field(min_length=3)
    author: str = Field(min_length=1)
    description: str = Field(
        min_length=1,
        max_length=100
    )
    rating: int = Field(
        gt=0,
        lt=6
    )

    model_config = {
        "json_schema_extra": {
            "example": {
                "title": "A New Book",
                "author": "Coding with Ruby",
                "description": "A new description of a book",
                "rating": 5
            }
        }
    }
```

---

# Understanding `model_config`

The `model_config` section allows us to customize how the model appears in generated documentation.

In this case:

```python
"json_schema_extra"
```

adds custom schema information.

And:

```python
"example"
```

defines the example request shown in Swagger UI.

# Why Is `id` Missing from the Example?

Notice that:

```json
{
  "title": "A New Book",
  "author": "Coding with Ruby",
  "description": "A new description of a book",
  "rating": 5
}
```

does **not** contain an ID.

This is intentional.

The example only contains fields that a client should provide when creating a book.

The server will generate the ID automatically.

# Example Request Flow

Client sends:

```json
{
  "title": "A New Book",
  "author": "Coding with Ruby",
  "description": "A new description of a book",
  "rating": 5
}
```

Pydantic creates:

```python
BookRequest(
    id=None,
    title="A New Book",
    author="Coding with Ruby",
    description="A new description of a book",
    rating=5
)
```

Then:

```python
find_book_id(new_book)
```

assigns:

```python
id = 7
```

Finally:

```python
books.append(new_book)
```

# Changes Between Older and Newer FastAPI/Pydantic Versions

You may encounter older tutorials that use:

```python
class Config:
```

instead of:

```python
model_config = {}
```

Modern FastAPI with **Pydantic v2** uses:

```python
model_config
```

---

You may also see older code like:

```python
id: Optional[int]
```

without:

```python
= None
```

or

```python
default=None
```

This worked in older versions but should now be written as:

```python
id: Optional[int] = Field(
    description="ID is not needed on create",
    default=None
)
```

for proper optional-field behavior.

# ✅ Key Takeaways

- `Field(description=...)` improves API documentation.
    
- `default=None` is important for optional fields in **Pydantic v2**.
    
- Swagger UI automatically displays field descriptions.
    
- `model_config` customizes schema behavior.
    
- `json_schema_extra` allows custom examples.
    
- Realistic examples make APIs easier to understand.
    
- The client should not provide a book ID during creation.
    
- The server generates IDs automatically.
    
- `model_config` replaces the older `class Config` approach in modern FastAPI.
