# Creating a Todo with a POST Request

In the previous lesson, we created an endpoint that fetches a single todo by its ID.

In this lesson, we'll create a **POST endpoint** that allows users to create new todos and save them into the database.

This introduces several important concepts:

✅ Pydantic request models

✅ Request body validation

✅ Database inserts

✅ SQLAlchemy sessions

✅ Transactions and commits

---

# Why Do We Need a Request Model?

When a client sends a POST request, it typically includes data in the request body.

Example:

```json
{
  "title": "Learn FastAPI",
  "description": "Build API endpoints",
  "priority": 5,
  "complete": false
}
```

Before saving this data into the database, we should verify:

- Does the title exist?
    
- Is the title long enough?
    
- Is the priority valid?
    
- Is the data structure correct?
    

This is where **Pydantic models** help.

---

# Creating a Request Model

Create the following class below your dependency definitions:

```python
from pydantic import BaseModel

class TodoRequest(BaseModel):
    title: str
    description: str
    priority: int
    complete: bool
```

---

# Why Doesn't the Request Include an ID?

Our database table contains:

```python
id
title
description
priority
complete
```

However, the request model only contains:

```python
title
description
priority
complete
```

Notice that:

```python
id
```

is missing.

---

## Why?

Because:

```python
id
```

is the primary key.

The database automatically generates it.

---

### Example

Client sends:

```json
{
  "title": "Learn FastAPI",
  "description": "Build API endpoints",
  "priority": 5,
  "complete": false
}
```

Database stores:

```json
{
  "id": 4,
  "title": "Learn FastAPI",
  "description": "Build API endpoints",
  "priority": 5,
  "complete": false
}
```

The client never needed to provide:

```python
id
```

---

> [!tip]  
> 📌 Let the database manage primary keys.
> 
> This guarantees:
> 
> - Unique IDs
>     
> - Auto-increment behavior
>     
> - Fewer user errors
>

# Final Request Model

```python
class TodoRequest(BaseModel):
    title: str = Field(min_length=3)

    description: str = Field(
        min_length=3,
        max_length=100
    )

    priority: int = Field(
        gt=0,
        lt=6
    )

    complete: bool
```


> [!question]- What is `BaseModel` in `class TodoRequest(BaseModel):` ? I saw importing `Base` earlier i think it was to make a FastApi class convert/think the class as a SQL table (Correct me if i am wrong).
> Yes, your understanding is very close, but there are actually **two different "Base" classes** being used in this project:
> 
> |Class|Comes From|Purpose|
> |---|---|---|
> |`Base`|SQLAlchemy|Used to define database tables|
> |`BaseModel`|Pydantic|Used to define request/response data models|
> 
> ---
> 
> # 1. SQLAlchemy `Base`
> 
> Earlier you probably had something like:
> 
> ```python
> from sqlalchemy.orm import declarative_base
> 
> Base = declarative_base()
> ```
> 
> Then:
> 
> ```python
> class Todos(Base):
>     __tablename__ = "todos"
> 
>     id = Column(Integer, primary_key=True)
>     title = Column(String)
> ```
> 
> ---
> 
> ## What Does This Mean?
> 
> By inheriting from:
> 
> ```python
> class Todos(Base):
> ```
> 
> you are telling SQLAlchemy:
> 
> > "This class represents a database table."
> 
> SQLAlchemy then uses the class definition to create:
> 
> ```sql
> CREATE TABLE todos (
>     id INTEGER PRIMARY KEY,
>     title VARCHAR
> );
> ```
> 
> behind the scenes.
> 
> ---
> 
> ### Mental Model
> 
> ```text
> Python Class
>       │
>       ▼
> SQLAlchemy Base
>       │
>       ▼
> Database Table
> ```
> 
> So your statement:
> 
> > "Base is to make FastAPI think the class is a SQL table"
> 
> is almost correct.
> 
> More accurately:
> 
> > **Base makes SQLAlchemy treat the class as a database table.**
> 
> FastAPI is not involved here.
> 
> ---
> 
> # 2. Pydantic `BaseModel`
> 
> Now consider:
> 
> ```python
> from pydantic import BaseModel
> 
> class TodoRequest(BaseModel):
>     title: str
>     description: str
>     priority: int
>     complete: bool
> ```
> 
> This is completely different.
> 
> ---
> 
> ## What Does This Mean?
> 
> By inheriting from:
> 
> ```python
> BaseModel
> ```
> 
> you are telling Pydantic:
> 
> > "This class represents structured data that should be validated."
>

> [!question]- I belive `**todo_request.model_dump()` gives key value pairs from the dictionary. How we are able to use this as keyword arguments in`Todos( **todo_request.model_dump() )`. I do not see `__init__()` defined in `Todos`.
> Exactly. The interesting part here is that **you don't see an `__init__()` because SQLAlchemy generates one for you**.
> 
> Let's walk through it.
> 
> ---
> 
> # Step 1: What Does `model_dump()` Return?
> 
> Suppose:
> 
> ```python
> todo_request = TodoRequest(
>     title="Learn FastAPI",
>     description="Build APIs",
>     priority=5,
>     complete=False
> )
> ```
> 
> Then:
> 
> ```python
> todo_request.model_dump()
> ```
> 
> returns:
> 
> ```python
> {
>     "title": "Learn FastAPI",
>     "description": "Build APIs",
>     "priority": 5,
>     "complete": False
> }
> ```
> 
> ---
> 
> # Step 2: What Does `**` Do?
> 
> The `**` operator unpacks a dictionary into keyword arguments.
> 
> Example:
> 
> ```python
> data = {
>     "title": "Learn FastAPI",
>     "priority": 5
> }
> ```
> 
> Then:
> 
> ```python
> my_func(**data)
> ```
> 
> becomes:
> 
> ```python
> my_func(
>     title="Learn FastAPI",
>     priority=5
> )
> ```
> 
> Python literally rewrites it like this.
> 
> ---
> 
> # Step 3: Your Confusion
> 
> You are asking:
> 
> > How can this work if `Todos` does not define `__init__()`?
> 
> Good question.
> 
> You wrote:
> 
> ```python
> class Todos(Base):
>     __tablename__ = "todos"
> 
>     id = Column(Integer, primary_key=True)
>     title = Column(String)
>     description = Column(String)
>     priority = Column(Integer)
>     complete = Column(Boolean)
> ```
> 
> There is no visible:
> 
> ```python
> def __init__(...):
> ```
> 
> yet this works:
> 
> ```python
> Todos(
>     title="Learn FastAPI",
>     description="Build APIs",
>     priority=5,
>     complete=False
> )
> ```
> 
> ---
> 
> # Step 4: SQLAlchemy Generates `__init__()` Automatically
> 
> When SQLAlchemy sees:
> 
> ```python
> class Todos(Base):
> ```
> 
> it creates something conceptually similar to:
> 
> ```python
> def __init__(
>     self,
>     title=None,
>     description=None,
>     priority=None,
>     complete=None,
>     **kwargs
> ):
>     self.title = title
>     self.description = description
>     self.priority = priority
>     self.complete = complete
> ```
> 
> behind the scenes.
> 
> You never see this code because SQLAlchemy generates it dynamically.
> 
> ---
> 
> # What SQLAlchemy Actually Does
> 
> Conceptually:
> 
> ```python
> todo = Todos(
>     title="Learn FastAPI",
>     description="Build APIs",
>     priority=5,
>     complete=False
> )
> ```
> 
> becomes:
> 
> ```python
> todo = Todos()
> 
> todo.title = "Learn FastAPI"
> todo.description = "Build APIs"
> todo.priority = 5
> todo.complete = False
> ```
> 
> ---
> 
> # Why Can SQLAlchemy Do This?
> 
> Because each:
> 
> ```python
> title = Column(String)
> ```
> 
> is not a normal class variable.
> 
> It is a SQLAlchemy descriptor object.
> 
> When SQLAlchemy processes the model:
> 
> ```python
> class Todos(Base):
> ```
> 
> it collects all these columns:
> 
> ```python
> id
> title
> description
> priority
> complete
> ```
> 
> and builds the constructor automatically.
> 
> ---
> 
> # You Can Verify It
> 
> Try:
> 
> ```python
> print(Todos.__init__)
> ```
> 
> You will see something like:
> 
> ```python
> <function __init__ at 0x...>
> ```
> 
> even though you never wrote one.
> 
> Or:
> 
> ```python
> help(Todos)
> ```
> 
> and you'll find an auto-generated constructor.
> 
> ---
> 
> # Similar Example With Dataclasses
> 
> Python's dataclasses do something similar:
> 
> ```python
> from dataclasses import dataclass
> 
> @dataclass
> class Person:
>     name: str
>     age: int
> ```
> 
> You never write:
> 
> ```python
> def __init__(...):
> ```
> 
> but this works:
> 
> ```python
> Person(
>     name="Mukesh",
>     age=45
> )
> ```
> 
> because Python generates the constructor.
> 
> SQLAlchemy does a similar trick.
> 
> ---
>

> ```
>

>

> ```

> }
> ```
> 
> Then:
>

> ```
> 
> becomes:
> 
> ```python

>     title="Learn FastAPI",
>     description="Build APIs",
>     priority=5,
>     complete=False
> )
> ```
> 
> Then SQLAlchemy's generated constructor does roughly:
> 
> ```python

> 
> ---
> 
> > [!tip]  
> > A useful mental model:
> > 
> > ```python
> > class Todos(Base):
> > ```
> > 
> > does **two jobs**:
> > 
> > 1. Defines the database table structure.


## What Does `**` Do?

The double asterisk unpacks the dictionary.

Example:

```python

    "title": "Learn FastAPI",
```


# Committing the Transaction

```python

```

This performs the actual database transaction.

---

## Difference Between `add()` and `commit()`

### `add()`

```python
db.add(todo_model)
```

Means:

```text
Prepare this record
for insertion.
```

---

### `commit()`

```python
db.commit()
```

Means:

```text
Execute transaction
and save changes.
```

---

# Analogy

Imagine writing a cheque.

### Add

```text
Fill out the cheque
```

---

### Commit

```text
Hand it to the bank
for processing
```

Without committing:

```text
Nothing actually happens
```

# Complete Endpoint

```python
@app.post(
    "/todo",
    status_code=status.HTTP_201_CREATED
)
async def create_todo(
    db: DBDependency,
    todo_request: TodoRequest
):
    todo_model = Todos(
        **todo_request.dict()
    )

    db.add(todo_model)
    db.commit()
```

---
