3. **<span style="font-weight:bold; color:#000000">Recall(Sensitivity) or TPR(True positive rate)</span>**
$$\text{Recall} = \frac {\text{True positive}} {\text{Total actual positive}} = \frac {TP} {TP + FN}$$ i.e out of all the total actual positive how many i was able to predict correctly
$$\Large{Recall \propto \frac {1} {FN}}$$
> [!example]-
> Recall measures the ability of the model to correctly identify all the patients who actually have cancer. It's calculated as the ratio of true positives to the total number of patients who actually have cancer.
> 
> For example, let's say we have a dataset of 100 patients, out of which 30 patients actually have cancer (total actual positives). Now, if our model correctly identifies 25 out of these 30 patients as having cancer (true positives), then the recall would be:
> $\large{\text{Recall} = \frac{\text{True Positives}}{\text{Total Actual Positives}} = \frac{25}{30} = 0.833}$
> 
> This means that our model has a recall of 0.833 or 83.3%. In other words, it correctly identifies 83.3% of the patients who actually have cancer.
> 
> A high recall value indicates that the model is good at capturing most of the positive cases (patients with cancer) from the dataset, which is crucial in medical diagnosis to **<span style="font-weight:bold; color:#000000">ensure that as few cases as possible are missed(FN(missed) is as low as possible).</span>**

#mnemonics **<span style="font-weight:bold; color:rgb(0, 0, 0)">recall</span>** tells about the accuracy relative to reality

4. **<span style="font-weight:bold; color:#000000">Precision</span>**
$$\text{Precision} = \frac {\text{True positive}} {\text{Total positive prediction}} = \frac {TP} {TP + FP}$$ i.e out of all the total positive prediction how many i was able to predict correctly. or how many of my predictions are correct.

$$\Large{Precision \propto \frac {1} {FP}}$$
Make sure FP is high when we do not want to miss anything(FN). In the case of patients with cancer if we do not want to miss any cancel patient even at the cost of classifying a healthy person being classified as cancer patient so that we can conduct further test on them. i.e We do not mind spending some money on test rather than missing(FN) a cancer patient.
#mnemonics **<span style="font-weight:bold; color:rgb(0, 0, 0)">precision</span>** tells about the accuracy of your prediction

5) F1-score : Lets assume we are not sure which one to reduce FN or FP then we use F1-score. F1-score is a harmonic mean of both precision and recall so it takes both into consideration.
$$\large{\text{F1-score} = \frac{2}{\frac{1}{Precision} + \frac{1}{Recall}}}$$

6) True negative rate/specificity :
Rate of predicting negative cases correctly 
$$\text{TNR} = \frac {\text{True negative}} {\text{Total actual negative}} = \frac {TN} {TN + FP}$$ i.e out of all the total actual negative how many i was able to predict correctly

7) False positive rate:
Rate of incorrectly predicting negative cases
$$\text{FPR} = \frac {\text{Actual negative marked as positive}} {\text{Total actual negative}} = \frac {FP} {TN + FP}$$ 
8) False negative rate:

$$\text{FNR} = \frac {\text{Actual positive marked as negative}} {\text{Total actual positive}} = \frac {FN} {TN + FN}$$

> [!question]-  What is `fit_transform()` and why should we NOT use `fit_transform()` on test data?
> 
> ## 🔍 What is `fit_transform()`?
> 
> `fit_transform()` is a **two-step operation combined into one**:
> 
> ```python
> scaler.fit(X_train)        # Step 1: Learn parameters
> scaler.transform(X_train)  # Step 2: Apply transformation
> ```
> 
> 👉 Combined:
> 
> ```python
> X_train_scaled = scaler.fit_transform(X_train)
> ```
> 
> ---
> 
> ## 🧠 What Does “fit” Actually Mean?
> 
> > **Fit = learn statistics from data**
> 
> For example with `StandardScaler`:
> 
> - Mean (μ)
>     
> - Standard deviation (σ)
>     
> 
> ---
> 
> ## 📐 Transformation Formula
> 
> After fitting:
> 
> $$z = \frac{x - \mu}{\sigma}$$
> 
> So:
> 
> - `fit()` → computes μ and σ
>     
> - `transform()` → applies this formula
>     
> 
> ---
> 
> ## 🎯 Why `fit_transform()` is Used on Training Data
> 
> ```python
> X_train_scaled = scaler.fit_transform(X_train)
> ```
> 
> 👉 Correct because:
> 
> - Model is allowed to **learn from training data**
>     
> - Scaling parameters come from **training distribution**
>     
> 
> ---
> 
> ## ❌ Why We SHOULD NOT Use `fit_transform()` on Test Data
> 
> ```python
> # ❌ WRONG
> X_test_scaled = scaler.fit_transform(X_test)
> ```
> 
> ---
> 
> ## 🚨 Problem: Data Leakage
> 
> > You are allowing the model to **learn from test data**
> 
> ### What happens internally?
> 
> - New mean and std are computed from **test data**
>     
> - Test data influences preprocessing
>     
> - Your evaluation becomes **biased**
>     
> 
> ---
> 
> ## 🔬 Correct Approach
> 
> ```python
> # ✅ Correct
> X_train_scaled = scaler.fit_transform(X_train)
> X_test_scaled  = scaler.transform(X_test)
> ```
> 
> 👉 Important:
> 
> - Test data uses **training μ and σ**
>     
> - No learning from test set
>     
> 
> ---
> 
> ## 🧪 Intuition (Very Important)
> 
> Think of test data as:
> 
> > “Future unseen data”
> 
> You **cannot**:
> 
> - Recompute scaling using future data ❌
>     
> 
> You **must**:
> 
> - Use knowledge learned from past (training) ✔
>     
> 
> ---
> 
> ## ⚠️ What Goes Wrong If You Ignore This
> 
> |Mistake|Consequence|
> |---|---|
> |`fit_transform` on test|Data leakage|
> |Leakage|Inflated performance|
> |Inflated performance|Model fails in real world|
> 
> ---
> 
> ## 🎯 Real Example
> 
> Imagine:
> 
> - Train mean = 30
>     
> - Test mean = 100
>     
> 
> If you scale test separately:
> 
> - Model sees **normalized values differently**
>     
> - Predictions become inconsistent
>     
> 
> ---
> 
> ## 📌 Rule to Remember
> 
> > **Fit ONLY on training data**  
> > **Transform everywhere else**
> 
> ---
> 
> ## 🎯 Final Answer
> 
> > `fit_transform()` learns parameters (like mean and std) and applies transformation.  
> > We must NOT use it on test data because it causes **data leakage**, leading to **unrealistically good model performance**.
>

> [!question]- Should we only and always compare the test and the training predictions based on `accuracy_score` in classification as seen above to decide how well the test data is performing compared to training data?
> 
> No, we should not only and always compare the test and training predictions based on `accuracy_score` in classification tasks to decide how well the test data is performing compared to the training data. While accuracy is an important metric, it may not provide a complete picture of model performance, especially when dealing with imbalanced datasets or when different types of errors have different costs.
> 
> Here are some additional metrics and methods that can provide a more comprehensive evaluation:
> 
> 1. **Confusion Matrix**:
>     - Provides detailed information about true positives (TP), true negatives (TN), false positives (FP), and false negatives (FN).
>     - Helps understand the types of errors the model is making.

> In practice:
> - If you have an imbalanced dataset where one class significantly outnumbers another class (`e.g.,` fraud detection or medical diagnosis), accuracy alone can be misleading because a high accuracy might simply reflect the predominance of the majority class rather than actual predictive capability on minority classes.
>   
> - Use a combination of these metrics tailored to your specific problem to get a holistic view of your model’s performance.
> 
> By considering these additional metrics and methods alongside accuracy score comparison between training and test predictions you will gain deeper insights into how well your machine learning model generalizes from training data to unseen test data while addressing any potential issues like overfitting or underfitting effectively
> 
> > [!question]- So we should apply all these to both training data (`x_train`) and test data (`x_test`) to compare ?
> > 
> > 
> > No, you don't necessarily need to apply all these metrics to both the training data (`X_train`) and the test data (`X_test`). Typically, you would apply these metrics primarily to the test data for evaluation purposes. Here's why:
> > 
> > 1. **Training Metrics**:
> > 	- **Purpose**: The primary purpose of calculating metrics on the training data is to get an understanding of how well your model has learned from the training set.
> > 	- **Usage**: These metrics are useful for diagnosing overfitting or underfitting.
> > 	- **Indicators**:
> > 		- High accuracy on training data but significantly lower accuracy on test data indicates overfitting.
> > 		- Low accuracy on both training and test data indicates underfitting.
> > 
> > 2. **Test Metrics**:
> > 	- **Purpose**: The primary purpose of calculating metrics on the test data is to evaluate how well your model generalizes to unseen data.

> 
> ### What is `predict_proba()`?
> 
> The `predict_proba()` method in scikit-learn's classifiers (including `LogisticRegression`) returns the probability estimates for each class. For a binary classification problem, it provides two probabilities for each sample:
> 
> 1. The probability that the sample belongs to the negative class (class 0).
> 2. The probability that the sample belongs to the positive class (class 1).
> 
> The output of `predict_proba()` is an array where each row corresponds to a sample and each column corresponds to a class. For binary classification, this array has shape `(n_samples, 2)`.
> 
> ### Why do we use `predict_proba()`?
> 
> We use `predict_proba()` when we need the probability estimates for each class rather than just hard predictions. This can be particularly useful for:
> 
> 1. **ROC Curve**: The Receiver Operating Characteristic (ROC) curve is a graphical representation of a classifier's performance across different thresholds. To plot an ROC curve, we need to vary the threshold and compute the True Positive Rate (TPR) and False Positive Rate (FPR) at each threshold. This requires probability estimates, which are provided by `predict_proba()`.
>   
> 2. **AUC-ROC Score**: The Area Under the ROC Curve (AUC-ROC) is a single metric that summarizes the performance of a classifier across all possible thresholds. Again, computing this requires probability estimates.
> 
> 3. **Threshold Tuning**: If you want to set a custom decision threshold instead of using 0.5 by default, you need probability estimates to make this decision.
> 
> ### Example

$$

}
$$
Where TP = True positive

$$\text{Recall} = \frac {\text{True positive}} {\text{Total actual positive}} = \frac {TP} {TP + FN}$$ i.e out of all the total actual positive how many i was able to predict correctly
$$\Large{Recall \propto \frac {1} {FN}}$$
> [!example]-

> $\large{\text{Recall} = \frac{\text{True Positives}}{\text{Total Actual Positives}} = \frac{25}{30} = 0.833}$
>
